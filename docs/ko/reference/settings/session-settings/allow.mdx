  ## allow_key_condition_coalesce_rewrite
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.5"},{"label": "1"},{"label": "`coalesce(a_1, ..., a_N) <op> const` 형태의 프레디케이트(및 이에 해당하는 `ifNull`, 또는 상수가 왼쪽에 있는 동일한 형태)를 인덱스 분석 전에 논리합으로 재작성하여 각 `a_i`에 대한 컬럼별 프라이머리 키와 스킵 인덱스를 사용할 수 있게 하는 새로운 설정입니다. `coalesce(a, 42, b)` 및 `coalesce(a, b, 42)`와 같은 일부 상수 형태도 처리합니다."}]}]} />

`coalesce` 또는 `ifNull`이 포함된 `WHERE`/`PREWHERE` 프레디케이트에서 MergeTree 프라이머리 키(primary key)와 스킵 인덱스가 그래뉼을 프루닝할 수 있도록 합니다. 이 설정이 없으면 이러한 프레디케이트는 인덱스 분석에서 해석되지 않아 프루닝이 수행되지 않으므로, 일치할 수 없는 그래뉼도 읽게 됩니다. 이 설정은 어떤 그래뉼을 읽을지에만 영향을 주며, 행은 여전히 원래 프레디케이트로 필터링되므로 쿼리 결과는 변경되지 않습니다.

인덱스 분석 전에 다음 2가지 프레디케이트 형태를 재작성합니다:

* `coalesce(a, b) = 5`와 같이 `coalesce`/`ifNull`에 비교를 적용한 형태는 각 인수의 인덱스로 프루닝할 수 있도록 논리합으로 변환됩니다: `a = 5 OR (a IS NULL AND b = 5)`. 인수가 더 많아도 같은 방식으로 확장됩니다.
* `ifNull(a = 5, 0)` 또는 `coalesce(a = 5, 0)`처럼 거짓으로 평가되는(0) 상수 기본값을 조건으로 직접 사용하는 `coalesce`/`ifNull`은 내부 프레디케이트 `a = 5`로 풀어집니다. 이러한 래퍼는 내부 프레디케이트의 3값 결과를 확정적인 불리언 값으로 축약합니다(`NULL`을 `false`로 매핑).

`Tuple(...)` 타입의 추출된 서브컬럼을 `Nullable(Tuple(...))`로 지정할 수 있는지 제어합니다.

* `false`: `Tuple(...)`를 반환하고, 서브컬럼이 없는 행에는 기본 Tuple 값을 사용합니다.
* `true`: `Nullable(Tuple(...))`를 반환하고, 서브컬럼이 없는 행에는 `NULL`을 사용합니다.

이 설정은 추출된 서브컬럼의 동작만 제어합니다.
테이블에서 `Nullable(Tuple(...))` 컬럼을 생성할 수 있는지는 제어하지 않으며, 이는 `enable_nullable_tuple_type`에서 제어됩니다.

ClickHouse는 서버 시작 시 로드된 이 설정 값을 사용합니다.
`SET` 또는 쿼리 수준 `SETTINGS`로 변경해도 추출된 서브컬럼의 동작은 바뀌지 않습니다.
추출된 서브컬럼의 동작을 변경하려면 시작 프로필 구성(예: users.xml)에서 `allow_nullable_tuple_in_extracted_subcolumns` 값을 수정한 뒤 서버를 재시작하십시오.
