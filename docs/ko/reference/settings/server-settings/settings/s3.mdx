이 설정은 ClickHouse 서버를 구성하며, ClickHouse 소스에서 자동으로 생성됩니다.

<div id="s3_allow_server_credentials_for_system_table_disks">
  ## s3_allow_server_credentials_for_system_table_disks
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

`system` 데이터베이스의 테이블에 있는 동적 `disk(type = s3, ...)`가 서버 자체의 ambient S3 자격 증명을 사용할 수 있도록 하며, `s3_allow_server_credentials_in_user_queries` 제한에서는 예외로 처리됩니다. 이는 서버의 아이덴티티를 사용해 시스템 테이블을 S3로 내보내는 서버 내부 인프라를 위한 설정이며, 클라우드 운영자가 이를 `system` 데이터베이스에 attach합니다. session 설정과 달리 이것은 서버 수준 설정이므로, 재시작 시 메타데이터에서 테이블을 다시 로드할 때도 이 예외가 적용됩니다. 또한 이 설정의 범위는 `system` 데이터베이스로 한정되며, 일반 사용자는 이 데이터베이스에 테이블을 생성할 수 없으므로 사용자 쿼리에 대한 제한이 완화되지는 않습니다.

<div id="s3_load_table_anonymously_if_credentials_restricted">
  ## s3_load_table_anonymously_if_credentials_restricted
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

지속형 `S3` 또는 `S3Queue` 테이블(table), 동적 `disk(type = s3, ...)`, 또는 `DataLakeCatalog`(Glue, BigLake) 데이터베이스를
기존 메타데이터(metadata)(서버 시작 또는 `RESTORE`)에서 로드할 때, 해당 정의가 사용자
쿼리에서 `s3_allow_server_credentials_in_user_queries`에 의해 차단된 서버 자체의 (ambient) S3/cloud 자격 증명으로
해석되는 경우 이를 어떻게 처리할지 제어합니다.

활성화되면(기본값) 시작을 중단하지 않고 해당 자격 증명 없이 객체를 로드합니다. 즉,
`S3`/`S3Queue` 테이블 또는 동적 S3 디스크는 익명 S3 클라이언트로 생성되고, `DataLakeCatalog`
데이터베이스는 사용할 수 없는 카탈로그 상태로 남습니다. 서버는 시작되지만 객체에는 접근할 수 없습니다(프라이빗 버킷에 대한 요청은
거부되거나, 카탈로그가 명확한 오류를 보고합니다). 자격 증명이 다시 허용된 자격 증명 소스로 해석될 때까지
이 상태가 유지됩니다. 서버의 아이덴티티를 자동으로 다시 획득하는 일은 결코 없습니다. 이렇게 하면 이러한 객체 하나만으로도 —
예를 들어 이러한 제한이 생기기 전의 이전 version에서 생성되었거나, 해당 명명된 컬렉션이
나중에 `use_environment_credentials = 1`로 다시 바인딩된 경우 — 서버 시작이 중단되는 일을 막을 수 있습니다.

비활성화하면 이러한 객체의 로드는 대신 실패하며, 이로 인해 서버가 시작되지 않을 수 있습니다. 테이블, 디스크 또는 카탈로그 데이터베이스가
조용히 접근 불가 상태가 되는 것보다 명시적인 실패를 선호하는 경우에만 이 설정을 사용하십시오.
