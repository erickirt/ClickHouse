이 설정은 ClickHouse 서버를 구성하는 항목이며 ClickHouse 소스에서 자동 생성됩니다.

<div id="use_minimalistic_part_header_in_zookeeper">
  ## use_minimalistic_part_header_in_zookeeper
</div>

ZooKeeper에서 데이터 파트 헤더를 저장하는 방식입니다. 이 설정은 [`MergeTree`](/ko/reference/engines/table-engines/mergetree-family/index) 계열에만 적용됩니다. 다음과 같이 지정할 수 있습니다.

**`config.xml` 파일의 [merge&#95;tree](/ko/reference/settings/server-settings/settings/merge#merge_tree) 섹션에서 전역으로**

ClickHouse는 서버의 모든 테이블에 이 설정을 적용합니다. 이 설정은 언제든지 변경할 수 있습니다. 기존 테이블은 설정이 변경되면 동작 방식이 바뀝니다.

**각 테이블별로**

테이블을 생성할 때 해당 [engine setting](/ko/reference/engines/table-engines/mergetree-family/mergetree#table_engine-mergetree-creating-a-table)을 지정하십시오. 이 설정이 지정된 기존 테이블의 동작은 전역 설정이 변경되더라도 바뀌지 않습니다.

**가능한 값**

* `0` — 기능이 비활성화됩니다.
* `1` — 기능이 활성화됩니다.

[`use_minimalistic_part_header_in_zookeeper = 1`](/ko/reference/settings/server-settings/settings/use#use_minimalistic_part_header_in_zookeeper)인 경우, [복제된](/ko/reference/engines/table-engines/mergetree-family/replication) 테이블은 단일 `znode`를 사용해 데이터 파트의 헤더를 간결한 형태로 저장합니다. 테이블에 컬럼이 많으면 이 저장 방식은 ZooKeeper에 저장되는 데이터 양을 크게 줄여줍니다.

<Note>
  `use_minimalistic_part_header_in_zookeeper = 1`을 적용한 후에는 이 설정을 지원하지 않는 버전으로 ClickHouse 서버를 다운그레이드할 수 없습니다. cluster의 서버에서 ClickHouse를 업그레이드할 때는 주의하십시오. 모든 서버를 한 번에 업그레이드하지 마십시오. 테스트 환경 또는 cluster의 일부 서버에서만 새 ClickHouse 버전을 테스트하는 것이 더 안전합니다.

  이 설정으로 이미 저장된 데이터 파트 헤더는 이전의(간결하지 않은) 형태로 복원할 수 없습니다.
</Note>

<div id="use_separate_cache_arena">
  ## use_separate_cache_arena
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

캐시 할당에 사용할 전용 jemalloc arena를 활성화합니다(마크 캐시, 압축 해제 캐시, 페이지 캐시).
캐시 데이터를 쿼리 처리 중의 메모리 할당과 분리하여 메모리 단편화를 줄입니다.

<div id="use_shared_merge_tree_log_pipeline">
  ## use_shared_merge_tree_log_pipeline
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

ClickHouse Cloud에서만 사용할 수 있습니다. 활성화하면 `system.*_log` 테이블은 S3 기반 파이프라인을 통해 `SharedMergeTree`를 사용하는 방식으로 구성됩니다.
각 로그 `<log>`에 대해 시작 시 다음 객체가 자동으로 생성됩니다:

* `system.<log>_s3` — 직접 플러시 대상이 되는 `S3` 기반 테이블입니다. 각
  `SYSTEM FLUSH LOGS` 호출은 여기에 새 파티션 파일을 기록합니다.
* `system.<log>_s3queue` — `<log>_s3`에서 파일을 가져와
  행을 다운스트림으로 스트리밍하는 `S3Queue` 테이블(ordered mode)입니다. 각 node는
  `partition_regex` 기반 파티셔닝을 통해 자신에게 해당하는 파일만 처리합니다.
* `system.<log>_mv` — `<log>_s3queue`의 행을
  최종 `SharedMergeTree` 테이블로 전달하는 `MATERIALIZED VIEW`입니다.
* `system.<log>` — 행이 누적되어 쿼리되는 최종 `SharedMergeTree` 테이블입니다.

스키마 또는 설정이 변경되면 영향을 받는 테이블의 이름이 `<log>_0`, `<log>_1` 등으로 바뀐 후
기존 `SystemLog` 교체 동작과 동일한 방식으로 다시 생성됩니다.

서버 구성에서 `<shared_log_pipeline><endpoint>`를 설정해야 합니다.
관련 항목: `shared_log_pipeline.enable_sync_flush`, `shared_log_pipeline.flush_timeout_seconds`.
