  대체 인증 메서드를 고려하십시오.
</Warning>

<div id="compression">
  ## 압축
</div>

[MergeTree](/ko/reference/engines/table-engines/mergetree-family/mergetree)-엔진 테이블의 데이터 압축 설정입니다.

<Note>
  ClickHouse를 막 사용하기 시작했다면 이 값을 변경하지 않는 것이 좋습니다.
</Note>

**구성 템플릿**:

```xml
<compression>
    <case>
      <min_part_size>...</min_part_size>
      <min_part_size_ratio>...</min_part_size_ratio>
      <method>...</method>
      <level>...</level>
    </case>
    ...
</compression>
```

**`<case>` 필드**:

* `min_part_size` – 데이터 파트(data part)의 최소 크기입니다.
* `min_part_size_ratio` – 데이터 파트 크기와 테이블 크기의 비율입니다.
* `method` – 압축 방식입니다. 허용되는 값: `lz4`, `lz4hc`, `zstd`,`deflate_qpl`.
* `level` – 압축 수준입니다. [코덱](/ko/reference/statements/create/table#general-purpose-codecs)을 참조하십시오.

<Note>
  여러 개의 `<case>` 섹션을 구성할 수 있습니다.
</Note>

**조건 충족 시 동작**:

* 데이터 파트가 조건 Set과 일치하면 ClickHouse는 지정된 압축 방식을 사용합니다.
* 데이터 파트가 여러 조건 Set과 일치하면 ClickHouse는 가장 먼저 일치한 조건 Set을 사용합니다.

<Note>
  데이터 파트에 대해 충족되는 조건이 없으면 ClickHouse는 `lz4` 압축을 사용합니다.
</Note>

**예시**

```xml
<compression incl="clickhouse_compression">
    <case>
        <min_part_size>10000000000</min_part_size>
        <min_part_size_ratio>0.01</min_part_size_ratio>
        <method>zstd</method>
        <level>1</level>
    </case>
</compression>
```

<div id="crash_log">
  ## crash_log
</div>

[crash&#95;log](/ko/reference/system-tables/crash_log) 시스템 테이블 동작에 대한 설정입니다.

다음 설정은 하위 태그로 설정할 수 있습니다:

| 설정                                 | 설명                                                                                                                                  | 기본값                 | 참고                                                                                    |
| ---------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- | ------------------- | ------------------------------------------------------------------------------------- |
| `buffer_size_rows_flush_threshold` | 행 수에 대한 임계값입니다. 이 임계값에 도달하면 로그를 디스크로 플러시하는 작업이 백그라운드에서 시작됩니다.                                                                       | `max_size_rows / 2` |                                                                                       |
| `database`                         | 데이터베이스 이름입니다.                                                                                                                       |                     |                                                                                       |
| `engine`                           | 시스템 테이블용 [MergeTree 엔진 정의](/ko/reference/engines/table-engines/mergetree-family/mergetree#table_engine-mergetree-creating-a-table)입니다. |                     | `partition_by` 또는 `order_by`가 정의된 경우에는 사용할 수 없습니다. 지정하지 않으면 기본적으로 `MergeTree`가 선택됩니다. |
| `flush_interval_milliseconds`      | 메모리 버퍼의 데이터를 테이블로 플러시하는 간격입니다.                                                                                                      | `7500`              |                                                                                       |
| `flush_on_crash`                   | 크래시가 발생했을 때 로그를 디스크로 덤프할지 여부를 설정합니다.                                                                                                | `false`             |                                                                                       |
| `max_size_rows`                    | 로그의 최대 행 수입니다. 플러시되지 않은 로그 수가 max&#95;size에 도달하면 로그가 디스크로 덤프됩니다.                                                                    | `1024`              |                                                                                       |
| `order_by`                         | 시스템 테이블용 [사용자 지정 정렬 키](/ko/reference/engines/table-engines/mergetree-family/mergetree#order_by)입니다. `engine`이 정의된 경우 사용할 수 없습니다.       |                     | 시스템 테이블에 `engine`이 지정된 경우 `order_by` 매개변수는 &#39;engine&#39; 내부에 직접 지정해야 합니다.          |
| `partition_by`                     | 시스템 테이블용 [사용자 지정 파티셔닝 키](/ko/reference/engines/table-engines/mergetree-family/custom-partitioning-key)입니다.                             |                     | 시스템 테이블에 `engine`이 지정된 경우 `partition_by` 매개변수는 &#39;engine&#39; 내부에 직접 지정해야 합니다.      |
| `reserved_size_rows`               | 로그용으로 미리 할당된 메모리 크기(행 수)입니다.                                                                                                        | `1024`              |                                                                                       |
| `settings`                         | MergeTree의 동작을 제어하는 [추가 매개변수](/ko/reference/engines/table-engines/mergetree-family/mergetree#settings)(선택 사항)입니다.                      |                     | 시스템 테이블에 `engine`이 지정된 경우 `settings` 매개변수는 &#39;engine&#39; 내부에 직접 지정해야 합니다.          |
| `storage_policy`                   | 테이블에 사용할 스토리지 정책 이름입니다(선택 사항).                                                                                                      |                     | 시스템 테이블에 `engine`이 지정된 경우 `storage_policy` 매개변수는 &#39;engine&#39; 내부에 직접 지정해야 합니다.    |
| `table`                            | 시스템 테이블 이름입니다.                                                                                                                      |                     |                                                                                       |
| `ttl`                              | 테이블 [TTL](/ko/reference/engines/table-engines/mergetree-family/mergetree#table_engine-mergetree-ttl)을 지정합니다.                           |                     | 시스템 테이블에 `engine`이 지정된 경우 `ttl` 매개변수는 &#39;engine&#39; 내부에 직접 지정해야 합니다.               |

기본 서버 구성 파일 `config.xml`에는 다음 설정 섹션이 포함되어 있습니다:

```xml
<crash_log>
    <database>system</database>
    <table>crash_log</table>
    <partition_by>toYYYYMM(event_date)</partition_by>
    <flush_interval_milliseconds>7500</flush_interval_milliseconds>
    <max_size_rows>1024</max_size_rows>
    <reserved_size_rows>1024</reserved_size_rows>
    <buffer_size_rows_flush_threshold>512</buffer_size_rows_flush_threshold>
    <flush_on_crash>false</flush_on_crash>
</crash_log>
```

<div id="encryption">
  ## 암호화
</div>

[암호화 코덱](/ko/reference/statements/create/table#encryption-codecs)에서 사용할 키를 가져오는 명령을 설정합니다. 키(또는 여러 개의 키)는 환경 변수에 저장하거나 설정 파일에 지정해야 합니다.

키는 16바이트 길이의 16진수 값 또는 문자열일 수 있습니다.

**예시**

구성에서 불러오기:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key>1234567812345678</key>
    </aes_128_gcm_siv>
</encryption_codecs>
```

<Note>
  설정 파일에 키를 저장하는 것은 권장되지 않습니다. 보안상 안전하지 않습니다. 키를 안전한 디스크의 별도 설정 파일로 옮기고, 해당 설정 파일을 가리키는 심볼릭 링크를 `config.d/` 폴더에 둘 수 있습니다.
</Note>

키가 16진수 형식일 때 구성에서 로드하는 방법:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex>00112233445566778899aabbccddeeff</key_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

환경 변수에서 키를 불러옵니다:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex from_env="ENVVAR"></key_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

여기서 `current_key_id`는 암호화에 사용할 현재 키를 지정하며, 지정된 모든 키는 복호화에 사용할 수 있습니다.

이 각각의 방법은 여러 키에 적용할 수 있습니다:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex id="0">00112233445566778899aabbccddeeff</key_hex>
        <key_hex id="1" from_env="ENVVAR"></key_hex>
        <current_key_id>1</current_key_id>
    </aes_128_gcm_siv>
</encryption_codecs>
```

여기서 `current_key_id`는 암호화에 사용되는 현재 키를 나타냅니다.

또한 사용자는 반드시 12바이트 길이의 nonce를 추가할 수 있습니다(기본적으로 암호화 및 복호화 과정에서는 0바이트로만 이루어진 nonce를 사용합니다):

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <nonce>012345678910</nonce>
    </aes_128_gcm_siv>
</encryption_codecs>
```

또는 16진수 형식으로 설정할 수 있습니다:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <nonce_hex>abcdefabcdef</nonce_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

<Note>
  위에서 설명한 모든 내용은 `aes_256_gcm_siv`에도 적용됩니다(단, 키는 32바이트여야 합니다).
</Note>

<div id="ldap_servers">
  ## ldap_servers
</div>

여기에 LDAP 서버와 해당 connection 매개변수를 나열하여 다음과 같이 사용할 수 있습니다.

* &#39;password&#39; 대신 &#39;ldap&#39; authentication 메커니즘이 지정된 전용 로컬 사용자의 authenticator로 사용합니다
* 원격 user directories로 사용합니다.

다음 설정은 sub-tags로 구성할 수 있습니다.

| Setting                        | Description                                                                                                                                                                                                                                                              |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `bind_dn`                      | bind에 사용할 DN을 구성하는 템플릿입니다. 각 authentication 시도 시 템플릿의 모든 `\{user_name\}` substrings가 실제 사용자 이름으로 치환되어 최종 DN이 구성됩니다.                                                                                                                                                      |
| `enable_tls`                   | LDAP 서버에 대한 secure connection 사용을 trigger하는 플래그입니다. 일반 텍스트(`ldap://`) protocol에는 `no`를 지정하십시오(권장하지 않음). SSL/TLS를 통한 LDAP(`ldaps://`) protocol에는 `yes`를 지정하십시오(권장되며 기본값임). 레거시 StartTLS protocol에는 `starttls`를 지정하십시오(일반 텍스트(`ldap://`) protocol을 TLS로 upgrade함).         |
| `host`                         | LDAP 서버의 호스트명 또는 IP입니다. 이 매개변수는 필수이며 비워 둘 수 없습니다.                                                                                                                                                                                                                        |
| `port`                         | LDAP 서버 포트입니다. `enable_tls`가 true로 설정된 경우 기본값은 636이고, 그렇지 않으면 `389`입니다.                                                                                                                                                                                                  |
| `tls_ca_cert_dir`              | CA certificates가 포함된 directory의 경로입니다.                                                                                                                                                                                                                                   |
| `tls_ca_cert_file`             | CA certificate file의 경로입니다.                                                                                                                                                                                                                                              |
| `tls_cert_file`                | certificate file의 경로입니다.                                                                                                                                                                                                                                                 |
| `tls_cipher_suite`             | 허용되는 cipher suite(OpenSSL 표기법)입니다.                                                                                                                                                                                                                                       |
| `tls_key_file`                 | certificate key file의 경로입니다.                                                                                                                                                                                                                                             |
| `tls_minimum_protocol_version` | SSL/TLS의 minimum protocol version입니다. 허용되는 값은 `ssl2`, `ssl3`, `tls1.0`, `tls1.1`, `tls1.2`(기본값)입니다.                                                                                                                                                                      |
| `tls_require_cert`             | SSL/TLS peer certificate verification 동작입니다. 허용되는 값은 `never`, `allow`, `try`, `demand`(기본값)입니다.                                                                                                                                                                          |
| `user_dn_detection`            | bind된 사용자의 실제 user DN을 확인하기 위한 LDAP search 매개변수가 포함된 섹션입니다. 이는 서버가 Active Directory인 경우 추가 역할 매핑을 위한 search filter에서 주로 사용됩니다. 최종 user DN은 허용된 위치에서 `\{user_dn\}` substrings를 치환할 때 사용됩니다. 기본적으로 user DN은 bind DN과 동일하게 설정되지만, search가 수행되면 실제로 확인된 user DN 값으로 업데이트됩니다. |
| `verification_cooldown`        | bind 시도가 성공한 후의 시간(초)으로, 이 기간 동안에는 LDAP 서버에 문의하지 않아도 이후의 모든 consecutive requests에 대해 사용자가 성공적으로 authenticated된 것으로 assume됩니다. caching을 비활성화하고 각 authentication request마다 LDAP 서버에 강제로 문의하려면 `0`(기본값)을 지정하십시오.                                                            |

`user_dn_detection` 설정은 sub-tags로 구성할 수 있습니다.

| Setting         | Description                                                                                                                                                                                                        |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `base_dn`       | LDAP search에 사용할 base DN을 구성하는 템플릿입니다. LDAP search 중 템플릿의 모든 `\{user_name\}` 및 &#39;&#123;bind&#95;dn&#125;&#39; substrings가 실제 사용자 이름과 bind DN으로 치환되어 최종 DN이 구성됩니다.                                               |
| `scope`         | LDAP search 범위입니다. 허용되는 값은 `base`, `one_level`, `children`, `subtree`(기본값)입니다.                                                                                                                                     |
| `search_filter` | LDAP search에 사용할 search filter를 구성하는 템플릿입니다. LDAP search 중 템플릿의 모든 `\{user_name\}`, `\{bind_dn\}`, `\{base_dn\}` substrings가 실제 사용자 이름, bind DN, base DN으로 치환되어 최종 filter가 구성됩니다. XML에서는 특수 문자를 올바르게 escape해야 합니다. |

예시:

```xml
<my_ldap_server>
    <host>localhost</host>
    <port>636</port>
    <bind_dn>uid={user_name},ou=users,dc=example,dc=com</bind_dn>
    <verification_cooldown>300</verification_cooldown>
    <enable_tls>yes</enable_tls>
    <tls_minimum_protocol_version>tls1.2</tls_minimum_protocol_version>
    <tls_require_cert>demand</tls_require_cert>
    <tls_cert_file>/path/to/tls_cert_file</tls_cert_file>
    <tls_key_file>/path/to/tls_key_file</tls_key_file>
    <tls_ca_cert_file>/path/to/tls_ca_cert_file</tls_ca_cert_file>
    <tls_ca_cert_dir>/path/to/tls_ca_cert_dir</tls_ca_cert_dir>
    <tls_cipher_suite>ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:AES256-GCM-SHA384</tls_cipher_suite>
</my_ldap_server>
```

예시 (후속 역할 매핑을 위해 user DN 감지가 구성된 일반적인 Active Directory):

```xml
<my_ad_server>
    <host>localhost</host>
    <port>389</port>
    <bind_dn>EXAMPLE\{user_name}</bind_dn>
    <user_dn_detection>
        <base_dn>CN=Users,DC=example,DC=com</base_dn>
        <search_filter>(&amp;(objectClass=user)(sAMAccountName={user_name}))</search_filter>
    </user_dn_detection>
    <enable_tls>no</enable_tls>
</my_ad_server>
```

| `%H` | 시를 10진수로 표시, 24시간제 (범위 [00-23])                                                                                                   | `18`                       |

**syslog**

log messages를 syslog에도 추가로 기록하려면 다음과 같이 합니다:

```xml
<logger>
    <use_syslog>1</use_syslog>
    <syslog>
        <address>syslog.remote:10514</address>
        <hostname>myhost.local</hostname>
        <facility>LOG_LOCAL6</facility>
        <format>syslog</format>
    </syslog>
</logger>
```

`<syslog>`의 키:

| Key        | Description                                                                                                                                                                                                       |
| ---------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `address`  | `host\[:port\]` 포맷의 syslog 주소입니다. 생략하면 로컬 데몬을 사용합니다.                                                                                                                                                              |
| `hostname` | 로그를 전송하는 원본 호스트명입니다(선택 사항).                                                                                                                                                                                       |
| `facility` | syslog [facility 키워드](https://en.wikipedia.org/wiki/Syslog#Facility)입니다. `LOG_USER`, `LOG_DAEMON`, `LOG_LOCAL3` 등과 같이 `LOG_` 접두사가 붙은 대문자로 지정해야 합니다. 기본값은 `address`가 지정된 경우 `LOG_USER`이고, 그렇지 않으면 `LOG_DAEMON`입니다. |
| `format`   | 로그 메시지 포맷입니다. 가능한 값: `bsd` 및 `syslog`.                                                                                                                                                                            |

**로그 포맷**

콘솔 로그에 출력할 로그 포맷을 지정할 수 있습니다. 현재는 JSON만 지원됩니다.

**예시**

다음은 출력되는 JSON 로그의 예시입니다:

```json
{
  "date_time_utc": "2024-11-06T09:06:09Z",
  "date_time": "1650918987.180175",
  "thread_name": "#1",
  "thread_id": "254545",
  "level": "Trace",
  "query_id": "",
  "logger_name": "BaseDaemon",
  "message": "Received signal 2",
  "source_file": "../base/daemon/BaseDaemon.cpp; virtual void SignalListener::run()",
  "source_line": "192"
}
```

JSON 로깅을 사용하려면 다음 스니펫을 사용하세요:

```xml
<logger>
    <formatting>
        <type>json</type>
        <!-- Can be configured on a per-channel basis (log, errorlog, console, syslog), or globally for all channels (then just omit it). -->
        <!-- <channel></channel> -->
        <names>
            <date_time>date_time</date_time>
            <thread_name>thread_name</thread_name>
            <thread_id>thread_id</thread_id>
            <level>level</level>
            <query_id>query_id</query_id>
            <logger_name>logger_name</logger_name>
            <message>message</message>
            <source_file>source_file</source_file>
            <source_line>source_line</source_line>
        </names>
    </formatting>
</logger>
```

**JSON 로그의 키 이름 변경**

키 이름은 `<names>` 태그 안의 태그 값을 변경하여 수정할 수 있습니다. 예를 들어 `DATE_TIME`를 `MY_DATE_TIME`로 바꾸려면 `<date_time>MY_DATE_TIME</date_time>`를 사용할 수 있습니다.

**JSON 로그의 키 생략**

속성을 주석 처리하면 JSON 로그에서 해당 로그 속성을 생략할 수 있습니다. 예를 들어 로그에 `query_id`가 출력되지 않도록 하려면 `<query_id>` 태그를 주석 처리하면 됩니다.

<div id="mmap_cache_size">
  ## mmap_cache_size
</div>

<SettingsInfoBlock type="UInt64" default_value="1024" changeable_without_restart="Yes" />

이 설정을 사용하면 빈번한 open/close 호출(연이은 페이지 폴트로 인해 비용이 매우 큼)을 줄이고, 여러 스레드와 쿼리에서 매핑을 재사용할 수 있습니다. 설정값은 매핑된 영역의 개수(보통 매핑된 파일 수와 동일)입니다.

매핑된 파일의 데이터 양은 다음 시스템 테이블에서 다음 메트릭으로 모니터링할 수 있습니다.

* [`system.metrics`](/ko/reference/system-tables/metrics), [`system.metric_log`](/ko/reference/system-tables/metric_log)의 `MMappedFiles`/`MMappedFileBytes`/`MMapCacheCells`
* [`system.events`](/ko/reference/system-tables/events), [`system.processes`](/ko/reference/system-tables/processes), [`system.query_log`](/ko/reference/system-tables/query_log), [`system.query_thread_log`](/ko/reference/system-tables/query_thread_log), [`system.query_views_log`](/ko/reference/system-tables/query_views_log)의 `CreatedReadBufferMMap`/`CreatedReadBufferMMapFailed`/`MMappedFileCacheHits`/`MMappedFileCacheMisses`

<Note>
  매핑된 파일의 데이터 양은 메모리를 직접 소비하지 않으며, 쿼리 또는 server 메모리 사용량에도 포함되지 않습니다. 이는 이 메모리가 OS 페이지 캐시와 유사하게 폐기될 수 있기 때문입니다. 이 캐시는 MergeTree 엔진 계열 테이블에서 오래된 파트가 제거되면 자동으로 해제되고(파일이 닫힘), `SYSTEM DROP MMAP CACHE` 쿼리로 수동 해제할 수도 있습니다.

  이 설정은 런타임에 변경할 수 있으며 즉시 적용됩니다.
</Note>

각 스레드가 공유 per-CPU 예산에 접근하지 않고 보유할 수 있는 추적되지 않은 메모리의 양(바이트)입니다. 이는 작은 메모리 할당에 대한 per-CPU bookkeeping 비용을 분산시키며, 최악의 경우 `max_per_cpu_untracked_memory * number_of_cpus` 위에 추가되는 여유분입니다. 값이 `0`이면 이 여유분이 제거됩니다. 즉, 모든 allocation이 공유 per-CPU 카운터를 업데이트합니다(가장 정확하지만 contention이 더 커집니다). Linux에서만 지원됩니다.

<div id="point_in_polygon_cache_size">
  ## point_in_polygon_cache_size
</div>

<SettingsInfoBlock type="UInt64" default_value="268435456" changeable_without_restart="Yes" />

상수 다각형 인수를 사용하는 `pointInPolygon` 함수에서 사용되는 전처리된 다각형 캐시의 최대 크기(바이트)입니다.

<div id="proxy">
  ## 프록시
</div>

현재 S3 스토리지, S3 테이블 함수 및 URL 함수에서 지원하는 HTTP 및 HTTPS 요청용 프록시 서버를 정의합니다.

프록시 서버를 정의하는 방법은 세 가지입니다.

* 환경 변수
* 프록시 목록
* 원격 프록시 리졸버

특정 호스트에 대해서는 `no_proxy`를 사용해 프록시 서버를 우회할 수도 있습니다.

**환경 변수**

`http_proxy` 및 `https_proxy` 환경 변수를 사용하면 지정한
프로토콜에 대한 프록시 서버를 설정할 수 있습니다. 시스템에 설정되어 있다면 별도 설정 없이 원활하게 작동합니다.

이 방법은 지정한 프로토콜에
프록시 서버가 하나뿐이고 해당 프록시 서버가 변경되지 않을 때 가장 간단합니다.

**프록시 목록**

이 방법을 사용하면 지정한 프로토콜에 대해 하나 이상의
프록시 서버를 설정할 수 있습니다. 프록시 서버가 둘 이상 정의되면
ClickHouse는 서로 다른 프록시를 라운드 로빈 방식으로 사용하여
서버 간에 부하를 분산합니다. 이 방법은 프로토콜에 대해 프록시 서버가 둘 이상 있고
프록시 서버 목록이 변경되지 않을 때 가장 간단합니다.

**구성 템플릿**

```xml
<proxy>
    <http>
        <uri>http://proxy1</uri>
        <uri>http://proxy2:3128</uri>
    </http>
    <https>
        <uri>http://proxy1:3128</uri>
    </https>
</proxy>
```

아래 탭에서 상위 필드를 선택하면 해당 하위 필드를 볼 수 있습니다:

<Tabs>
  <Tab title="<proxy>">
    | 필드        | 설명                  |
    | --------- | ------------------- |
    | `<http>`  | 하나 이상의 HTTP 프록시 목록  |
    | `<https>` | 하나 이상의 HTTPS 프록시 목록 |
  </Tab>

  <Tab title="<http> 및 <https>">
    | 필드      | 설명       |
    | ------- | -------- |
    | `<uri>` | 프록시의 URI |
  </Tab>
</Tabs>

**원격 프록시 리졸버**

프록시 서버는 동적으로 변경될 수 있습니다. 이 경우
리졸버의 엔드포인트를 정의할 수 있습니다. ClickHouse는
해당 엔드포인트로 본문이 없는 GET 요청을 보내고, 원격 리졸버는 프록시 호스트를 반환해야 합니다.
ClickHouse는 이를 사용하여 다음 템플릿으로 프록시 URI를 구성합니다: `\{proxy_scheme\}://\{proxy_host\}:{proxy_port}`

**구성 템플릿**

```xml
<proxy>
    <http>
        <resolver>
            <endpoint>http://resolver:8080/hostname</endpoint>
            <proxy_scheme>http</proxy_scheme>
            <proxy_port>80</proxy_port>
            <proxy_cache_time>10</proxy_cache_time>
        </resolver>
    </http>

    <https>
        <resolver>
            <endpoint>http://resolver:8080/hostname</endpoint>
            <proxy_scheme>http</proxy_scheme>
            <proxy_port>3128</proxy_port>
            <proxy_cache_time>10</proxy_cache_time>
        </resolver>
    </https>

</proxy>
```

아래 탭에서 상위 필드를 선택하여 하위 항목을 확인하십시오:

<Tabs>
  <Tab title="<proxy>">
    | Field     | 설명                 |
    | --------- | ------------------ |
    | `<http>`  | 하나 이상의 리졸버 목록* |
    | `<https>` | 하나 이상의 리졸버 목록* |
  </Tab>

  <Tab title="<http> 및 <https>">
    | Field        | 설명                    |
    | ------------ | --------------------- |
    | `<resolver>` | 리졸버의 endpoint 및 기타 정보 |

    <Note>
      여러 개의 `<resolver>` 원소를 둘 수 있지만, 지정된 protocol에 대해서는 첫 번째
      `<resolver>`만 사용됩니다. 해당 protocol의 다른 `<resolver>`
      원소는 모두 무시됩니다. 즉, load balancing이
      필요하면 원격 리졸버에서 구현해야 합니다.
    </Note>
  </Tab>

  <Tab title="<resolver>">
    | Field                | 설명                                                                                                       |
    | -------------------- | -------------------------------------------------------------------------------------------------------- |
    | `<endpoint>`         | 프록시 리졸버의 URI                                                                                             |
    | `<proxy_scheme>`     | 최종 프록시 URI의 protocol입니다. `http` 또는 `https` 중 하나일 수 있습니다.                                                 |
    | `<proxy_port>`       | 프록시 리졸버의 포트 번호                                                                                           |
    | `<proxy_cache_time>` | 리졸버의 값을 ClickHouse가 캐시해야 하는 시간(초)입니다. 이 값을 `0`으로 설정하면 ClickHouse는 모든 HTTP 또는 HTTPS request마다 리졸버에 연결합니다. |
  </Tab>
</Tabs>

**우선순위**

프록시 설정은 다음 순서로 결정됩니다:

| Order | Setting    |
| ----- | ---------- |
| 1.    | 원격 프록시 리졸버 |
| 2.    | 프록시 목록     |
| 3.    | 환경 변수      |

ClickHouse는 요청 프로토콜에 대해 우선순위가 가장 높은 리졸버 유형을 확인합니다. 해당 항목이 정의되어 있지 않으면,
환경 리졸버에 도달할 때까지 그다음으로 우선순위가 높은 리졸버 유형을 차례로 확인합니다.
따라서 리졸버 유형을 혼합하여 사용할 수도 있습니다.

<div id="wait_dictionaries_load_at_startup">
  ## wait_dictionaries_load_at_startup
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

이 설정은 `dictionaries_lazy_load`가 `false`일 때의 동작을 지정합니다.
(`dictionaries_lazy_load`가 `true`이면 이 설정은 아무런 영향이 없습니다.)

`wait_dictionaries_load_at_startup`가 `false`이면 서버는 시작 시 모든 딕셔너리 로드를 시작하며, 로드와 동시에 연결도 수신합니다.
딕셔너리가 쿼리에서 처음 사용될 때 아직 로드되지 않았다면, 해당 쿼리는 딕셔너리 로드가 완료될 때까지 대기합니다.
`wait_dictionaries_load_at_startup`를 `false`로 설정하면 ClickHouse를 더 빠르게 시작할 수 있지만, 일부 쿼리는 더 느리게 실행될 수 있습니다
(일부 딕셔너리가 로드될 때까지 기다려야 하기 때문입니다).

`wait_dictionaries_load_at_startup`가 `true`이면 서버는 시작 시 모든 딕셔너리의 로드가 완료될 때까지
(성공 여부와 관계없이) 대기한 후 연결을 수신합니다.

**예시**

```xml
<wait_dictionaries_load_at_startup>true</wait_dictionaries_load_at_startup>
```
