이 설정은 ClickHouse 서버 구성에 사용되며 ClickHouse 소스 코드에서 자동으로 생성되었습니다.

<div id="max_active_parts_loading_thread_pool_size">
  ## max_active_parts_loading_thread_pool_size
</div>

<SettingsInfoBlock type="UInt64" default_value="64" changeable_without_restart="Yes" />시작 시 활성 데이터 파트 집합(즉, 활성 상태의 파트)을 로드하는 데 사용되는 스레드 수입니다.

<div id="max_authentication_methods_per_user">
  ## max_authentication_methods_per_user
</div>

<SettingsInfoBlock type="UInt64" default_value="100" changeable_without_restart="No" />

사용자를 생성하거나 변경할 때 설정할 수 있는 최대 인증 메서드 수입니다.
이 설정을 변경해도 기존 사용자에게는 영향을 주지 않습니다. 이 설정에서 지정한 한도를 초과하면 인증 관련 CREATE/ALTER 쿼리가 실패합니다.
인증과 관련 없는 CREATE/ALTER 쿼리는 성공합니다.

<Note>
  값이 `0`이면 무제한을 의미합니다.
</Note>

<div id="max_os_cpu_wait_time_ratio_to_drop_connection">
  ## max_os_cpu_wait_time_ratio_to_drop_connection

<div id="max_partition_size_to_drop">
  ## max_partition_size_to_drop
</div>

<SettingsInfoBlock type="UInt64" default_value="50000000000" changeable_without_restart="Yes" />

파티션 삭제 제한입니다.

[MergeTree](/ko/reference/engines/table-engines/mergetree-family/mergetree) 테이블의 크기가 [`max_partition_size_to_drop`](/ko/reference/settings/server-settings/settings/max#max_partition_size_to_drop) 값을 초과하면(바이트 단위) [DROP PARTITION](/ko/reference/statements/alter/partition#drop-partitionpart) 쿼리로 파티션을 삭제할 수 없습니다.
이 설정은 적용할 때 ClickHouse 서버를 재시작할 필요가 없습니다. 제한을 비활성화하는 또 다른 방법은 `<clickhouse-path>/flags/force_drop_table` 파일을 생성하는 것입니다.

<Note>
  값 `0`은 아무 제한 없이 파티션을 삭제할 수 있음을 의미합니다.

  이 제한은 drop table 및 truncate table에는 적용되지 않습니다. [max&#95;table&#95;size&#95;to&#95;drop](/ko/reference/settings/session-settings#max_table_size_to_drop)를 참조하십시오.
</Note>

**예시**

```xml
<max_partition_size_to_drop>0</max_partition_size_to_drop>
```

<div id="max_per_cpu_untracked_memory">
  ## max_per_cpu_untracked_memory
</div>

<SettingsInfoBlock type="UInt64" default_value="8388608" changeable_without_restart="Yes" />

한 CPU에서 실행되는 모든 스레드가 메모리 추적기(memory tracker)에 플러시되기 전에 동시에 보유할 수 있는 미추적 메모리의 상한값(바이트 단위)입니다. `max_untracked_memory`가 단일 스레드의 한계를 정하는 반면, 이 설정은 CPU별 총량을 제한하므로 여러 스레드가 스레드별 허용량만큼 누적하여 서버 전체에서 큰 오버커밋이 발생하는 것을 방지합니다. 따라서 전체 미추적 메모리는 대략 `number_of_cpus * max_per_cpu_untracked_memory`로 제한됩니다. 값이 `0`이면 CPU별 제한이 비활성화되며(스레드별 `max_untracked_memory`만 적용됨), Linux에서만 사용할 수 있습니다.

<div id="max_waiting_queries">
  ## max_waiting_queries
</div>

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="Yes" />

동시에 대기할 수 있는 전체 쿼리 수의 한도입니다.
필요한 테이블이 비동기로 로드되는 동안에는 대기 중인 쿼리의 실행이 차단됩니다([`async_load_databases`](/ko/reference/settings/server-settings/settings/async-load#async_load_databases) 참조).

<Note>
  다음 설정으로 제어되는 한도를 확인할 때는 대기 중인 쿼리를 계산에 포함하지 않습니다:

  * [`max_concurrent_queries`](/ko/reference/settings/server-settings/settings/max-concurrent#max_concurrent_queries)
  * [`max_concurrent_insert_queries`](/ko/reference/settings/server-settings/settings/max-concurrent#max_concurrent_insert_queries)
  * [`max_concurrent_select_queries`](/ko/reference/settings/server-settings/settings/max-concurrent#max_concurrent_select_queries)
  * [`max_concurrent_queries_for_user`](/ko/reference/settings/session-settings#max_concurrent_queries_for_user)
  * [`max_concurrent_queries_for_all_users`](/ko/reference/settings/session-settings#max_concurrent_queries_for_all_users)

  이 보정은 server 시작 직후 이러한 한도에 도달하는 일을 방지하기 위해 적용됩니다.
</Note>

<Note>
  `0` 값(기본값)은 한도가 없음을 의미합니다.

  이 설정은 런타임에 수정할 수 있으며 즉시 적용됩니다. 이미 실행 중인 쿼리에는 영향을 주지 않습니다.
</Note>
