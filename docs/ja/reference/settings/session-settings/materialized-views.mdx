<div id="materialized_views_ignore_errors">
  ## materialized_views_ignore_errors
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

有効にすると、依存する materialized view へのデータのプッシュ中 (その `SELECT` または内部テーブル sink 内) に発生した例外は警告として記録され、`INSERT` ステートメントは成功します。無効な場合 (既定値) 、このような例外は伝播し、`INSERT` ステートメントは失敗します。

この設定が制御するのはエラー報告のみです。source table への書き込みはロールバックされず、依存 view の pipeline でエラーが発生した時点で、元の block がすでに source table に commit 済みかどうかも保証されません。無効な場合 (既定値) 、view のエラーにより `INSERT` は失敗します。この場合は、source table とすべての dependent views に対して exactly-once 配信を実現するため、挿入の重複排除 (`insert_deduplicate`, `deduplicate_blocks_in_dependent_materialized_views`) を使用して再試行してください。有効な場合、失敗した view とその下流の chains には部分的にしか配信されていなくても、`INSERT` は成功として報告されます。これは、view 側の問題によって source table への書き込みをブロックしてはならない場合にのみ使用してください (たとえば `system.*_log` tables) 。完全な意味については、`CREATE VIEW` のドキュメントを参照してください。

<div id="materialized_views_squash_parallel_inserts">
  ## materialized_views_squash_parallel_inserts
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "25.10"},{"label": "1"},{"label": "必要に応じて従来の動作を維持するための設定を追加しました。"}]}]} />

生成されるパーツ数を減らすため、単一の INSERT クエリによる並列 insert で materialized view の宛先テーブルに対して行われる insert をまとめます。
false に設定され、`parallel_view_processing` が有効な場合、INSERT クエリは `max_insert_thread` ごとに宛先テーブルにパーツを生成します。
