これらの設定はClickHouseサーバーを構成するもので、ClickHouseのソースコードから自動生成されています。

<div id="memory_worker_correct_memory_tracker">
  ## memory_worker_correct_memory_tracker
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

バックグラウンドメモリワーカーが、jemalloc や cgroups などの外部情報に基づいて内部のメモリトラッカーを補正するかどうか

<div id="memory_worker_decay_adjustment_period_ms">
  ## memory_worker_decay_adjustment_period_ms
</div>

<SettingsInfoBlock type="UInt64" default_value="5000" changeable_without_restart="No" />

メモリ逼迫が継続してから jemalloc の `dirty_decay_ms` を動的に調整するまでの期間 (Milliseconds) です。メモリ使用量がこの期間にわたって purge のしきい値を上回り続けると、メモリを積極的に回収するため、ダーティページの自動減衰は無効になります (`dirty_decay_ms=0`) 。一方、使用量がこの期間にわたってしきい値を下回り続けると、既定の減衰動作が復元されます。0 に設定すると、動的調整を無効にして jemalloc の既定の減衰設定を使用します。

<div id="memory_worker_dynamic_hard_limit">
  ## memory_worker_dynamic_hard_limit
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

バックグラウンドメモリワーカーが、実行時にサーバーのハードメモリ制限を `(resident memory + system available memory) * max_server_memory_usage_to_ram_ratio` として定期的に再計算するかどうかを指定します。これにより、同じホスト上で実行されているほかのプロセスのために、サーバーがヘッドルームを確保できます。

無効にすると、`max_server_memory_usage_to_ram_ratio` は以前のバージョンと同様、起動時および設定の再読み込み時にのみ、ハードメモリ制限の上限を静的に設定します。

`max_server_memory_usage_to_ram_ratio` が `0` の場合、この設定は効果がありません。

<div id="memory_worker_period_ms">
  ## memory_worker_period_ms
</div>

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="No" />

メモリ使用量が高いときに、メモリトラッカー が記録するメモリ使用量を補正し、未使用ページをクリーンアップするバックグラウンドメモリワーカーのティック間隔です。0 に設定すると、メモリ使用量の取得元に応じたデフォルト値が使用されます

<div id="memory_worker_purge_dirty_pages_threshold_ratio">
  ## memory_worker_purge_dirty_pages_threshold_ratio
</div>

<SettingsInfoBlock type="Double" default_value="0.2" changeable_without_restart="No" />

ClickHouse サーバー で使用可能なメモリに対する jemalloc の ダーティページ のしきい値比率です。ダーティページ のサイズがこの比率を超えると、background memory worker が ダーティページ のパージを強制します。0 に設定すると、ダーティページ の比率に基づく強制パージは無効になります。

<div id="memory_worker_purge_total_memory_threshold_ratio">
  ## memory_worker_purge_total_memory_threshold_ratio
</div>

<SettingsInfoBlock type="Double" default_value="0.9" changeable_without_restart="No" />

ClickHouse サーバー で使用可能なメモリに対して、jemalloc のパージを実行するしきい値の比率です。総メモリ使用量がこの比率を超えると、バックグラウンドメモリワーカーがダーティページのパージを強制的に実行します。0 に設定すると、総メモリに基づく強制パージは無効になります。

<div id="memory_worker_rss_speculative_reserve_ratio">
  ## memory_worker_rss_speculative_reserve_ratio
</div>

<SettingsInfoBlock type="Double" default_value="1" changeable_without_restart="No" />

`MemoryWorker` の各ティックで、観測された RSS に加えて
`ratio * min(resident - previous_resident, resident - tracked)` を追加で予約します。これは、次のティックでも前回と同じ
量だけ増加する可能性があるという前提に基づいています (`resident - previous_resident` は直前のティック
における RSS の増加量です) 。この増加量の上限は `resident - tracked`、つまりグローバルメモリトラッカーからは見えない RSS
の部分です。すでに追跡されている増加分は通常のハードリミットチェックで処理されるためです。この予約は、グローバルなハードリミットチェック
が `MemoryTracker::allocImpl` 経由で参照する `rss` カウンターに適用されるため、外挿値が
`max_server_memory_usage` を超えると、カーネルの OOM-killer が発動する前に、その後の割り当てで
`MEMORY_LIMIT_EXCEEDED` 例外が発生します。値を `0`
にすると予測は無効になり (`rss = resident` に戻ります) 、既定値の `1`
では次の interval に備えて増加量 1 回分のヘッドルームを予約します。サニタイザ (`ASan`, `UBSan`, `MSan`, `TSan`) 環境では既定値は `0`
です。これは、`resident - tracked` のギャップの大半が、トラッカーの計上の遅れではなく、サニタイザのシャドー領域やランタイム
オーバーヘッドによるものだからです。
