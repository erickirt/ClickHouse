<div id="bcrypt_workfactor">
  ## bcrypt_workfactor
</div>

[Bcrypt algorithm](https://wildlyinaccurate.com/bcrypt-choosing-a-work-factor/) を使用する `bcrypt_password` 認証方式のワークファクターです。
ワークファクターは、ハッシュの計算とパスワードの検証に必要な計算量と時間を定義します。

```xml
<bcrypt_workfactor>12</bcrypt_workfactor>
```

<Warning>
  高頻度で認証を行うアプリケーションでは、
  work factor が高い bcrypt では
  計算負荷が大きくなるため、
  別の認証方式の利用を検討してください。
</Warning>

<div id="compression">
  ## 圧縮
</div>

[MergeTree](/ja/reference/engines/table-engines/mergetree-family/mergetree) エンジンのテーブルに対するデータ圧縮の設定です。

<Note>
  ClickHouse を使い始めたばかりの場合は、これを変更しないことをおすすめします。
</Note>

**設定テンプレート**:

```xml
<compression>
    <case>
      <min_part_size>...</min_part_size>
      <min_part_size_ratio>...</min_part_size_ratio>
      <method>...</method>
      <level>...</level>
    </case>
    ...
</compression>
```

**`<case>` フィールド**:

* `min_part_size` – data part の最小サイズ。
* `min_part_size_ratio` – data part のサイズとテーブルサイズの比率。
* `method` – 圧縮方式。有効な値: `lz4`, `lz4hc`, `zstd`,`deflate_qpl`。
* `level` – 圧縮レベル。[Codecs](/ja/reference/statements/create/table#general-purpose-codecs) を参照してください。

<Note>
  `<case>` セクションは複数設定できます。
</Note>

**条件を満たした場合の動作**:

* data part が条件セットに一致した場合、ClickHouse は指定された圧縮方式を使用します。
* data part が複数の条件セットに一致した場合、ClickHouse は最初に一致した条件セットを使用します。

<Note>
  data part がどの条件にも一致しない場合、ClickHouse は `lz4` 圧縮を使用します。
</Note>

**例**

```xml
<compression incl="clickhouse_compression">
    <case>
        <min_part_size>10000000000</min_part_size>
        <min_part_size_ratio>0.01</min_part_size_ratio>
        <method>zstd</method>
        <level>1</level>
    </case>
</compression>
```

[crash&#95;log](/ja/reference/system-tables/crash_log) システムテーブルの動作に関する設定です。

以下の設定はサブタグで構成できます。

| Setting                            | Description                                                                                                                         | Default             | Note                                                                                |
| ---------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- | ------------------- | ----------------------------------------------------------------------------------- |
| `buffer_size_rows_flush_threshold` | 行数のしきい値です。しきい値に達すると、ログをディスクへフラッシュする処理がバックグラウンドで開始されます。                                                                              | `max_size_rows / 2` |                                                                                     |
| `database`                         | データベース名。                                                                                                                            |                     |                                                                                     |
| `engine`                           | システムテーブル用の [MergeTree エンジン定義](/ja/reference/engines/table-engines/mergetree-family/mergetree#table_engine-mergetree-creating-a-table)。 |                     | `partition_by` または `order_by` が定義されている場合は使用できません。指定しない場合は、既定で `MergeTree` が選択されます   |
| `flush_interval_milliseconds`      | メモリ上のバッファからテーブルへデータをフラッシュする間隔です。                                                                                                    | `7500`              |                                                                                     |
| `flush_on_crash`                   | クラッシュ時にログをディスクへダンプするかどうかを設定します。                                                                                                     | `false`             |                                                                                     |
| `max_size_rows`                    | ログの最大行数です。未フラッシュのログ数が max&#95;size に達すると、ログがディスクへダンプされます。                                                                           | `1024`              |                                                                                     |
| `order_by`                         | システムテーブル用の [カスタムソートキー](/ja/reference/engines/table-engines/mergetree-family/mergetree#order_by)。`engine` が定義されている場合は使用できません。           |                     | システムテーブルに `engine` を指定する場合、`order_by` パラメータは &#39;engine&#39; 内に直接指定する必要があります       |
| `partition_by`                     | システムテーブル用の [カスタムパーティションキー](/ja/reference/engines/table-engines/mergetree-family/custom-partitioning-key)。                              |                     | システムテーブルに `engine` を指定する場合、`partition_by` パラメータは &#39;engine&#39; 内に直接指定する必要があります   |
| `reserved_size_rows`               | ログ用に事前確保されるメモリの行数です。                                                                                                                | `1024`              |                                                                                     |
| `settings`                         | MergeTree の動作を制御する [追加パラメータ](/ja/reference/engines/table-engines/mergetree-family/mergetree#settings) (省略可) 。                          |                     | システムテーブルに `engine` を指定する場合、`settings` パラメータは &#39;engine&#39; 内に直接指定する必要があります       |
| `storage_policy`                   | テーブルに使用するストレージポリシー名 (省略可) 。                                                                                                         |                     | システムテーブルに `engine` を指定する場合、`storage_policy` パラメータは &#39;engine&#39; 内に直接指定する必要があります |
| `table`                            | システムテーブル名。                                                                                                                          |                     |                                                                                     |
| `ttl`                              | テーブルの [有効期限 (TTL)](/ja/reference/engines/table-engines/mergetree-family/mergetree#table_engine-mergetree-ttl) を指定します。                  |                     | システムテーブルに `engine` を指定する場合、`ttl` パラメータは &#39;engine&#39; 内に直接指定する必要があります            |

<div id="encryption">
  ## 暗号化
</div>

[暗号化コーデック](/ja/reference/statements/create/table#encryption-codecs)で使用するキーを取得するコマンドを設定します。キーは環境変数に記述するか、設定ファイルで設定する必要があります。

キーには、16バイト長の16進数表現または文字列を指定できます。

**例**

config から読み込む場合:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key>1234567812345678</key>
    </aes_128_gcm_siv>
</encryption_codecs>
```

<Note>
  秘密鍵を設定ファイルに保存することは推奨されません。安全ではないためです。秘密鍵は安全なディスク上の別の設定ファイルに移し、その設定ファイルへのシンボリックリンクを `config.d/` フォルダーに配置できます。
</Note>

秘密鍵が16進数の場合に設定から読み込む例:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex>00112233445566778899aabbccddeeff</key_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

環境変数から秘密鍵を読み込む:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex from_env="ENVVAR"></key_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

ここで、`current_key_id` は暗号化に使用する現在のキーを設定し、指定したすべてのキーを復号に使用できます。

これらの各方法は、複数のキーに適用できます。

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex id="0">00112233445566778899aabbccddeeff</key_hex>
        <key_hex id="1" from_env="ENVVAR"></key_hex>
        <current_key_id>1</current_key_id>
    </aes_128_gcm_siv>
</encryption_codecs>
```

ここで `current_key_id` は、暗号化に使用中の現在の鍵を示します。

また、ユーザーは長さが12バイトの nonce を追加できます (デフォルトでは、暗号化および復号の処理にはゼロバイトのみで構成された nonce が使用されます) :

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <nonce>012345678910</nonce>
    </aes_128_gcm_siv>
</encryption_codecs>
```

また、16進数で設定することもできます:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <nonce_hex>abcdefabcdef</nonce_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

<Note>
  前述の内容はすべて `aes_256_gcm_siv` にも適用できます (ただし、キーの長さは 32 バイトである必要があります) 。
</Note>

```xml
    <error_log remove="1" />
</clickhouse>
```

<SettingsInfoBlock type="String" default_value="new_unified_hash" changeable_without_restart="No" />非推奨の移行ガードです。このバージョンでサポートされる統一された挿入の重複排除ハッシュは `new_unified_hash` のみであり、この設定がそれ以外の値 (`old_separate_hashes` や `compatible_double_hashes` など) で指定されている場合、サーバーは起動を拒否します。アップグレード前に、前のバージョンで `compatible_double_hashes` (legacy ハッシュと統一ハッシュの両方を書き込みます) を実行し、重複排除の移行を完了してください。レプリケートテーブルでは、少なくとも `replicated_deduplication_window_seconds` (デフォルトでは 1 時間) の間これを実行してください。デフォルトの window は、その期間内のすべての insert の統一ハッシュを保持するため、insert の再試行ループをカバーするには十分と見なされます。`non_replicated_deduplication_window` &gt; 0 の非レプリケートテーブルでは、window は時間ベースではなく件数ベースになるため、アップグレード前に少なくともその件数の insert に対して `compatible_double_hashes` を実行してください。

<div id="ldap_servers">
  ## ldap_servers
</div>

ここでは、接続パラメータとともに LDAP サーバーを列挙します。用途は次のとおりです。

* `'password'` ではなく `'ldap'` 認証メカニズムが指定された専用のローカルユーザーに対して、authenticator として使用する
* リモートの user directories として使用する。

以下の設定はサブタグで構成できます。

| Setting                        | Description                                                                                                                                                                                                                                                                                 |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `bind_dn`                      | バインド先の DN を構築するためのテンプレートです。認証の試行ごとに、テンプレート内のすべての `\{user_name\}` substring を実際のユーザー名に置き換えて、最終的な DN が構築されます。                                                                                                                                                                                 |
| `enable_tls`                   | LDAP サーバーへの secure connection の使用を有効にするフラグです。平文の (`ldap://`) プロトコルには `no` を指定します (非推奨) 。SSL/TLS 上の LDAP (`ldaps://`) プロトコルには `yes` を指定します (推奨、デフォルト) 。従来の StartTLS プロトコルには `starttls` を指定します (平文の (`ldap://`) プロトコルを TLS に upgrade します) 。                                                   |
| `host`                         | LDAP サーバーの hostname または IP です。この parameter は必須で、空にすることはできません。                                                                                                                                                                                                                               |
| `port`                         | LDAP サーバーのポートです。`enable_tls` が true に設定されている場合のデフォルトは 636、それ以外は `389` です。                                                                                                                                                                                                                   |
| `tls_ca_cert_dir`              | CA certificates を含む directory への path です。                                                                                                                                                                                                                                                   |
| `tls_ca_cert_file`             | CA certificate ファイルへの path です。                                                                                                                                                                                                                                                              |
| `tls_cert_file`                | certificate ファイルへの path です。                                                                                                                                                                                                                                                                 |
| `tls_cipher_suite`             | 使用を許可する暗号スイートです (OpenSSL 表記) 。                                                                                                                                                                                                                                                              |
| `tls_key_file`                 | certificate の秘密鍵ファイルへの path です。                                                                                                                                                                                                                                                             |
| `tls_minimum_protocol_version` | SSL/TLS の minimum protocol version です。指定可能な値は `ssl2`、`ssl3`、`tls1.0`、`tls1.1`、`tls1.2` (デフォルト) です。                                                                                                                                                                                          |
| `tls_require_cert`             | SSL/TLS ピアの certificate verification の動作です。指定可能な値は `never`、`allow`、`try`、`demand` (デフォルト) です。                                                                                                                                                                                               |
| `user_dn_detection`            | バインドされたユーザーの実際の ユーザー DN を検出するための LDAP search parameter を含むセクションです。主に、サーバーが Active Directory の場合に、以降の role mapping 用の search filter で使用されます。取得された ユーザー DN は、使用可能な箇所で `\{user_dn\}` substring を置き換える際に使われます。デフォルトでは user DN は bind DN と同じ値に設定されますが、search が実行されると、実際に検出された ユーザー DN の値に更新されます。 |
| `verification_cooldown`        | bind の試行が成功した後、LDAP サーバーに接続することなく、その後の連続するすべてのリクエストでユーザーが正常に authenticated されたものとみなされる期間です。単位は秒です。Caching を無効にし、各認証リクエストごとに LDAP サーバーへ接続するようにするには、`0` (デフォルト) を指定します。                                                                                                                       |

`user_dn_detection` 設定はサブタグで構成できます。

| Setting         | Description                                                                                                                                                                                                          |
| --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `base_dn`       | LDAP search の ベース DN を構築するためのテンプレートです。LDAP search の実行時に、テンプレート内のすべての `\{user_name\}` および `\{bind_dn\}` substring を実際のユーザー名とバインド DN に置き換えて、最終的な DN が構築されます。                                                           |
| `scope`         | LDAP search のスコープです。指定可能な値は `base`、`one_level`、`children`、`subtree` (デフォルト) です。                                                                                                                                      |
| `search_filter` | LDAP search の 検索フィルタ を構築するためのテンプレートです。LDAP search の実行時に、テンプレート内のすべての `\{user_name\}`、`\{bind_dn\}`、`\{base_dn\}` substring を実際のユーザー名、バインド DN、ベース DN に置き換えて、最終的な filter が構築されます。なお、特殊文字は XML 内で適切に escaped する必要があります。 |

例:

```xml
<my_ldap_server>
    <host>localhost</host>
    <port>636</port>
    <bind_dn>uid={user_name},ou=users,dc=example,dc=com</bind_dn>
    <verification_cooldown>300</verification_cooldown>
    <enable_tls>yes</enable_tls>
    <tls_minimum_protocol_version>tls1.2</tls_minimum_protocol_version>
    <tls_require_cert>demand</tls_require_cert>
    <tls_cert_file>/path/to/tls_cert_file</tls_cert_file>
    <tls_key_file>/path/to/tls_key_file</tls_key_file>
    <tls_ca_cert_file>/path/to/tls_ca_cert_file</tls_ca_cert_file>
    <tls_ca_cert_dir>/path/to/tls_ca_cert_dir</tls_ca_cert_dir>
    <tls_cipher_suite>ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:AES256-GCM-SHA384</tls_cipher_suite>
</my_ldap_server>
```

例 (一般的な Active Directory で、後続のロールマッピング用にユーザー DN の検出が設定されている場合) :

```xml
<my_ad_server>
    <host>localhost</host>
    <port>389</port>
    <bind_dn>EXAMPLE\{user_name}</bind_dn>
    <user_dn_detection>
        <base_dn>CN=Users,DC=example,DC=com</base_dn>
        <search_filter>(&amp;(objectClass=user)(sAMAccountName={user_name}))</search_filter>
    </user_dn_detection>
    <enable_tls>no</enable_tls>
</my_ad_server>
```

コンソールログに出力するログフォーマットを指定できます。現在サポートされているのは JSON のみです。

```json
  "date_time": "1650918987.180175",

```

JSONロギングを有効にするには、次のスニペットを使用します。

```xml
<logger>
    <formatting>
        <type>json</type>

        <!-- <channel></channel> -->
        <names>
            <date_time>date_time</date_time>
            <thread_name>thread_name</thread_name>
            <thread_id>thread_id</thread_id>
            <level>level</level>
            <query_id>query_id</query_id>
            <logger_name>logger_name</logger_name>
            <message>message</message>
            <source_file>source_file</source_file>
            <source_line>source_line</source_line>
        </names>
    </formatting>
</logger>
```

**JSONログのキー名を変更する**

キー名は、`<names>` タグ内の各タグの値を変更することで変えられます。たとえば、`DATE_TIME` を `MY_DATE_TIME` に変更するには、`<date_time>MY_DATE_TIME</date_time>` を使用します。

**JSONログのキーを省略する**

ログのプロパティは、該当するプロパティをコメントアウトすることで省略できます。たとえば、ログに `query_id` を出力したくない場合は、`<query_id>` タグをコメントアウトします。

<div id="mmap_cache_size">
  ## mmap_cache_size
</div>

<SettingsInfoBlock type="UInt64" default_value="1024" changeable_without_restart="Yes" />

この設定により、頻繁な open/close 呼び出し (それに伴うページフォールトのため非常に高コスト) を回避し、複数のスレッドやクエリ間でマッピングを再利用できます。設定値は、マップされた領域の数です (通常はマップされたファイル数と同じです) 。

マップされたファイル内のデータ量は、次のシステムテーブルで以下のメトリクスを使って監視できます。

* [`system.metrics`](/ja/reference/system-tables/metrics)、[`system.metric_log`](/ja/reference/system-tables/metric_log) の `MMappedFiles`/`MMappedFileBytes`/`MMapCacheCells`
* [`system.events`](/ja/reference/system-tables/events)、[`system.processes`](/ja/reference/system-tables/processes)、[`system.query_log`](/ja/reference/system-tables/query_log)、[`system.query_thread_log`](/ja/reference/system-tables/query_thread_log)、[`system.query_views_log`](/ja/reference/system-tables/query_views_log) の `CreatedReadBufferMMap`/`CreatedReadBufferMMapFailed`/`MMappedFileCacheHits`/`MMappedFileCacheMisses`

<Note>
  マップされたファイル内のデータ量はメモリを直接消費せず、クエリやサーバーのメモリ使用量にも計上されません。これは、このメモリが OS page cache と同様に破棄できるためです。MergeTree family のテーブルでは、古いパーツが削除されるとこの cache は自動的に破棄され (ファイルはクローズされます) 、`SYSTEM DROP MMAP CACHE` クエリで手動で破棄することもできます。

  この設定は実行時に変更でき、すぐに反映されます。
</Note>

```xml

        <certificateFile>/etc/clickhouse-server/server.crt</certificateFile>
        <privateKeyFile>/etc/clickhouse-server/server.key</privateKeyFile>
        <!-- openssl dhparam -out /etc/clickhouse-server/dhparam.pem 4096 -->

        <verificationMode>none</verificationMode>
        <loadDefaultCAFile>true</loadDefaultCAFile>
        <cacheSessions>true</cacheSessions>
        <disableProtocols>sslv2,sslv3</disableProtocols>
        <preferServerCiphers>true</preferServerCiphers>
    </server>
    <client>
        <loadDefaultCAFile>true</loadDefaultCAFile>
        <cacheSessions>true</cacheSessions>
        <disableProtocols>sslv2,sslv3</disableProtocols>
        <preferServerCiphers>true</preferServerCiphers>
        <!-- Use for self-signed: <verificationMode>none</verificationMode> -->
        <invalidCertificateHandler>
            <!-- Use for self-signed: <name>AcceptCertificateHandler</name> -->
            <name>RejectCertificateHandler</name>
        </invalidCertificateHandler>
    </client>
</openSSL>
```

<div id="proxy">
  ## proxy
</div>

HTTP および HTTPS のリクエスト用のプロキシサーバーを定義します。現在サポートされているのは、S3 storage、S3 table functions、および URL 関数です。

プロキシサーバーを定義する方法は 3 つあります。

* 環境変数
* プロキシリスト
* リモートプロキシリゾルバ。

また、`no_proxy` を使用して特定のホストに対するプロキシサーバーの使用を回避することもできます。

**環境変数**

`http_proxy` および `https_proxy` 環境変数を使用すると、指定した
プロトコル用のプロキシサーバーを指定できます。これらがシステムで設定されていれば、そのまま問題なく動作するはずです。

これは、あるプロトコルに対して
プロキシサーバーが 1 つしかなく、そのプロキシサーバーが変わらない場合に最も簡単な方法です。

**プロキシリスト**

この方法では、プロトコルごとに 1 つ以上の
プロキシサーバーを指定できます。複数のプロキシサーバーが定義されている場合、
ClickHouse はそれらをラウンドロビン方式で使用し、
各サーバーに負荷を分散します。これは、あるプロトコルに対して
複数のプロキシサーバーがあり、そのリストが変わらない場合に最も簡単な方法です。

**設定テンプレート**

```xml
<proxy>
    <http>
        <uri>http://proxy1</uri>
        <uri>http://proxy2:3128</uri>
    </http>
    <https>
        <uri>http://proxy1:3128</uri>
    </https>
</proxy>
```

子フィールドを表示するには、以下のタブで親フィールドを選択します。

<Tabs>
  <Tab title="<proxy>">
    | フィールド     | 説明                    |
    | --------- | --------------------- |
    | `<http>`  | 1 つ以上の HTTP プロキシのリスト  |
    | `<https>` | 1 つ以上の HTTPS プロキシのリスト |
  </Tab>

  <Tab title="<http> と <https>">
    | フィールド   | 説明        |
    | ------- | --------- |
    | `<uri>` | プロキシの URI |
  </Tab>
</Tabs>

**リモートプロキシリゾルバ**

プロキシサーバーが動的に変わる場合があります。その場合は、リゾルバのエンドポイントを定義できます。ClickHouse はそのエンドポイントに空の GET リクエストを送信し、リモートリゾルバはプロキシホストを返す必要があります。
ClickHouse はそれを使用して、次のテンプレートでプロキシ URI を構築します: `\{proxy_scheme\}://\{proxy_host\}:{proxy_port}`

**設定テンプレート**

```xml
<proxy>
    <http>
        <resolver>
            <endpoint>http://resolver:8080/hostname</endpoint>
            <proxy_scheme>http</proxy_scheme>
            <proxy_port>80</proxy_port>
            <proxy_cache_time>10</proxy_cache_time>
        </resolver>
    </http>

    <https>
        <resolver>
            <endpoint>http://resolver:8080/hostname</endpoint>
            <proxy_scheme>http</proxy_scheme>
            <proxy_port>3128</proxy_port>
            <proxy_cache_time>10</proxy_cache_time>
        </resolver>
    </https>

</proxy>
```

以下のタブで親フィールドを選択すると、対応する子要素を確認できます。

<Tabs>
  <Tab title="<proxy>">
    | フィールド     | 説明                  |
    | --------- | ------------------- |
    | `<http>`  | 1 つ以上のリゾルバ*のリスト |
    | `<https>` | 1 つ以上のリゾルバ*のリスト |
  </Tab>

  <Tab title="<http> and <https>">
    | フィールド        | 説明                     |
    | ------------ | ---------------------- |
    | `<resolver>` | リゾルバの エンドポイント とその他の詳細 |

    <Note>
      `<resolver>` 要素は複数指定できますが、特定のプロトコルに対して使用されるのは
      最初の `<resolver>` のみです。そのプロトコルに対するそれ以外の `<resolver>`
      要素は無視されます。つまり、負荷分散
      (必要な場合) はリモートリゾルバ側で実装する必要があります。
    </Note>
  </Tab>

  <Tab title="<resolver>">
    | フィールド                | 説明                                                                                                                |
    | -------------------- | ----------------------------------------------------------------------------------------------------------------- |
    | `<endpoint>`         | プロキシリゾルバの URI                                                                                                     |
    | `<proxy_scheme>`     | 最終的なプロキシ URI のプロトコルです。`http` または `https` のいずれかを指定できます。                                                            |
    | `<proxy_port>`       | プロキシリゾルバのポート番号                                                                                                    |
    | `<proxy_cache_time>` | リゾルバからの値を ClickHouse が cache する期間 (秒単位) です。この値を `0` に設定すると、ClickHouse は HTTP または HTTPS のすべてのリクエストごとにリゾルバへ問い合わせます。 |
  </Tab>
</Tabs>

**優先順位**

プロキシ設定は、次の順序で決定されます。

| 順序 | 設定           |
| -- | ------------ |
| 1. | リモートプロキシリゾルバ |
| 2. | プロキシリスト      |
| 3. | 環境変数         |

ClickHouse は、リクエストのプロトコルに対して最も優先度の高い種類のリゾルバを確認します。定義されていない場合は、
環境リゾルバに達するまで、次に優先度の高い種類のリゾルバを順に確認します。
これにより、複数の種類のリゾルバを組み合わせて使用することもできます。

**例**

```xml
<throw_on_unknown_workload>true</throw_on_unknown_workload>
```

<div id="wait_dictionaries_load_at_startup">
  ## wait_dictionaries_load_at_startup
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

この設定では、`dictionaries_lazy_load` が `false` の場合の動作を指定できます。
(`dictionaries_lazy_load` が `true` の場合、この設定は何にも影響しません。)

`wait_dictionaries_load_at_startup` が `false` の場合、サーバーは起動時にすべての Dictionary のロードを開始し、そのロードと並列に接続を受け付けます。
Dictionary がクエリで初めて使用されたとき、まだロードされていなければ、そのクエリは Dictionary のロードが完了するまで待機します。
`wait_dictionaries_load_at_startup` を `false` に設定すると ClickHouse はより速く起動できますが、一部のクエリは実行が遅くなる可能性があります
(これは、一部の Dictionary のロード完了を待つ必要があるためです) 。

`wait_dictionaries_load_at_startup` が `true` の場合、サーバーは起動時に
すべての Dictionary のロードが完了するまで (成功・失敗を問わず) 待機し、その後で接続を受け付けます。

**例**

```xml
<wait_dictionaries_load_at_startup>true</wait_dictionaries_load_at_startup>
```
