これらの設定はClickHouseサーバーを構成するもので、ClickHouseのソースから自動生成されています。

<div id="interserver_listen_host">
  ## interserver_listen_host
</div>

ClickHouse servers 間でデータを交換できる hosts を制限します。
Keeper を使用している場合、異なる Keeper インスタンス間の通信にも同じ制限が適用されます。

<Note>
  デフォルトでは、値は [`listen_host`](/ja/reference/settings/server-settings/settings/listen#listen_host) 設定と同じです。
</Note>

**例**

```xml
<interserver_listen_host>::ffff:a00:1</interserver_listen_host>
<interserver_listen_host>10.0.0.1</interserver_listen_host>
```

型:

デフォルト値:

<div id="interserver_tables_status_require_auth">
  ## interserver_tables_status_require_auth
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

interserver `TablesStatusRequest` について、クラスター
`<secret>` による認証を必須にします。シークレットのハッシュを送信できる十分に新しいクライアント (プロトコル リビジョン
`DBMS_MIN_REVISION_WITH_INTERSERVER_SECRET_TABLES_STATUS`) は常に検証されます。この
設定ではさらに、ハッシュを送信しない古いクライアントも拒否します。これにより、未認証の
table-status 情報の漏えいはデフォルトで防止されます。

デフォルト値は `true` です (デフォルトで安全) 。ローリングアップグレード中は、まだ upgrade されていない
ノードは古いプロトコルで通信するため、ハッシュを送信しません。そのため、そのようなノードで開始された
`Distributed` クエリが、すでに upgrade 済みのノードを対象とした場合、その `TablesStatusRequest`
は拒否されます。異なるバージョンが混在するクラスターを運用する必要がある場合は、すべてのノードの upgrade が完了するまで、upgrade 済みの
ノードでこれを `false` に設定し、その後オーバーライドを削除してください。
