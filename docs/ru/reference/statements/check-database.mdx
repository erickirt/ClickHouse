Запрос `CHECK DATABASE` проверяет работоспособность базы данных.

В основном он используется с движком базы данных [`DataLakeCatalog`](/ru/reference/engines/database-engines/datalake): в этом случае проверяется, доступен ли внешний каталог, на котором основана база данных, и можно ли получить список его таблиц. Это легковесная проверка: она подтверждает подключение и аутентификацию, не считывая данные из таблиц.

<div id="syntax">
  ## Синтаксис
</div>

Основной синтаксис запроса выглядит следующим образом:

```sql
CHECK DATABASE database_name
```

* `database_name`: Указывает имя базы данных, которую нужно проверить.

Запрос не возвращает результирующий набор. Если проверка проходит успешно, запрос завершается, не возвращая ни одной строки. Если проверка завершается ошибкой — например, если каталог недоступен или учетные данные больше не действительны, — запрос генерирует исключение с описанием ошибки.

<div id="behavior">
  ## Поведение
</div>

Для базы данных с движком [`DataLakeCatalog`](/ru/reference/engines/database-engines/datalake) команда `CHECK DATABASE`:

* Подключается к внешнему каталогу (например, AWS Glue, Databricks Unity, Hive Metastore или Iceberg REST-каталогу) и получает список таблиц.
* Сообщает об успешном результате, если каталог доступен и аутентификация настроена корректно.
* Не требует, чтобы каталог содержал какие-либо таблицы: пустой, но доступный каталог всё равно считается исправным.

Для движков баз данных, в которых не реализована отдельная проверка работоспособности, `CHECK DATABASE` успешно завершается без выполнения каких-либо действий.

Для выполнения `CHECK DATABASE` требуется привилегия `CHECK` на базу данных (см. [`GRANT`](/ru/reference/statements/grant)).

<div id="examples">
  ## Примеры
</div>

Проверьте базу данных `datalake` типа `DataLakeCatalog`:

```sql title="Query"
CHECK DATABASE datalake;
```

Если каталог доступен, а учетные данные корректны, запрос успешно завершается и не возвращает ни одной строки.

Если к каталогу не удаётся подключиться, запрос вместо этого генерирует исключение. Точное сообщение зависит от типа каталога и причины сбоя (например, ошибки подключения, HTTP-статуса, такого как `401 Unauthorized`, или сбоя при DNS-разрешении).
