Эти настройки доступны в [system.merge&#95;tree&#95;settings](/ru/reference/system-tables/merge_tree_settings) и автоматически генерируются на основе исходного кода ClickHouse.

<div id="concurrent_part_removal_threshold">
  ## concurrent_part_removal_threshold
</div>

<SettingsInfoBlock type="UInt64" default_value="100" />

Включать параллельное удаление частей (см. &#39;max&#95;part&#95;removal&#95;threads&#39;) только если
число неактивных частей данных как минимум равно этому значению.

<div id="concurrent_part_removal_threshold_for_remote_disk">
  ## concurrent_part_removal_threshold_for_remote_disk
</div>

<SettingsInfoBlock type="UInt64" default_value="16" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.5"},{"label": "16"},{"label": "Новая настройка. Более низкий порог для перехода к параллельному удалению частей, когда хотя бы одна удаляемая часть находится на удалённом диске, где каждое удаление обычно требует одного сетевого round-trip. Старое значение (100) соответствует устаревшему значению по умолчанию `concurrent_part_removal_threshold`, поэтому более старые режимы `compatibility` сохраняют прежнее поведение."}]}]} />

То же, что и `concurrent_part_removal_threshold`, но используется, когда хотя бы одна
удаляемая часть хранится на удалённом диске. Значение по умолчанию ниже,
поскольку удаление каждой части в удалённом хранилище обычно требует
одного сетевого round-trip (например, один HTTP `DELETE` на часть в Объектном хранилище), поэтому
последовательное удаление даже 100 частей может задержать `DROP TABLE` на десятки
секунд.
