Эти настройки используются для конфигурации сервера ClickHouse и автоматически генерируются из исходного кода ClickHouse.

<div id="cpu_slot_preemption">
  ## cpu_slot_preemption
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Yes" />

Определяет, как выполняется планирование рабочих нагрузок для ресурсов CPU (MASTER THREAD и WORKER THREAD).

* Если `true` (по умолчанию), учет ведется по фактически потребленному времени CPU. Между конкурирующими рабочими нагрузками справедливо распределяется процессорное время. Слоты выделяются на ограниченное время, а после его истечения запрашиваются заново. При перегрузке ресурсов CPU запрос слота может блокировать выполнение потока, то есть возможна преэмпция. Это обеспечивает справедливое распределение процессорного времени.
* Если `false`, учет ведется по количеству выделенных слотов CPU. Между конкурирующими рабочими нагрузками справедливо распределяется количество слотов CPU. Слот выделяется при запуске потока, удерживается на протяжении всего времени выполнения и освобождается после завершения потока. Число потоков, выделенных для выполнения запроса, может только увеличиваться — от 1 до `max_threads` — и никогда не уменьшается. Этот вариант лучше подходит для длительно выполняющихся запросов, но может привести к дефициту CPU для коротких запросов.

**Пример**

```xml
<cpu_slot_preemption>true</cpu_slot_preemption>
```

<div id="cpu_slot_preemption_timeout_ms">
  ## cpu_slot_preemption_timeout_ms
</div>

<SettingsInfoBlock type="UInt64" default_value="1000" changeable_without_restart="Yes" />

Определяет, сколько миллисекунд воркер-поток может ждать во время преэмпции, то есть в ожидании выделения другого слота CPU. По истечении этого тайм-аута, если поток не смог получить новый слот CPU, он завершится, а число одновременно выполняющихся потоков запроса будет динамически уменьшено. Обратите внимание, что основной поток никогда не уменьшается, но может вытесняться неограниченно долго. Имеет смысл только тогда, когда `cpu_slot_preemption` включен и ресурс CPU определен для WORKER THREAD.

**Пример**

```xml
<cpu_slot_preemption_timeout_ms>1000</cpu_slot_preemption_timeout_ms>
```

**См. также**

* [Планирование рабочих нагрузок](/ru/concepts/features/configuration/server-config/workload-scheduling)

<div id="cpu_slot_quantum_ns">
  ## cpu_slot_quantum_ns
</div>

<SettingsInfoBlock type="UInt64" default_value="10000000" changeable_without_restart="Yes" />

Определяет, сколько наносекунд CPU поток может использовать после получения слота CPU, прежде чем ему потребуется запросить следующий слот CPU. Имеет смысл только при включенном `cpu_slot_preemption` и если ресурс CPU определен для MASTER THREAD или WORKER THREAD.

**Пример**

```xml
<cpu_slot_quantum_ns>10000000</cpu_slot_quantum_ns>
```

* [Планирование рабочих нагрузок](/ru/concepts/features/configuration/server-config/workload-scheduling)
