Эти настройки относятся к серверу ClickHouse и автоматически генерируются на основе исходного кода ClickHouse.

<div id="query_cache">
  ## query_cache
</div>

Конфигурация [кэша запросов](/ru/concepts/features/performance/caches/query-cache).

Доступны следующие настройки:

| Настройка                 | Описание                                                                                                   | Значение по умолчанию |
| ------------------------- | ---------------------------------------------------------------------------------------------------------- | --------------------- |
| `max_entries`             | Максимальное количество результатов запросов `SELECT`, хранящихся в кэше.                                  | `1024`                |
| `max_entry_size_in_bytes` | Максимальный размер в байтах, который могут иметь результаты запросов `SELECT` для сохранения в кэше.      | `1048576`             |
| `max_entry_size_in_rows`  | Максимальное количество строк, которое могут содержать результаты запросов `SELECT` для сохранения в кэше. | `30000000`            |
| `max_size_in_bytes`       | Максимальный размер кэша в байтах. `0` означает, что кэш запросов отключён.                                | `1073741824`          |

<Note>
  * Изменённые настройки вступают в силу немедленно.
  * Для кэша запросов память выделяется в DRAM. Если оперативной памяти недостаточно, задайте небольшое значение `max_size_in_bytes` или полностью отключите кэш запросов.
</Note>

**Пример**

```xml
<query_cache>
    <max_size_in_bytes>1073741824</max_size_in_bytes>
    <max_entries>1024</max_entries>
    <max_entry_size_in_bytes>1048576</max_entry_size_in_bytes>
    <max_entry_size_in_rows>30000000</max_entry_size_in_rows>
</query_cache>
```

<div id="query_log">
  ## query_log
</div>

Настройка логирования запросов, полученных с параметром [log&#95;queries=1](/ru/reference/settings/session-settings).

Запросы записываются в таблицу [system.query&#95;log](/ru/reference/system-tables/query_log), а не в отдельный файл. Имя таблицы можно изменить в параметре `table` (см. ниже).

<SystemLogParameters />

Если таблица не существует, ClickHouse создаст её. Если при обновлении сервера ClickHouse изменилась структура журнала запросов, таблица со старой структурой переименовывается, а новая таблица создаётся автоматически.

**Пример**

```xml
<query_log>
    <database>system</database>
    <table>query_log</table>
    <engine>Engine = MergeTree PARTITION BY event_date ORDER BY event_time TTL event_date + INTERVAL 30 day</engine>
    <flush_interval_milliseconds>7500</flush_interval_milliseconds>
    <max_size_rows>1048576</max_size_rows>
    <reserved_size_rows>8192</reserved_size_rows>
    <buffer_size_rows_flush_threshold>524288</buffer_size_rows_flush_threshold>
    <flush_on_crash>false</flush_on_crash>
</query_log>
```

<div id="query_masking_rules">
  ## query_masking_rules
</div>

Правила на основе Regexp, которые применяются к запросам, а также ко всем сообщениям лога перед их сохранением в серверном журнале,
таблицах [`system.query_log`](/ru/reference/system-tables/query_log), [`system.text_log`](/ru/reference/system-tables/text_log), [`system.processes`](/ru/reference/system-tables/processes), а также в журналах, отправляемых клиенту. Это позволяет предотвратить
утечку в журналы конфиденциальных данных из SQL-запросов, таких как имена, адреса электронной почты, персональные идентификаторы или номера кредитных карт.

**Пример**

```xml
<query_masking_rules>
    <rule>
        <name>hide SSN</name>
        <regexp>(^|\D)\d{3}-\d{2}-\d{4}($|\D)</regexp>
        <replace>000-00-0000</replace>
    </rule>
</query_masking_rules>
```

**Поля config**:

| Параметр  | Описание                                                                                       |
| --------- | ---------------------------------------------------------------------------------------------- |
| `name`    | имя правила (необязательно)                                                                    |
| `regexp`  | регулярное выражение, совместимое с RE2 (обязательно)                                          |
| `replace` | строка подстановки для конфиденциальных данных (необязательно, по умолчанию — шесть звёздочек) |

Правила маскирования применяются ко всему запросу (чтобы предотвратить утечки конфиденциальных данных из некорректно сформированных / неразбираемых запросов).

В таблице [`system.events`](/ru/reference/system-tables/events) есть Counter `QueryMaskingRulesMatch`, который показывает общее количество срабатываний правил маскирования запросов.

Для распределённых запросов каждый server нужно настраивать отдельно, иначе подзапросы, передаваемые на другие
узлы, будут сохраняться без маскирования.

<div id="query_metric_log">
  ## query_metric_log
</div>

По умолчанию журнал отключён.

**Включение**

Чтобы вручную включить сбор истории метрик [`system.query_metric_log`](/ru/reference/system-tables/query_metric_log), создайте файл `/etc/clickhouse-server/config.d/query_metric_log.xml` со следующим содержимым:

```xml
<clickhouse>
    <query_metric_log>
        <database>system</database>
        <table>query_metric_log</table>
        <flush_interval_milliseconds>7500</flush_interval_milliseconds>
        <collect_interval_milliseconds>1000</collect_interval_milliseconds>
        <max_size_rows>1048576</max_size_rows>
        <reserved_size_rows>8192</reserved_size_rows>
        <buffer_size_rows_flush_threshold>524288</buffer_size_rows_flush_threshold>
        <flush_on_crash>false</flush_on_crash>
    </query_metric_log>
</clickhouse>
```

**Отключение**

Чтобы отключить настройку `query_metric_log`, создайте файл `/etc/clickhouse-server/config.d/disable_query_metric_log.xml` со следующим содержимым:

```xml
<clickhouse>
    <query_metric_log remove="1" />
</clickhouse>
```

<div id="query_thread_log">
  ## query_thread_log
</div>

Настройка для логирования потоков запросов, полученных с параметром [log&#95;query&#95;threads=1](/ru/reference/settings/session-settings#log_query_threads).

Запросы записываются в таблицу [system.query&#95;thread&#95;log](/ru/reference/system-tables/query_thread_log), а не в отдельный файл. Имя таблицы можно изменить в параметре `table` (см. ниже).

<SystemLogParameters />

Если таблица не существует, ClickHouse создаст ее. Если при обновлении сервера ClickHouse структура журнала потоков запросов изменилась, таблица со старой структурой будет переименована, а новая таблица создастся автоматически.

**Пример**

```xml
<query_thread_log>
    <database>system</database>
    <table>query_thread_log</table>
    <partition_by>toMonday(event_date)</partition_by>
    <flush_interval_milliseconds>7500</flush_interval_milliseconds>
    <max_size_rows>1048576</max_size_rows>
    <reserved_size_rows>8192</reserved_size_rows>
    <buffer_size_rows_flush_threshold>524288</buffer_size_rows_flush_threshold>
    <flush_on_crash>false</flush_on_crash>
</query_thread_log>
```

Запросы записываются в таблицу [system.query&#95;views&#95;log](/ru/reference/system-tables/query_views_log), а не в отдельный файл. Имя таблицы можно изменить в параметре `table` (см. ниже).
