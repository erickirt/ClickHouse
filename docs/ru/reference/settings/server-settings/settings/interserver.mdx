Эти настройки используются для настройки сервера ClickHouse и автоматически сгенерированы на основе исходного кода ClickHouse.

<div id="interserver_listen_host">
  ## interserver_listen_host
</div>

Ограничение на хосты, которым разрешён обмен данными между серверами ClickHouse.
Если используется Keeper, это же ограничение применяется к обмену данными между различными экземплярами Keeper.

<Note>
  По умолчанию значение совпадает со значением настройки [`listen_host`](/ru/reference/settings/server-settings/settings/listen#listen_host).
</Note>

**Пример**

```xml
<interserver_listen_host>::ffff:a00:1</interserver_listen_host>
<interserver_listen_host>10.0.0.1</interserver_listen_host>
```

<div id="interserver_tables_status_require_auth">
  ## interserver_tables_status_require_auth
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

Требует, чтобы межсерверный `TablesStatusRequest` аутентифицировался с помощью
`<secret>` кластера. Клиенты, достаточно новые для отправки хеша секрета (ревизия протокола
`DBMS_MIN_REVISION_WITH_INTERSERVER_SECRET_TABLES_STATUS`), всегда проходят проверку; этот
параметр дополнительно отклоняет более старые клиенты, которые не отправляют хеш, что по умолчанию
закрывает возможность неаутентифицированного раскрытия статуса таблиц.

По умолчанию — `true` (безопасно по умолчанию). Во время поэтапного обновления узел, еще не обновленный
до новой версии, использует старый протокол и не отправляет хеш, поэтому `Distributed`-запрос, инициированный на
таком узле к уже обновленному узлу, получит отказ для своего `TablesStatusRequest`.
Если необходимо запустить кластер со смешанными версиями, установите здесь `false` на обновленных
узлах, пока не будут обновлены все узлы, а затем удалите переопределение.
