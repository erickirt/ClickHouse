Эти настройки относятся к серверу ClickHouse и автоматически генерируются из исходного кода ClickHouse.

<div id="memory_worker_correct_memory_tracker">
  ## memory_worker_correct_memory_tracker
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

Должен ли фоновый воркер памяти корректировать внутренний трекер памяти на основе данных из внешних источников, таких как jemalloc и cgroups

<div id="memory_worker_decay_adjustment_period_ms">
  ## memory_worker_decay_adjustment_period_ms
</div>

<SettingsInfoBlock type="UInt64" default_value="5000" changeable_without_restart="No" />

Длительность в миллисекундах, в течение которой должно сохраняться состояние memory pressure, прежде чем jemalloc динамически скорректирует `dirty_decay_ms`. Если использование памяти остается выше порога purge в течение этого периода, автоматическое устаревание грязных страниц отключается (`dirty_decay_ms=0`) для более агрессивного освобождения памяти. Если использование памяти остается ниже порога в течение этого периода, восстанавливается поведение устаревания по умолчанию. Установите 0, чтобы отключить динамическую корректировку и использовать настройки устаревания jemalloc по умолчанию.

<div id="memory_worker_dynamic_hard_limit">
  ## memory_worker_dynamic_hard_limit
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

Определяет, будет ли фоновый воркер памяти периодически пересчитывать жёсткий лимит памяти сервера во время выполнения по формуле `(резидентная память + доступная системная память) * max_server_memory_usage_to_ram_ratio`, чтобы сервер сохранял запас ресурсов для других процессов, работающих на том же хосте.

Если этот параметр отключён, `max_server_memory_usage_to_ram_ratio` ограничивает жёсткий лимит памяти только статически — при запуске и при перезагрузке конфигурации, как в предыдущих версиях.

Не влияет, если `max_server_memory_usage_to_ram_ratio` равно `0`.

<div id="memory_worker_period_ms">
  ## memory_worker_period_ms
</div>

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="No" />

Период срабатывания фонового воркера памяти, который корректирует использование памяти трекером памяти и очищает неиспользуемые страницы при повышенном потреблении памяти. Если установлено значение 0, будет использоваться значение по умолчанию в зависимости от источника использования памяти

<div id="memory_worker_purge_dirty_pages_threshold_ratio">
  ## memory_worker_purge_dirty_pages_threshold_ratio
</div>

<SettingsInfoBlock type="Double" default_value="0.2" changeable_without_restart="No" />

Пороговое соотношение объема грязных страниц jemalloc к объему памяти, доступной серверу ClickHouse. Когда объем грязных страниц превышает это соотношение, фоновый воркер памяти принудительно очищает грязные страницы. Если задано значение 0, принудительная очистка на основе соотношения грязных страниц отключается.

<div id="memory_worker_purge_total_memory_threshold_ratio">
  ## memory_worker_purge_total_memory_threshold_ratio
</div>

<SettingsInfoBlock type="Double" default_value="0.9" changeable_without_restart="No" />

Пороговый коэффициент для очистки jemalloc относительно памяти, доступной серверу ClickHouse. Когда общее использование памяти превышает этот коэффициент, фоновый воркер памяти принудительно очищает грязные страницы. Если задано значение 0, принудительная очистка на основе общего объема памяти отключается.

<div id="memory_worker_rss_speculative_reserve_ratio">
  ## memory_worker_rss_speculative_reserve_ratio
</div>

<SettingsInfoBlock type="Double" default_value="1" changeable_without_restart="No" />

На каждом такте `MemoryWorker` дополнительно резервируется
`ratio * min(resident - previous_resident, resident - tracked)` сверх
наблюдаемого RSS, исходя из предположения, что на следующем такте рост может быть таким же,
как и на предыдущем (`resident - previous_resident` — это рост RSS
за предыдущий такт). Рост ограничивается величиной `resident - tracked` — той частью
RSS, которую не видит глобальный трекер памяти, поскольку рост, который
уже отслеживается, обрабатывается обычной проверкой жёсткого лимита. Резерв
применяется к счётчику `rss`, к которому обращается глобальная проверка жёсткого лимита
через `MemoryTracker::allocImpl`, поэтому, когда экстраполированное значение превышает
`max_server_memory_usage`, последующие выделения памяти генерируют исключение
`MEMORY_LIMIT_EXCEEDED` до того, как срабатывает OOM-killer ядра. Значение `0`
отключает спекулятивное резервирование (с возвратом к `rss = resident`); значение по умолчанию `1`
резервирует полный прирост в качестве запаса на следующий интервал. При использовании
санитайзеров (`ASan`, `UBSan`, `MSan`, `TSan`) значением по умолчанию является `0`, поскольку
разрыв `resident - tracked` в основном обусловлен накладными расходами shadow/runtime санитайзера,
а не задержкой учёта в трекере.
