<div id="parallel_replicas_custom_key">
  ## parallel_replicas_custom_key
</div>

<BetaBadge />

Произвольное целочисленное выражение, которое можно использовать для распределения работы между репликами для конкретной таблицы.
Значением может быть любое целочисленное выражение.

Предпочтительны простые выражения на основе первичных ключей.

Если эта настройка используется в кластере, состоящем из одного сегмента и нескольких реплик, эти реплики будут преобразованы в виртуальные сегменты.
В противном случае она будет работать так же, как ключ `SAMPLE`: будут использоваться несколько реплик каждого сегмента.

Примечание: эта настройка не приводит к дополнительной фильтрации данных при обработке запросов, а лишь изменяет точки, в которых фильтр диапазона разбивает диапазон `[0, INT_MAX]` для параллельной обработки.

Представляет локальную/удалённую границу параллельных реплик в виде преобразования плана. Планировщик строит обычный локальный план; затем над шагом чтения вставляется шаг split, который заменяется на `UNION` из локального и удалённого чтения части плана ниже шага split, благодаря чему этот фрагмент выполняется на репликах, а остальная часть — на координаторе. Экспериментальная; имеет приоритет над `parallel_replicas_local_plan`.

<div id="parallel_replicas_prefer_local_replica">
  ## parallel_replicas_prefer_local_replica
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.5"},{"label": "1"},{"label": "Новая настройка. Если отключена, реплики для параллельного чтения выбираются исключительно алгоритмом балансировки нагрузки, без принудительного включения локальной реплики в набор."}]}]} />

Если включено (по умолчанию), локальная реплика всегда включается в набор реплик, используемых для параллельного чтения.

Это позволяет направлять запросы с `max_parallel_replicas = 1` на другой хост, что может улучшить локальность кэша, когда множество коротких запросов распределено по кластеру.
