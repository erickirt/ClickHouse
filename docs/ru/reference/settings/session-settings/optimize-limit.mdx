Исключает функции от других ключей в секции LIMIT BY.

Пример: `LIMIT 5 BY x, f(x)` превращается в `LIMIT 5 BY x`.

  ## optimize_limit_by_in_order
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.6"},{"label": "1"},{"label": "Новая настройка для оптимизации запросов `LIMIT BY`, когда столбцы `BY` образуют префикс ключа сортировки таблицы."}]}]} />

Оптимизирует запросы `SELECT ... LIMIT N BY <cols>`, когда `<cols>` (в любом порядке) образуют префикс ключа сортировки таблицы либо становятся им после того, как `WHERE col = const` фиксирует начальные столбцы. Если настройка включена, источник читает данные в порядке первичного ключа, поэтому строки с одинаковыми значениями в столбцах `BY` идут подряд в пределах каждого потока. Когда данные поступают одним отсортированным потоком, `LIMIT BY` фильтрует их в потоковом режиме с памятью O(1), вместо того чтобы строить хеш-таблицу для всех встретившихся различных комбинаций столбцов `BY`. Когда отсортированные данные поступают в нескольких потоках и одни и те же значения `BY` могут встречаться более чем в одном из них, каждый поток сначала предварительно фильтруется в потоковом режиме до не более чем `LIMIT + OFFSET` строк на группу, затем потоки объединяются, и завершающий `LIMIT BY` на основе хеширования выполняет дедупликацию групп, охватывающих несколько потоков. На этом завершающем этапе по-прежнему хранится запись для каждой различной комбинации столбцов `BY`, но обрабатываются только предварительно отфильтрованные строки.
