<div id="allow_key_condition_coalesce_rewrite">
  ## allow_key_condition_coalesce_rewrite
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.5"},{"label": "1"},{"label": "Новая настройка для переписывания предикатов вида `coalesce(a_1, ..., a_N) <op> const` (и аналогично для `ifNull`, а также с константой слева) в дизъюнкцию перед анализом индексов, чтобы можно было использовать первичный ключ по каждому столбцу и индексы пропуска данных для каждого `a_i`. Также поддерживаются формы с частично константными аргументами, такие как `coalesce(a, 42, b)` и `coalesce(a, b, 42)`."}]}]} />

Разрешает первичному ключу и индексам пропуска данных MergeTree отсекать гранулы для предикатов `WHERE`/`PREWHERE`, в которых используется `coalesce` или `ifNull`. Без этой настройки такие предикаты непрозрачны для анализа индексов и не позволяют отсекать данные, поэтому гранулы, которые заведомо не могут подойти, всё равно читаются. Это влияет только на то, какие гранулы читаются; результаты запроса не меняются, поскольку строки по-прежнему фильтруются исходным предикатом.

Перед анализом индексов переписываются две формы предикатов:

* Сравнение с `coalesce`/`ifNull`, например `coalesce(a, b) = 5`, преобразуется в дизъюнкцию, чтобы индекс по каждому аргументу мог отсекать данные: `a = 5 OR (a IS NULL AND b = 5)`, с соответствующим расширением для большего числа аргументов.
* `coalesce`/`ifNull` с ложной (нулевой) константой по умолчанию, используемый непосредственно как условие, например `ifNull(a = 5, 0)` или `coalesce(a = 5, 0)`, разворачивается во внутренний предикат `a = 5`. Такие обёртки сводят трёхзначный результат внутреннего предиката к однозначному булевому значению (сопоставляя `NULL` со `false`).

<div id="allow_nullable_tuple_in_extracted_subcolumns">
  ## allow_nullable_tuple_in_extracted_subcolumns
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.3"},{"label": "0"},{"label": "Новая настройка, определяющая, могут ли извлечённые подстолбцы Tuple быть Nullable."}]}]} />

Определяет, могут ли извлечённые подстолбцы типа `Tuple(...)` иметь тип `Nullable(Tuple(...))`.

* `false`: Возвращает `Tuple(...)` и использует значения кортежа по умолчанию для строк, в которых подстолбец отсутствует.
* `true`: Возвращает `Nullable(Tuple(...))` и использует `NULL` для строк, в которых подстолбец отсутствует.

Эта настройка управляет только поведением извлечённых подстолбцов.
Она не определяет, можно ли создавать в таблицах столбцы `Nullable(Tuple(...))`; это регулируется параметром `enable_nullable_tuple_type`.

ClickHouse использует значение этой настройки, загруженное при запуске сервера.
Изменения, внесённые с помощью `SET` или `SETTINGS` на уровне запроса, не меняют поведение извлечённых подстолбцов.
Чтобы изменить поведение извлечённых подстолбцов, обновите `allow_nullable_tuple_in_extracted_subcolumns` в конфигурации профиля запуска (например, в users.xml) и перезапустите сервер.

Разрешает передавать аргументы в оконные функции `RANK` и `DENSE_RANK` для обратной совместимости.

Согласно стандарту SQL, `RANK` и `DENSE_RANK` не принимают аргументов — они ранжируют строки только по
окну `OVER (ORDER BY ...)`. В версиях ClickHouse до 26.5 запросы вида
`RANK(x) OVER (...)` молча принимали и игнорировали аргумент, что приводило к путанице у пользователей
(наличие аргумента создавало впечатление, что он влияет на ранжирование, хотя это было не так).

генерируют ошибку `NUMBER_OF_ARGUMENTS_DOESNT_MATCH`. Если установить значение `true`, восстанавливается прежнее более мягкое поведение —
аргументы молча игнорируются, как и до версии 26.5.

По умолчанию это запрещено: `REPLACE PARTITION` из источника, в котором нет данных в запрошенной партиции, вызывает исключение, поскольку в таком случае операция фактически превращается в молчаливый `DROP PARTITION` для пункта назначения (данные в партиции пункта назначения удаляются, но взамен ничего не записывается), что нередко приводит к случайной потере данных (см. [#23727](https://github.com/ClickHouse/ClickHouse/issues/23727)).

Включите эту настройку, чтобы вернуть прежнее поведение, например если вы намеренно используете пустую исходную партицию для очистки данных в пункте назначения. Если нужно просто удалить партицию без дополнительных условий, предпочтительнее использовать `ALTER TABLE ... DROP PARTITION ...`.

  ## allow_settings_after_format_in_insert
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

Управляет тем, разрешён ли `SETTINGS` после `FORMAT` в `INSERT`-запросах. Использовать это не рекомендуется, поскольку часть `SETTINGS` может быть интерпретирована как значения.

Пример:

```sql
INSERT INTO FUNCTION null('foo String') SETTINGS max_threads=1 VALUES ('bar');
```

Но следующий запрос будет работать только при включённой настройке `allow_settings_after_format_in_insert`:

```sql
SET allow_settings_after_format_in_insert=1;
INSERT INTO FUNCTION null('foo String') VALUES ('bar') SETTINGS max_threads=1;
```

Возможные значения:

* 0 — Запрещено.
* 1 — Разрешено.

<Note>
  Используйте эту настройку только для обратной совместимости, если ваши сценарии зависят от старого синтаксиса.
</Note>
