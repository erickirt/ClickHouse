<div id="analyzer_compatibility_allow_compound_identifiers_in_unflatten_nested">
  ## analyzer_compatibility_allow_compound_identifiers_in_unflatten_nested
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "25.8"},{"label": "1"},{"label": "Новая настройка."}]}]} />

Разрешает добавлять составные идентификаторы в Nested. Это настройка совместимости, поскольку она меняет результат запроса. Когда она отключена, `SELECT a.b.c FROM table ARRAY JOIN a` не работает, а `SELECT a FROM table` не включает столбец `a.b.c` в результат `Nested a`.

<div id="analyzer_compatibility_allow_non_aggregate_in_having">
  ## analyzer_compatibility_allow_non_aggregate_in_having
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.7"},{"label": "0"},{"label": "Новая настройка совместимости. Если она включена, анализатор имитирует прежнее преобразование `HAVING` в `WHERE` для неагрегатных AND-конъюнктов вместо выдачи `NOT_AN_AGGREGATE`."}]}]} />

Если параметр включен, анализатор имитирует прежнее поведение, при котором неагрегатные AND-конъюнкты переносятся из `HAVING` в предложение WHERE вместо выдачи `NOT_AN_AGGREGATE`. Отклонение в соответствии со стандартом используется по умолчанию; эта настройка помогает при миграции запросов, которые старый анализатор (`enable_analyzer = 0`) молча принимал. Конъюнкты, содержащие агрегатные функции, `grouping` или недетерминированные функции, остаются в `HAVING`. Если хотя бы один конъюнкт содержит window function или функцию с сохранением состояния (например, `rowNumberInBlock`), преобразование отключается для всего `HAVING`, что соответствует прежнему поведению `PredicateExpressionsOptimizer`. Эта настройка также игнорируется, если в `GROUP BY` используются `WITH CUBE`, `WITH ROLLUP`, `WITH TOTALS` или `GROUPING SETS`.

<div id="analyzer_compatibility_join_using_top_level_identifier">
  ## analyzer_compatibility_join_using_top_level_identifier
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "24.3"},{"label": "0"},{"label": "Принудительно разрешать идентификатор в JOIN USING по проекции"}]}]} />

Принудительно разрешать идентификатор в JOIN USING по проекции (например, в `SELECT a + 1 AS b FROM t1 JOIN t2 USING (b)` JOIN будет выполняться по условию `t1.a + 1 = t2.b`, а не `t1.b = t2.b`).

<div id="analyzer_compatibility_prefer_alias_over_subcolumn">
  ## analyzer_compatibility_prefer_alias_over_subcolumn
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.6"},{"label": "0"},{"label": "Новая настройка совместимости"}]}]} />

Если составной идентификатор, такой как `b.id`, может ссылаться либо на столбец `id` таблицы с псевдонимом `b`, либо на подстолбец `b.id` типа Tuple другого столбца, будет выбрана интерпретация с префиксом-псевдонимом (столбец `id` таблицы `b`). По умолчанию анализатор предпочитает подстолбец. Включите эту настройку, чтобы использовать правила разрешения имен старого анализатора.
