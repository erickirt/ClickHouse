<div id="prefer_column_name_to_alias">
  ## prefer_column_name_to_alias
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

Включает или отключает использование исходных имен столбцов вместо псевдонимов в выражениях запроса и секциях. Это особенно важно, когда псевдоним совпадает с именем столбца, см. [Псевдонимы выражений](/ru/reference/syntax#notes-on-usage). Включите этот параметр, чтобы правила синтаксиса псевдонимов в ClickHouse лучше соответствовали правилам большинства других движков баз данных.

Возможные значения:

* 0 — Имя столбца заменяется псевдонимом.
* 1 — Имя столбца не заменяется псевдонимом.

**Пример**

Разница между включенным и выключенным состоянием:

Запрос:

```sql
SET prefer_column_name_to_alias = 0;
SELECT avg(number) AS number, max(number) FROM numbers(10);
```

Результат:

```text
Received exception from server (version 21.5.1):
Code: 184. DB::Exception: Received from localhost:9000. DB::Exception: Aggregate function avg(number) is found inside another aggregate function in query: While processing avg(number) AS number.
```

Запрос:

```sql
SET prefer_column_name_to_alias = 1;
SELECT avg(number) AS number, max(number) FROM numbers(10);
```

Результат:

```text
┌─number─┬─max(number)─┐
│    4.5 │           9 │
└────────┴─────────────┘
```

<div id="prefer_global_in_and_join">
  ## prefer_global_in_and_join
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

Включает замену операторов `IN`/`JOIN` на `GLOBAL IN`/`GLOBAL JOIN`.

Возможные значения:

* 0 — Отключено. Операторы `IN`/`JOIN` не заменяются на `GLOBAL IN`/`GLOBAL JOIN`.
* 1 — Включено. Операторы `IN`/`JOIN` заменяются на `GLOBAL IN`/`GLOBAL JOIN`.

**Использование**

Хотя `SET distributed_product_mode=global` может изменить поведение запросов для распределённых таблиц, он не подходит для локальных таблиц или таблиц из внешних источников. В таких случаях используется настройка `prefer_global_in_and_join`.

Например, у нас есть узлы, обслуживающие запросы, с локальными таблицами, которые не подходят для распределения. Их данные нужно на лету разослать в ходе распределённой обработки с помощью ключевого слова `GLOBAL` — `GLOBAL IN`/`GLOBAL JOIN`.

Ещё один сценарий использования `prefer_global_in_and_join` — доступ к таблицам, созданным с помощью внешних движков. Эта настройка помогает сократить количество обращений к внешним источникам при JOIN таких таблиц: до одного вызова на запрос.

**См. также:**

* [Распределённые подзапросы](/ru/reference/statements/in#distributed-subqueries) — дополнительная информация об использовании `GLOBAL IN`/`GLOBAL JOIN`

<div id="prefer_localhost_replica">
  ## prefer_localhost_replica
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

Включает или отключает предпочтительное использование реплики на localhost при обработке распределённых запросов.

Возможные значения:

* 1 — ClickHouse всегда отправляет запрос реплике на localhost, если она существует.
* 0 — ClickHouse использует стратегию балансировки, указанную в настройке [load&#95;balancing](/ru/reference/settings/session-settings/load-balancing#load_balancing).

<Note>
  Отключите эту настройку, если используете [max&#95;parallel&#95;replicas](/ru/reference/settings/session-settings/max#max_parallel_replicas) без [parallel&#95;replicas&#95;custom&#95;key](/ru/reference/settings/session-settings/parallel-replicas#parallel_replicas_custom_key).
  Если задан [parallel&#95;replicas&#95;custom&#95;key](/ru/reference/settings/session-settings/parallel-replicas#parallel_replicas_custom_key), отключайте эту настройку только в том случае, если она используется в кластере с несколькими сегментами, содержащими несколько реплик.
  Если она используется в кластере с одним сегментом и несколькими репликами, отключение этой настройки приведёт к нежелательным последствиям.
</Note>

  ## prefer_warmed_unmerged_parts_seconds
</div>

<CloudOnlyBadge />

<SettingsInfoBlock type="Int64" default_value="0" />

Действует только в ClickHouse Cloud. Если слитая часть была создана менее указанного числа секунд назад и не была предварительно прогрета (см. [cache&#95;populated&#95;by&#95;fetch](/ru/reference/settings/merge-tree-settings#cache_populated_by_fetch)), но все её исходные части доступны и предварительно прогреты, запросы SELECT будут читать данные из этих частей вместо неё. Только для Replicated-/SharedMergeTree. Обратите внимание, что здесь проверяется только то, была ли часть обработана CacheWarmer; если часть была загружена в кэш каким-либо другим способом, она всё равно будет считаться холодной, пока до неё не дойдёт CacheWarmer; если она была прогрета, а затем вытеснена из кэша, она всё равно будет считаться прогретой.
