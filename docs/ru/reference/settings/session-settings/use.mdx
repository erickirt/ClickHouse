<div id="use_constant_folding_in_index_analysis">
  ## use_constant_folding_in_index_analysis
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.7"},{"label": "0"},{"label": "Новая настройка: подстановка констант уровня партиции в предикат фильтра для каждой части при анализе индексов MergeTree, что улучшает отсечение данных для фильтров, ветви которых зависят от значений партиции."}]}]} />

Подставляет константы уровня партиции в предикат фильтра при анализе первичного ключа и индекса пропуска данных для каждой части.

Если ключ партиционирования используется в фильтре вместе со столбцами первичного ключа или индекса пропуска данных, анализ индексов может отдельно сворачивать значение партиции в пределах каждой части. Это особенно полезно для дизъюнктивных фильтров, ветви которых относятся к разным партициям. Например, при `PARTITION BY a` и `ORDER BY b`:

```sql
SELECT * FROM t WHERE (a = 1 AND b >= 1) OR (a = 2 AND b > 10) OR (a = 3 AND b > 10)
```

Для части в партиции `a = 1` условие упрощается до `b >= 1`, а для партиций `a = 2` и `a = 3` — до `b > 10`, поэтому каждая часть анализируется с предикатом, который действительно к ней применим.

Возможные значения:

* 0 — Отключено.
* 1 — Включено.

<div id="use_hedged_requests">
  ## use_hedged_requests
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "21.9"},{"label": "1"},{"label": "По умолчанию включает возможность Hedged Requests"}]}]} />

Включает логику hedged requests для удалённых запросов. Позволяет устанавливать для одного запроса несколько соединений с разными репликами.
Новое соединение создаётся, если существующее соединение с репликой не было установлено в течение `hedged_connection_timeout`
или если в течение `receive_data_timeout` не были получены данные. Запрос использует первое соединение, которое отправит непустой пакет Прогресс (или пакет данные, если `allow_changing_replica_until_first_data_packet`);
остальные соединения отменяются. Поддерживаются запросы с `max_parallel_replicas > 1`.

Включено по умолчанию.

Значение по умолчанию в Cloud: `0`.

<div id="use_strict_insert_block_limits">
  ## use_strict_insert_block_limits
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.4"},{"label": "0"},{"label": "Новая настройка для строгого применения минимальных и максимальных границ блока при вставках. Когда min < max, приоритет имеют максимальные ограничения."}]}]} />

Если включено, строго применяются и минимальные, и максимальные ограничения на размер блока вставки.

* Минимальные пороги (побитовое И): достигнуты оба значения — min&#95;insert&#95;block&#95;size&#95;rows и min&#95;insert&#95;block&#95;size&#95;bytes.

* Минимальные пороги (OR): достигнуто min&#95;insert&#95;block&#95;size&#95;rows или min&#95;insert&#95;block&#95;size&#95;bytes.

По умолчанию отключено.

<div id="use_uncompressed_cache">
  ## use_uncompressed_cache
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

Определяет, использовать ли кэш несжатых данных. Допустимые значения: 0 или 1. По умолчанию — 0 (отключено).
Использование кэша несжатых данных (только для таблиц семейства MergeTree) может значительно снизить задержку и повысить пропускную способность при работе с большим количеством коротких запросов. Включайте этот параметр для пользователей, которые часто отправляют короткие запросы. Также обратите внимание на параметр конфигурации [uncompressed&#95;cache&#95;size](/ru/reference/settings/server-settings/settings#uncompressed_cache_size) (задается только в файле конфигурации) — размер блоков кэша несжатых данных. По умолчанию он составляет 8 GiB. Кэш несжатых данных заполняется по мере необходимости, а наименее используемые данные удаляются автоматически.

Для запросов, которые считывают хотя бы относительно большой объем данных (один миллион строк или более), кэш несжатых данных отключается автоматически, чтобы освободить место для действительно небольших запросов. Это означает, что параметр &#39;use&#95;uncompressed&#95;cache&#39; можно всегда оставлять равным 1.
