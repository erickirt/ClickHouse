O roteamento com reconhecimento de réplicas (também conhecido como sticky sessions, sticky routing ou afinidade de sessão) encaminha solicitações relacionadas para a mesma réplica do ClickHouse. Use-o quando você precisar que [tabelas temporárias](/sql-reference/statements/create/table#temporary-tables) ou [estado de sessão nomeado](/interfaces/http#using-clickhouse-sessions-in-the-http-protocol) permaneçam acessíveis entre consultas, ou quando quiser que consultas relacionadas reutilizem os mesmos caches locais da réplica.

Ele funciona em regime de melhor esforço e não garante isolamento. Scaling, upgrade e reinicializações podem remapear a réplica em que um determinado `session_id` vai cair.

<Warning>
  **Requer a interface HTTP**

  O roteamento com reconhecimento de réplicas é aplicado na camada de proxy sobre a [interface HTTP/HTTPS](/interfaces/http), usando o parâmetro de consulta `session_id` (veja abaixo). Ele **não está disponível no protocolo nativo** (porta nativa, por exemplo, o driver [clickhouse-go](/integrations/go) em seu modo nativo padrão). Clientes do protocolo nativo precisam mudar para HTTP e passar um `session_id` em cada solicitação. Para clickhouse-go (v2), defina `Protocol: clickhouse.HTTP` e passe `session_id` como uma [configuração](/pt-BR/integrations/language-clients/go/database-sql-api#sessions). O driver o envia como o parâmetro de consulta na URL sobre o qual o proxy calcula o hash.
</Warning>

<div id="prerequisites">
  ## Pré-requisitos
</div>

* Seu serviço precisa de **2 ou mais réplicas**. Em um serviço com apenas uma réplica, não há nada ao que se vincular.
* Um serviço **em execução**. Reativar um serviço inativo pode alterar a réplica à qual um `session_id` é mapeado.
* Disponível no **Enterprise** por padrão quando o recurso estiver em GA.
* Compatível com os serviços padrão do ClickHouse Cloud. O [BYOC](/cloud/reference/byoc/overview) ainda não é compatível.

<div id="configuring-replica-aware-routing">
  ## Configurando o roteamento com reconhecimento de réplicas
</div>

Abra um ticket de [suporte](https://clickhouse.com/support/program) e solicite a habilitação do roteamento sticky de réplicas via HTTP. Inclua o ID do seu serviço e o motivo da necessidade (tabelas temporárias, estado da sessão ou reutilização de cache). Depois que isso for habilitado, comece a enviar `?session_id=` nas solicitações HTTPS. Não é necessário reiniciar.

<div id="http-based-routing">
  ## Roteamento baseado em HTTP (`session_id`)
</div>

Para fixar uma carga de trabalho em uma réplica, defina o parâmetro de consulta `session_id` na [interface HTTPS](/interfaces/http). O proxy usa hash consistente nesse valor para selecionar uma réplica, de modo que todas as solicitações com o mesmo `session_id` vão para o mesmo servidor até que a topologia do cluster mude.

Você usa o hostname existente do seu service. Não são necessários hostnames sticky especiais nem alterações de DNS.

```bash
echo 'SELECT hostName()' | curl \
  -H 'X-ClickHouse-User: default' \
  -H 'X-ClickHouse-Key: <password>' \
  'https://<host>:8443/?session_id=my-workload-1' -d @-
```

Toda solicitação com `session_id=my-workload-1` vai para a mesma réplica. Um valor de `session_id` diferente passa por hash de forma independente e pode ir para a mesma réplica ou para outra réplica. O mapeamento é consistente, mas você não escolhe *para qual* réplica um determinado valor é mapeado.

`session_id` é qualquer string que você escolher (nome da aplicação, ID do usuário ou rótulo da carga de trabalho). Solicitações sem `session_id` mantêm o balanceamento de carga normal.

Qualquer cliente HTTP que possa adicionar um parâmetro de consulta funciona, incluindo `curl`, [clickhouse-connect](/integrations/python), JDBC/ODBC e outros. Para `clickhouse-go` (v2), use o modo HTTP conforme observado acima.

<div id="check-which-replica">
  ### Verifique qual réplica foi acessada
</div>

Execute novamente o exemplo `SELECT hostName()` acima com o mesmo `session_id`. Você deve obter o mesmo hostname. Um `session_id` diferente pode ser mapeado para outra réplica.

<div id="subdomain-based-routing-deprecated">
  ## Roteamento baseado em subdomínio (descontinuado)
</div>

<Danger>
  **Descontinuado**

  O mecanismo baseado em subdomínio está sendo **descontinuado** e não será mais habilitado em novos serviços. Ele não escala (cada endpoint sticky requer seu próprio certificado TLS). Use o [método `session_id` baseado em HTTP](#http-based-routing) no lugar. Se você já usa subdomínios sticky, entre em contato com o [suporte](https://clickhouse.com/support/program) para habilitar o roteamento `session_id`. Esta é uma mudança incompatível e exigirá migração.
</Danger>

Anteriormente, habilitar o roteamento com reconhecimento de réplicas permitia usar um subdomínio curinga sobre o hostname do serviço. Para um serviço com o host name `abcxyz123.us-west-2.aws.clickhouse.cloud`, qualquer hostname correspondente a `*.sticky.abcxyz123.us-west-2.aws.clickhouse.cloud` (por exemplo, `aaa.sticky.abcxyz123.us-west-2.aws.clickhouse.cloud`) era mapeado pelo Envoy, via hash, para uma réplica consistente. O hostname original continuava usando balanceamento de carga `LEAST_CONNECTION`, o algoritmo de roteamento padrão.

<div id="limitations-of-replica-aware-routing">
  ## Limitações do roteamento com reconhecimento de réplicas
</div>

<div id="replica-aware-routing-does-not-guarantee-isolation">
  ### A afinidade pode se perder durante alterações no serviço
</div>

Qualquer interrupção no serviço altera o hash ring de roteamento. Isso inclui reinicializações do pod do servidor (upgrade de versão, falha, escalonamento vertical) e aumento ou redução de escala. Requisições com o mesmo `session_id` podem, então, ser encaminhadas para um pod do servidor diferente. Se você depende de tabelas temporárias ou de configurações da sessão, esteja preparado para recriá-las após um remapeamento.

<div id="not-workload-isolation">
  ### O roteamento com reconhecimento de réplicas não é isolamento de workload
</div>

O sticky routing controla apenas *qual* réplica atende a uma solicitação. Essa réplica ainda pode atender outro tráfego. Para processamento dedicado, use [compute-compute separation](/cloud/reference/warehouses).

<div id="replica-aware-routing-does-not-work-out-of-the-box-with-private-link">
  ### Private Link e o método de subdomínio obsoleto
</div>

O roteamento HTTP de `session_id` funciona com [rede privada](/cloud/security/connectivity/private-networking) no hostname padrão do seu serviço. Nenhuma entrada DNS adicional é necessária.

O método de subdomínio obsoleto não funciona: você precisa adicionar DNS para o padrão de hostname `*.sticky.*`, e uma configuração incorreta pode desequilibrar a carga entre as réplicas.

<div id="replica-aware-routing-requires-http">
  ### O roteamento com reconhecimento de réplicas exige o protocolo HTTP
</div>

O roteamento sticky usa como chave o parâmetro de consulta `session_id`, que existe apenas na interface HTTP/HTTPS. O protocolo binário nativo não transporta esse parâmetro para que o proxy possa aplicar hash e, por isso, o roteamento com reconhecimento de réplicas não está disponível no protocolo nativo. Hoje, clientes do protocolo nativo precisam mover a carga de trabalho relevante para a interface HTTP para usar esse recurso.

<div id="troubleshooting">
  ## Solução de problemas
</div>

**As consultas ainda são direcionadas a réplicas diferentes com o mesmo `session_id`**

* Confirme que `session_id` é um parâmetro de consulta da URL (`?session_id=...`), e não um cabeçalho HTTP.
* Aguarde um pouco após a habilitação. Pode levar menos de um minuto para fazer efeito.
* Verifique se o serviço foi escalado ou reiniciado recentemente; é esperado que ocorra um remapeamento após mudanças na topologia. Use `SELECT hostName()` para descobrir o novo mapeamento.
