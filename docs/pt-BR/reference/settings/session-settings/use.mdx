<div id="use_constant_folding_in_index_analysis">
  ## use_constant_folding_in_index_analysis
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.7"},{"label": "0"},{"label": "Nova configuração para incorporar constantes no nível da partição ao predicado de filtro por parte durante a análise de índices do MergeTree, melhorando a eliminação de dados para filtros cujos ramos dependem de valores de partição."}]}]} />

Substitua constantes no nível da partição no predicado de filtro ao analisar, por parte, a chave primária e os skip indexes.

Quando a chave de partição aparece no filtro junto com colunas da chave primária ou de skip index, isso permite que a análise de índices incorpore o valor da partição separadamente em cada parte. Isso é mais útil para filtros disjuntivos cujos ramos visam partições diferentes. Por exemplo, com `PARTITION BY a` e `ORDER BY b`:

```sql
SELECT * FROM t WHERE (a = 1 AND b >= 1) OR (a = 2 AND b > 10) OR (a = 3 AND b > 10)
```

Para a parte da partição `a = 1`, a condição é simplificada para `b >= 1`, enquanto, para as partições `a = 2` e `a = 3`, ela é simplificada para `b > 10`, de modo que cada parte seja analisada com o predicado que de fato se aplica a ela.

Valores possíveis:

* 0 — Desativado.
* 1 — Ativado.

<div id="use_strict_insert_block_limits">
  ## use_strict_insert_block_limits
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.4"},{"label": "0"},{"label": "Nova configuração para usar limites mínimo e máximo estritos de inserção em inserts. Quando min < max, os limites máximos têm precedência."}]}]} />

Quando ativada, aplica estritamente os limites mínimo e máximo do tamanho do bloco de inserção.

Um bloco é emitido quando:

* Limiares mínimos (AND): Ambos min&#95;insert&#95;block&#95;size&#95;rows AND min&#95;insert&#95;block&#95;size&#95;bytes são atingidos.
* Limiares máximos (OR): max&#95;insert&#95;block&#95;size&#95;rows OR max&#95;insert&#95;block&#95;size&#95;bytes é atingido.

Quando desativada, um bloco é emitido quando:

* Limiares mínimos (OR):  min&#95;insert&#95;block&#95;size&#95;rows OR min&#95;insert&#95;block&#95;size&#95;bytes é atingido.

**Nota**: Se as configurações máximas forem menores que as configurações mínimas, os limites máximos têm precedência e os blocos serão emitidos antes que os limiares mínimos sejam atingidos.

**Nota**: Essa configuração é desativada automaticamente para async inserts, porque async inserts anexam tokens de desduplicação por entrada que são incompatíveis com a divisão de blocos necessária para aplicar limites estritos.

Desativada por padrão.

<div id="use_uncompressed_cache">
  ## use_uncompressed_cache
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

Define se deve usar um cache de blocos não comprimidos. Aceita 0 ou 1. O padrão é 0 (desabilitado).
Usar o cache não comprimido (apenas para tabelas da família MergeTree) pode reduzir significativamente a latência e aumentar a taxa de transferência ao trabalhar com um grande número de consultas curtas. Habilite essa configuração para usuários que enviam requisições curtas com frequência. Observe também o parâmetro de configuração [uncompressed&#95;cache&#95;size](/pt-BR/reference/settings/server-settings/settings#uncompressed_cache_size) (definido apenas no arquivo de configuração) – o tamanho dos blocos do cache não comprimido. Por padrão, ele é de 8 GiB. O cache não comprimido é preenchido conforme necessário, e os dados menos usados são excluídos automaticamente.

Para consultas que leem pelo menos um volume de dados razoavelmente grande (um milhão de linhas ou mais), o cache não comprimido é desabilitado automaticamente para economizar espaço para consultas realmente pequenas. Isso significa que você pode manter a configuração `use_uncompressed_cache` sempre definida como 1.
