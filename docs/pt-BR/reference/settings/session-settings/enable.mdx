<div id="enable_add_distinct_to_in_subqueries">
  ## enable_add_distinct_to_in_subqueries

<VersionHistory rows={[{"id": "row-1","items": [{"label": "25.8"},{"label": "0"},{"label": "Nova configuração para reduzir o tamanho das tabelas temporárias transferidas em subconsultas distribuídas com `IN`."}]}]} />

Habilita `DISTINCT` em subconsultas `IN`. Esta configuração envolve um trade-off: ao habilitá-la, é possível reduzir bastante o tamanho das tabelas temporárias transferidas em subconsultas distribuídas com `IN` e acelerar significativamente a transferência de dados entre shards, garantindo que apenas valores únicos sejam enviados.
No entanto, habilitar essa configuração acrescenta esforço extra de mesclagem em cada nó, já que a desduplicação (`DISTINCT`) precisa ser executada. Use esta configuração quando a transferência pela rede for um gargalo e o custo adicional de mesclagem for aceitável.

<div id="enable_order_by_all">
  ## enable_order_by_all
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

Ativa ou desativa a ordenação com a sintaxe `ORDER BY ALL`; consulte [ORDER BY](/pt-BR/reference/statements/select/order-by).

Valores possíveis:

* 0 — Desativa ORDER BY ALL.
* 1 — Ativa ORDER BY ALL.

**Exemplo**

Consulta:

```sql
CREATE TABLE TAB(C1 Int, C2 Int, ALL Int) ENGINE=Memory();

INSERT INTO TAB VALUES (10, 20, 30), (20, 20, 10), (30, 10, 20);

SELECT * FROM TAB ORDER BY ALL; -- returns an error that ALL is ambiguous

SELECT * FROM TAB ORDER BY ALL SETTINGS enable_order_by_all = 0;
```

Resultado:

```text
┌─C1─┬─C2─┬─ALL─┐
│ 20 │ 20 │  10 │
│ 30 │ 10 │  20 │
│ 10 │ 20 │  30 │
└────┴────┴─────┘
```

<div id="enable_shared_storage_snapshot_in_query">
  ## enable_shared_storage_snapshot_in_query
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "25.6"},{"label": "0"},{"label": "Uma nova configuração para compartilhar o snapshot de armazenamento na consulta"}]}, {"id": "row-2","items": [{"label": "25.12"},{"label": "1"},{"label": "Habilita por padrão o compartilhamento do snapshot de armazenamento na consulta"}]}]} />

Se estiver habilitada, todas as subconsultas dentro de uma única consulta compartilharão o mesmo StorageSnapshot para cada tabela.
Isso garante uma visão consistente dos dados em toda a consulta, mesmo se a mesma tabela for acessada várias vezes.

Isso é necessário para consultas em que a consistência interna das partes de dados é importante. Exemplo:

```sql
SELECT
    count()
FROM events
WHERE (_part, _part_offset) IN (
    SELECT _part, _part_offset
    FROM events
    WHERE user_id = 42
)
```

Sem essa configuração, as consultas externa e interna podem operar em snapshots de dados diferentes, levando a resultados incorretos.

<Note>
  Ativar essa configuração desabilita a otimização que remove partes de dados desnecessárias dos snapshots após a conclusão da etapa de planejamento.
  Como resultado, consultas de longa duração podem reter partes obsoletas durante toda a execução, atrasando a limpeza dessas partes e aumentando a pressão sobre o armazenamento.

  No momento, essa configuração se aplica apenas a tabelas da família MergeTree.
</Note>

Valores possíveis:

* 0 - Desativado
* 1 - Ativado
