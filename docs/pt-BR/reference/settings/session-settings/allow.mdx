<div id="allow_key_condition_coalesce_rewrite">
  ## allow_key_condition_coalesce_rewrite
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.5"},{"label": "1"},{"label": "Nova configuração para reescrever predicados da forma `coalesce(a_1, ..., a_N) <op> const` (e, de forma equivalente, `ifNull`, ou com a constante à esquerda) como uma disjunção antes da análise de índices, para que a chave primária por coluna e os índices de salto em cada `a_i` possam ser usados. Formas parcialmente constantes, como `coalesce(a, 42, b)` e `coalesce(a, b, 42)`, também são tratadas."}]}]} />

Permite que a chave primária e os índices de salto do MergeTree descartem grânulos para predicados `WHERE`/`PREWHERE` que envolvem `coalesce` ou `ifNull`. Sem essa configuração, esses predicados são opacos à análise de índices e não descartam grânulos, de modo que grânulos que não podem corresponder ainda são lidos. Isso afeta apenas quais grânulos são lidos; os resultados da consulta permanecem inalterados, porque as linhas ainda são filtradas pelo predicado original.

Duas formas de predicado são reescritas antes da análise de índices:

* Uma comparação com `coalesce`/`ifNull`, como `coalesce(a, b) = 5`, torna-se uma disjunção para que um índice em cada argumento possa descartar grânulos: `a = 5 OR (a IS NULL AND b = 5)`, com extensão para mais argumentos.
* Um `coalesce`/`ifNull` com uma constante padrão falsy (zero), usado diretamente como condição, como `ifNull(a = 5, 0)` ou `coalesce(a = 5, 0)`, é reduzido ao predicado interno `a = 5`. Esses encapsulamentos colapsam o resultado ternário do predicado interno em um booleano definido (mapeando `NULL` para `false`).

<div id="allow_nullable_tuple_in_extracted_subcolumns">
  ## allow_nullable_tuple_in_extracted_subcolumns
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.3"},{"label": "0"},{"label": "Nova configuração que controla se subcolunas extraídas de Tuple podem ser anuláveis."}]}]} />

Controla se subcolunas extraídas do tipo `Tuple(...)` podem ser tipadas como `Nullable(Tuple(...))`.

* `false`: Retorna `Tuple(...)` e usa os valores padrão da tupla para linhas em que a subcoluna está ausente.
* `true`: Retorna `Nullable(Tuple(...))` e usa `NULL` para linhas em que a subcoluna está ausente.

Para mudar o comportamento das subcolunas extraídas, atualize `allow_nullable_tuple_in_extracted_subcolumns` na configuração do perfil de inicialização (por exemplo, users.xml) e reinicie o servidor.

<div id="allow_rank_dense_rank_arguments">
  ## allow_rank_dense_rank_arguments
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.5"},{"label": "0"},{"label": "Nova configuração. Segundo o padrão SQL, `RANK` e `DENSE_RANK` aceitam zero argumentos — elas classificam as linhas com base apenas na janela

<div id="allow_replace_partition_from_empty_source">
  ## allow_replace_partition_from_empty_source
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.6"},{"label": "0"},{"label": "Nova verificação de segurança: `ALTER TABLE ... REPLACE PARTITION ... FROM ...` agora gera uma exceção quando a tabela de origem não tem partes na partição solicitada (corrige a perda silenciosa de dados em [#23727](https:\/\/github.com\/ClickHouse\/ClickHouse\/issues\/23727)). Permite que `ALTER TABLE ... REPLACE PARTITION ... FROM ...` remova silenciosamente a partição de destino quando a origem não tiver partes nessa partição.

Por padrão, isso não é permitido: `REPLACE PARTITION` a partir de uma origem sem dados na partição solicitada gera uma exceção, porque, nesse caso, a operação efetivamente se torna um `DROP PARTITION` silencioso no destino (os dados do destino são removidos e nada os substitui), uma causa comum de perda acidental de dados (veja [#23727](https://github.com/ClickHouse/ClickHouse/issues/23727)).

Ative essa configuração para restaurar o comportamento anterior, por exemplo, quando você usa intencionalmente uma partição de origem vazia para limpar dados no destino. Para uma remoção incondicional, prefira `ALTER TABLE ... DROP PARTITION ...`.

<div id="allow_settings_after_format_in_insert">
  ## allow_settings_after_format_in_insert
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "22.4"},{"label": "0"},{"label": "Não permite SETTINGS após FORMAT em consultas INSERT, porque o ClickHouse interpreta SETTINGS como valores, o que pode causar confusão"}]}]} />

Controla se `SETTINGS` após `FORMAT` em consultas `INSERT` é permitido ou não. Não é recomendável usar isso, pois parte de `SETTINGS` pode ser interpretada como valores.

Exemplo:

```sql
INSERT INTO FUNCTION null('foo String') SETTINGS max_threads=1 VALUES ('bar');
```

Mas a consulta a seguir só funcionará com `allow_settings_after_format_in_insert`:

```sql
SET allow_settings_after_format_in_insert=1;
INSERT INTO FUNCTION null('foo String') VALUES ('bar') SETTINGS max_threads=1;
```

Valores possíveis:

* 0 — Não permitir.
* 1 — Permitir.

<Note>
  Use esta configuração apenas por compatibilidade retroativa, caso seus casos de uso dependam da sintaxe antiga.
</Note>
