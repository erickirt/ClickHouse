<div id="concurrent_part_removal_threshold">
  ## concurrent_part_removal_threshold
</div>

<SettingsInfoBlock type="UInt64" default_value="100" />

Ativa a remoção concorrente de partes (consulte &#39;max&#95;part&#95;removal&#95;threads&#39;) somente se
o número de partes de dados inativas for pelo menos igual a este valor.

<div id="concurrent_part_removal_threshold_for_remote_disk">
  ## concurrent_part_removal_threshold_for_remote_disk
</div>

<SettingsInfoBlock type="UInt64" default_value="16" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.5"},{"label": "16"},{"label": "Nova configuração. Limite mais baixo para entrar no modo de remoção concorrente de partes quando qualquer parte removida estiver em um disco remoto, em que cada remoção normalmente representa uma ida e volta de rede. O valor antigo (100) corresponde ao valor padrão legado de `concurrent_part_removal_threshold`, portanto modos antigos de `compatibility` preservam o comportamento anterior."}]}]} />

Igual a `concurrent_part_removal_threshold`, mas usado quando pelo menos uma
parte que está sendo removida é armazenada em um disco remoto. O valor padrão é mais baixo
porque cada remoção de parte em armazenamento remoto normalmente exige uma
ida e volta de rede (por exemplo, um `DELETE` HTTP por parte no armazenamento de objetos), então uma
remoção em série de apenas 100 partes já pode fazer um `DROP TABLE` ficar bloqueado por dezenas de
segundos.
