Essas configurações são do servidor ClickHouse e são geradas automaticamente a partir do código-fonte do ClickHouse.

<div id="interserver_listen_host">
  ## interserver_listen_host
</div>

Restrição aos hosts que podem trocar dados entre os servidores ClickHouse.
Se o Keeper for usado, a mesma restrição será aplicada à comunicação entre diferentes instâncias do Keeper.

<Note>
  Por padrão, o valor é igual à configuração [`listen_host`](/pt-BR/reference/settings/server-settings/settings/listen#listen_host).
</Note>

**Exemplo**

```xml
<interserver_listen_host>::ffff:a00:1</interserver_listen_host>
<interserver_listen_host>10.0.0.1</interserver_listen_host>
```

Tipo:

Padrão:

<div id="interserver_tables_status_require_auth">
  ## interserver_tables_status_require_auth
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

Exige que a `TablesStatusRequest` interserver seja autenticada com o
`<secret>` do cluster. Clientes suficientemente novos para enviar um hash do
secret (revisão de protocolo
`DBMS_MIN_REVISION_WITH_INTERSERVER_SECRET_TABLES_STATUS`) são sempre validados; esta
configuração também rejeita clientes mais antigos que não enviam hash, o que
fecha por padrão a exposição não autenticada do status das tabelas.

O padrão é `true` (seguro por padrão). Durante uma atualização gradual, um nó
ainda não atualizado usa o protocolo antigo e não envia hash, portanto uma
consulta `Distributed` iniciada em um desses nós contra um nó já atualizado
terá sua `TablesStatusRequest` rejeitada. Se você precisar executar um cluster
com versões mistas, defina isso como `false` nos nós atualizados até que todos
os nós sejam atualizados e, em seguida, remova o override.
