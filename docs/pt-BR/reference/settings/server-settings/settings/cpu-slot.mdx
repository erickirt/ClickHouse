Essas configurações se aplicam ao servidor ClickHouse e são geradas automaticamente a partir do código-fonte do ClickHouse.

<div id="cpu_slot_preemption">
  ## cpu_slot_preemption
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Yes" />

Define como é feito o agendamento de workloads para recursos de CPU (MASTER THREAD e WORKER THREAD).

* Se `true` (padrão), a contabilização é feita com base no tempo de CPU efetivamente consumido. Uma quantidade justa de tempo de CPU é alocada aos workloads concorrentes. Os slots são alocados por um tempo limitado e solicitados novamente após expirarem. A solicitação de slots pode bloquear a execução da thread em caso de sobrecarga de recursos de CPU, ou seja, pode ocorrer preempção. Isso garante justiça no uso do tempo de CPU.
* Se `false`, a contabilização é baseada no número de slots de CPU alocados. Uma quantidade justa de slots de CPU é alocada aos workloads concorrentes. Um slot é alocado quando uma thread é iniciada, permanece alocado continuamente e é liberado quando a thread termina a execução. O número de threads alocadas para a execução da consulta só pode aumentar de 1 até `max_threads` e nunca diminuir. Isso favorece mais consultas de longa duração e pode fazer com que consultas curtas fiquem sem CPU.

**Exemplo**

```xml
<cpu_slot_preemption>true</cpu_slot_preemption>
```

<div id="cpu_slot_preemption_timeout_ms">
  ## cpu_slot_preemption_timeout_ms
</div>

<SettingsInfoBlock type="UInt64" default_value="1000" changeable_without_restart="Sim" />

Define por quantos milissegundos uma thread de trabalho pode esperar durante a preempção, isto é, enquanto aguarda a concessão de outro slot de CPU. Após esse timeout, se a thread não conseguir adquirir um novo slot de CPU, ela será encerrada, e a consulta será reduzida dinamicamente para um número menor de threads em execução concorrentemente. Observe que a thread mestre nunca é reduzida, mas pode ser preemptada indefinidamente. Isso só faz sentido quando `cpu_slot_preemption` está habilitado e o recurso de CPU está definido para WORKER THREAD.

**Exemplo**

```xml
<cpu_slot_preemption_timeout_ms>1000</cpu_slot_preemption_timeout_ms>
```

<div id="cpu_slot_quantum_ns">
  ## cpu_slot_quantum_ns
</div>

<SettingsInfoBlock type="UInt64" default_value="10000000" changeable_without_restart="Yes" />

Define quantos nanossegundos de CPU uma thread pode consumir depois de adquirir um slot de CPU e antes de precisar solicitar outro slot de CPU. Só faz sentido se `cpu_slot_preemption` estiver habilitado e o recurso de CPU estiver definido para MASTER THREAD ou WORKER THREAD.

**Exemplo**

```xml
<cpu_slot_quantum_ns>10000000</cpu_slot_quantum_ns>
```
