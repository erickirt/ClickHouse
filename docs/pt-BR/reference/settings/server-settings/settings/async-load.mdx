Estas configurações configuram o servidor ClickHouse e são geradas automaticamente a partir do código-fonte do ClickHouse.

<div id="async_load_databases">
  ## async_load_databases
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Não" />

Carregamento assíncrono de bancos de dados e tabelas.

* Se `true`, todos os bancos de dados não pertencentes ao sistema com engine `Ordinary`, `Atomic` e `Replicated` serão carregados de forma assíncrona após a inicialização do servidor ClickHouse. Consulte a tabela `system.asynchronous_loader` e as configurações do servidor `tables_loader_background_pool_size` e `tables_loader_foreground_pool_size`. Qualquer consulta que tente acessar uma tabela que ainda não foi carregada aguardará a inicialização dessa tabela específica. Se a tarefa de carregamento falhar, a consulta lançará o erro novamente (em vez de desligar o servidor inteiro no caso de `async_load_databases = false`). A tabela pela qual pelo menos uma consulta estiver aguardando será carregada com prioridade mais alta. Consultas DDL em um banco de dados aguardarão a inicialização desse banco de dados específico. Considere também definir um limite `max_waiting_queries` para o número total de consultas em espera.
* Se `false`, todos os bancos de dados são carregados quando o servidor é iniciado.

**Exemplo**

```xml
<async_load_databases>true</async_load_databases>
```

<div id="async_load_system_database">
  ## async_load_system_database
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="Não" />

Carregamento assíncrono das tabelas do sistema. Útil se houver uma grande quantidade de tabelas de log e partes no banco de dados `system`. Independente da configuração `async_load_databases`.

* Se definido como `true`, todos os bancos de dados do sistema com motores `Ordinary`, `Atomic` e `Replicated` serão carregados de forma assíncrona após a inicialização do servidor ClickHouse. Consulte a tabela `system.asynchronous_loader` e as configurações do servidor `tables_loader_background_pool_size` e `tables_loader_foreground_pool_size`. Qualquer consulta que tente acessar uma tabela do sistema que ainda não tenha sido carregada aguardará a inicialização dessa tabela. A tabela pela qual pelo menos uma consulta estiver aguardando será carregada com prioridade mais alta. Considere também definir a configuração `max_waiting_queries` para limitar o número total de consultas em espera.
* Se definido como `false`, o banco de dados do sistema é carregado antes da inicialização do servidor.

**Exemplo**

```xml
<async_load_system_database>true</async_load_system_database>
```
