Estas configurações se aplicam ao servidor ClickHouse e são geradas automaticamente a partir do código-fonte do ClickHouse.

<div id="s3_allow_server_credentials_for_system_table_disks">
  ## s3_allow_server_credentials_for_system_table_disks
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

Permite que um `disk(type = s3, ...)` dinâmico de uma tabela no banco de dados `system` use as credenciais
S3 do próprio servidor (obtidas do ambiente), isentando-o da restrição `s3_allow_server_credentials_in_user_queries`. Isso se destina
à infraestrutura interna do servidor que envia tabelas de sistema para o S3 com a identidade do servidor (anexadas pelo
operador de nuvem ao banco de dados `system`). Ao contrário da configuração de sessão, esta é uma configuração
no nível do servidor, portanto a isenção também se aplica quando a tabela é recarregada a partir dos metadados na reinicialização. Ela tem escopo restrito ao banco de dados `system`,
no qual usuários comuns não podem criar tabelas, portanto não flexibiliza a restrição para consultas de usuário.

<div id="s3_load_table_anonymously_if_credentials_restricted">
  ## s3_load_table_anonymously_if_credentials_restricted
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

Controla o que acontece quando uma tabela persistente `S3` ou `S3Queue`, um `disk(type = s3, ...)` dinâmico ou um
banco de dados `DataLakeCatalog` (Glue, BigLake) é carregado a partir de metadados existentes (inicialização do servidor ou `RESTORE`) e
sua definição resolveria as credenciais S3/cloud do próprio servidor (ambientais), que são bloqueadas para
consultas de usuário por `s3_allow_server_credentials_in_user_queries`.

Quando habilitado (o padrão), o objeto é carregado sem essas credenciais em vez de abortar a inicialização: uma
tabela `S3`/`S3Queue` ou um disco S3 dinâmico é criado com um cliente S3 anônimo, e um banco de dados `DataLakeCatalog`
fica com um catálogo indisponível. O servidor inicia, mas o objeto fica inacessível (suas requisições
para um bucket privado são negadas, ou o catálogo relata um erro claro) até que suas credenciais voltem a ser resolvidas para uma
origem permitida. Ele nunca recupera silenciosamente a identidade do servidor. Isso evita que um único objeto desse tipo —
por exemplo, um criado em uma versão mais antiga, antes de a restrição existir, ou um cuja coleção nomeada foi
posteriormente reassociada a `use_environment_credentials = 1` — aborte a inicialização do servidor.

Quando desabilitado, o carregamento desse tipo de objeto falha, o que pode impedir a inicialização do servidor. Use isso apenas
se você preferir uma falha explícita a uma tabela, disco ou banco de dados de catálogo silenciosamente inacessível.
