Estas configurações são usadas para configurar o servidor ClickHouse e são geradas automaticamente a partir do código-fonte do ClickHouse.

<div id="concurrent_threads_lazy_allocation">
  ## concurrent_threads_lazy_allocation
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Sim" />

Controla como os slots de CPU são alocados às consultas.

Quando `true` (padrão), uma consulta começa com um slot de CPU e solicita slots adicionais ao scheduler somente quando seu pipeline realmente passa a enviar mais trabalho paralelizável. Isso evita a situação em que uma consulta reserva até `max_threads` slots, mas acaba usando apenas uma fração deles, deixando outras consultas concorrentes sem recursos. Aplica-se tanto ao controle de concorrência quanto ao scheduler de CPU preemptivo para workloads.

Quando `false`, a consulta solicita todos os slots de `max_threads` logo no início.

<div id="concurrent_threads_scheduler">
  ## concurrent_threads_scheduler
</div>

<SettingsInfoBlock type="String" default_value="max_min_fair" changeable_without_restart="Sim" />

A política de como fazer o agendamento dos slots de CPU especificados por `concurrent_threads_soft_limit_num` e `concurrent_threads_soft_limit_ratio_to_cores`. Algoritmo usado para controlar como um número limitado de slots de CPU é distribuído entre consultas concorrentes. O scheduler pode ser alterado em tempo de execução sem reiniciar o servidor.

Possíveis valores:

* `round_robin` — Toda consulta com a configuração `use_concurrency_control` = 1 aloca até `max_threads` slots de CPU. Um slot por thread. Em caso de contenção, os slots de CPU são concedidos às consultas em round-robin. Observe que o primeiro slot é concedido incondicionalmente, o que pode levar à falta de equidade e ao aumento da latência de consultas com `max_threads` alto quando há um grande número de consultas com `max_threads` = 1.
* `fair_round_robin` — Toda consulta com a configuração `use_concurrency_control` = 1 aloca até `max_threads - 1` slots de CPU. Variação de `round_robin` que não exige um slot de CPU para a primeira thread de cada consulta. Dessa forma, consultas com `max_threads` = 1 não exigem nenhum slot e não podem alocar injustamente todos os slots. Nenhum slot é concedido incondicionalmente.
* `max_min_fair` — Toda consulta com a configuração `use_concurrency_control` = 1 aloca até `max_threads - 1` slots de CPU. Semelhante a `fair_round_robin`, mas os slots liberados são sempre concedidos à consulta com o menor número de slots alocados no momento. Isso oferece mais equidade em cenários de alta sobrealocação, quando muitas consultas competem por slots de CPU limitados. Consultas de curta duração não são prejudicadas por consultas de longa duração que acumularam mais slots ao longo do tempo.

<div id="concurrent_threads_soft_limit_num">
  ## concurrent_threads_soft_limit_num
</div>

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="Sim" />

O número máximo de threads de processamento de consultas, excluindo as threads usadas para recuperar dados de servidores remotos, que podem ser usadas para executar todas as consultas. Este não é um limite rígido. Caso o limite seja atingido, a consulta ainda receberá pelo menos uma thread para ser executada. A consulta poderá ser ampliada até o número desejado de threads durante a execução, se mais threads ficarem disponíveis.

<Note>
  Um valor de `0` (padrão) significa ilimitado.
</Note>

<div id="concurrent_threads_soft_limit_ratio_to_cores">
  ## concurrent_threads_soft_limit_ratio_to_cores
</div>

<SettingsInfoBlock type="UInt64" default_value="2" changeable_without_restart="Sim" />Igual a [`concurrent_threads_soft_limit_num`](/pt-BR/reference/settings/server-settings/settings/concurrent-threads#concurrent_threads_soft_limit_num), mas como razão em relação aos núcleos.
