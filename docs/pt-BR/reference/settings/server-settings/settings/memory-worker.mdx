Estas configurações definem o servidor ClickHouse e são geradas automaticamente a partir do código-fonte do ClickHouse.

<div id="memory_worker_correct_memory_tracker">
  ## memory_worker_correct_memory_tracker
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

Se o worker de memória em segundo plano deve corrigir o rastreador de memória interno com base em informações de fontes externas, como jemalloc e cgroups

<div id="memory_worker_decay_adjustment_period_ms">
  ## memory_worker_decay_adjustment_period_ms
</div>

<SettingsInfoBlock type="UInt64" default_value="5000" changeable_without_restart="No" />

Duração, em milissegundos, pela qual a pressão de memória deve persistir antes de ajustar dinamicamente o `dirty_decay_ms` do jemalloc. Quando o uso de memória permanece acima do limiar de purga durante esse período, o decaimento automático de páginas sujas é desativado (`dirty_decay_ms=0`) para recuperar memória de forma agressiva. Quando o uso permanece abaixo do limiar durante esse período, o comportamento de decaimento padrão é restaurado. Defina como 0 para desativar o ajuste dinâmico e usar as configurações de decaimento padrão do jemalloc.

<div id="memory_worker_dynamic_hard_limit">
  ## memory_worker_dynamic_hard_limit
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

Define se o worker de memória em segundo plano recalcula periodicamente, em tempo de execução, o limite rígido de memória do servidor como `(resident memory + system available memory) * max_server_memory_usage_to_ram_ratio`, para que o servidor reserve uma margem para outros processos em execução no mesmo host.

Quando desativado, `max_server_memory_usage_to_ram_ratio` apenas restringe estaticamente o limite rígido de memória, na inicialização e na recarga da configuração, como nas versões anteriores.

Não tem efeito quando `max_server_memory_usage_to_ram_ratio` é `0`.

<div id="memory_worker_period_ms">
  ## memory_worker_period_ms
</div>

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="No" />

Período de tick do worker de memória em segundo plano, que corrige o uso de memória do rastreador de memória e limpa páginas não utilizadas durante períodos de maior uso de memória. Se definido como 0, será usado o valor padrão, dependendo da origem do uso de memória.

<div id="memory_worker_purge_dirty_pages_threshold_ratio">
  ## memory_worker_purge_dirty_pages_threshold_ratio
</div>

<SettingsInfoBlock type="Double" default_value="0.2" changeable_without_restart="No" />

A razão de limiar das páginas sujas do jemalloc em relação à memória disponível para o servidor ClickHouse. Quando o tamanho das páginas sujas excede essa razão, o worker de memória em segundo plano força a purga dessas páginas. Se definido como 0, a purga forçada com base na razão de páginas sujas é desativada.

<div id="memory_worker_purge_total_memory_threshold_ratio">
  ## memory_worker_purge_total_memory_threshold_ratio
</div>

<SettingsInfoBlock type="Double" default_value="0.9" changeable_without_restart="No" />

A razão de limiar para a purga do jemalloc em relação à memória disponível para o servidor ClickHouse. Quando o uso total de memória excede essa razão, o worker de memória em segundo plano força a purga de páginas sujas. Se definido como 0, a purga forçada com base na memória total é desativada.

<div id="memory_worker_rss_speculative_reserve_ratio">
  ## memory_worker_rss_speculative_reserve_ratio
</div>

<SettingsInfoBlock type="Double" default_value="1" changeable_without_restart="No" />

A cada tick de `MemoryWorker`, reserve uma
`ratio * min(resident - previous_resident, resident - tracked)` adicional sobre o
RSS observado, partindo da suposição de que o próximo tick pode crescer na mesma
quantidade que o último (`resident - previous_resident` é o crescimento do RSS
no último tick). O crescimento é limitado por `resident - tracked`, a parte
do RSS não visível para o rastreador de memória global, porque o crescimento que
já é rastreado é tratado pela verificação comum de limite rígido. A reserva
é aplicada ao contador `rss` que a verificação global de limite rígido consulta
por meio de `MemoryTracker::allocImpl`, de modo que, quando o valor extrapolado ultrapassa
`max_server_memory_usage`, as alocações subsequentes geram
`MEMORY_LIMIT_EXCEEDED` antes que o OOM-killer do kernel seja acionado. Um valor de `0`
desabilita a especulação (voltando para `rss = resident`); o padrão `1`
reserva uma defasagem de crescimento completa como capacidade extra para o próximo intervalo. Em
sanitizers (`ASan`, `UBSan`, `MSan`, `TSan`), o padrão é `0`, porque a
lacuna `resident - tracked` é dominada pela sobrecarga de shadow/runtime do sanitizer,
e não pela defasagem na contabilidade do rastreador.

  ## memory_worker_use_cgroup
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />Usa as informações atuais de uso de memória do cgroup para corrigir o rastreamento de memória.
