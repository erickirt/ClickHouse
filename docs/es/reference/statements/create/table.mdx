Crea una tabla nueva. Esta consulta puede tener varias formas sintГЎcticas segГәn el caso de uso.

De forma predeterminada, las tablas se crean solo en el servidor actual. Las consultas DDL distribuidas se implementan mediante la clГЎusula `ON CLUSTER`, que se [describe por separado](/es/reference/statements/distributed-ddl).

<div id="syntax-forms">
  ## Variantes de sintaxis
</div>

<div id="with-explicit-schema">
  ### Con esquema explГӯcito
</div>

```sql
CREATE TABLE [IF NOT EXISTS] [db.]table_name [ON CLUSTER cluster]
(
    name1 [type1] [NULL|NOT NULL] [DEFAULT|MATERIALIZED|EPHEMERAL|ALIAS expr1] [COMMENT 'comment for column'] [compression_codec] [TTL expr1],
    name2 [type2] [NULL|NOT NULL] [DEFAULT|MATERIALIZED|EPHEMERAL|ALIAS expr2] [COMMENT 'comment for column'] [compression_codec] [TTL expr2],
    ...
) ENGINE = engine
  [COMMENT 'comment for table']
```

Crea una tabla llamada `table_name` en la base de datos `db` o en la base de datos actual si no se ha definido `db`, con la estructura especificada entre corchetes y el motor `engine`.
La estructura de la tabla es una lista de descripciones de columnas, Гӯndices secundarios, proyecciones y restricciones. Si el motor admite [clave primaria](#primary-key), esta se indicarГЎ como parГЎmetro del motor de tabla.

Una descripciГіn de columna es `name type` en el caso mГЎs simple. Ejemplo: `RegionID UInt32`.

TambiГ©n se pueden definir expresiones para los valores predeterminados (vГ©ase mГЎs abajo).

Si es necesario, se puede especificar la clave primaria, con una o mГЎs expresiones de clave.

Se pueden aГұadir comentarios a las columnas y a la tabla.

<div id="with-a-schema-similar-to-other-table">
  ### Con el esquema de una tabla existente
</div>

ClickHouse permite copiar el esquema y los datos de una tabla existente.

Para replicar el esquema de una tabla existente:

```sql
CREATE TABLE [IF NOT EXISTS] [db2.]table_clone AS [db.]table [ENGINE = engine]
```

Esto crea una tabla con la misma estructura que otra.

<div id="with-a-schema-and-data-cloned-from-another-table">
  ### Con el esquema y los datos de una tabla existente
</div>

Para replicar el esquema y los datos de una tabla existente:

```sql
CREATE TABLE [IF NOT EXISTS] [db2.]table_clone CLONE AS [db.]table [ENGINE = engine]
```

Esto crea una tabla con el mismo esquema y los mismos datos que una tabla existente.  Una vez creada la nueva tabla, se le adjuntan todas las particiones de `db.table`. En otras palabras, los datos de `db.table` se clonan en `db2.table_clone` en el momento de su creaciГіn. Esta consulta es equivalente a la siguiente:

```sql
CREATE TABLE [IF NOT EXISTS] [db2.]table_clone AS [db.]table [ENGINE = engine];
ALTER TABLE [db2.]table_clone ATTACH PARTITION ALL FROM [db.]table;
```

Para ambas funcionalidades, puede especificar un motor diferente para la tabla. Si no se especifica el motor, se usarГЎ el mismo motor que para la tabla original (`db.table`).

<div id="from-a-table-function">
  ### A partir de una funciГіn de tabla
</div>

```sql
CREATE TABLE [IF NOT EXISTS] [db.]table_name AS table_function()
```

Crea una tabla con el mismo resultado que la [funciГіn de tabla](/es/reference/functions/table-functions/index) especificada. La tabla creada tambiГ©n funcionarГЎ del mismo modo que la funciГіn de tabla correspondiente que se haya especificado.

<div id="from-select-query">
  ### A partir de una consulta SELECT
</div>

```sql
CREATE TABLE [IF NOT EXISTS] [db.]table_name[(name1 [type1], name2 [type2], ...)] ENGINE = engine AS SELECT ...
```

Crea una tabla con una estructura similar al resultado de la consulta `SELECT`, usa el motor `engine` y la rellena con datos de `SELECT`. TambiГ©n puede especificar explГӯcitamente la definiciГіn de las columnas.

Si la tabla ya existe y se especifica `IF NOT EXISTS`, la consulta no harГЎ nada.

Puede haber otras clГЎusulas despuГ©s de la clГЎusula `ENGINE` en la consulta. Consulte la documentaciГіn detallada sobre cГіmo crear tablas en las descripciones de los [motores de tabla](/es/reference/engines/table-engines/index).

**Ejemplo**

```sql title="Query"
CREATE TABLE t1 (x String) ENGINE = Memory AS SELECT 1;
SELECT x, toTypeName(x) FROM t1;
```

<div id="null-or-not-null-modifiers">
  ## Modificadores `NULL` o `NOT NULL`
</div>

Los modificadores `NULL` y `NOT NULL` despuГ©s del tipo de dato en la definiciГіn de una columna permiten o impiden que sea [Nullable](/es/reference/data-types/nullable).

Si el tipo no es `Nullable` y se especifica `NULL`, se tratarГЎ como `Nullable`; si se especifica `NOT NULL`, no. Por ejemplo, `INT NULL` es equivalente a `Nullable(INT)`. Si el tipo es `Nullable` y se especifican los modificadores `NULL` o `NOT NULL`, se generarГЎ una excepciГіn.

VГ©ase tambiГ©n la opciГіn de configuraciГіn [data&#95;type&#95;default&#95;nullable](/es/reference/settings/session-settings#data_type_default_nullable).

<div id="default_values">
  ## Valores predeterminados
</div>

La descripciГіn de la columna puede especificar una expresiГіn de valor predeterminado en la forma `DEFAULT expr`, `MATERIALIZED expr` o `ALIAS expr`. Ejemplo: `URLDomain String DEFAULT domain(URL)`.

La expresiГіn `expr` es opcional. Si se omite, el tipo de la columna debe especificarse explГӯcitamente y el valor predeterminado serГЎ `0` para las columnas numГ©ricas, `''` (la cadena vacГӯa) para las columnas String, `[]` (el Array vacГӯo) para las columnas Array, `1970-01-01` para las columnas de fecha o `NULL` para las columnas Nullable.

El tipo de la columna con valor predeterminado puede omitirse, en cuyo caso se infiere a partir del tipo de `expr`. Por ejemplo, el tipo de la columna `EventDate DEFAULT toDate(EventTime)` serГЎ date.

Si se especifican tanto un tipo de dato como una expresiГіn de valor predeterminado, se inserta una funciГіn implГӯcita de conversiГіn de tipos que convierte la expresiГіn al tipo especificado. Ejemplo: `Hits UInt32 DEFAULT 0` se representa internamente como `Hits UInt32 DEFAULT toUInt32(0)`.

Una expresiГіn de valor predeterminado `expr` puede hacer referencia a columnas de tabla arbitrarias y constantes. ClickHouse comprueba que los cambios en la estructura de la tabla no introduzcan bucles en el cГЎlculo de la expresiГіn. Para INSERT, comprueba que las expresiones puedan resolverse; es decir, que se hayan pasado todas las columnas a partir de las cuales pueden calcularse.

<div id="ephemeral">
  ### EPHEMERAL
</div>

`EPHEMERAL [expr]`

Columna efГӯmera. Las columnas de este tipo no se almacenan en la tabla y no es posible hacer `SELECT` de ellas. El Гәnico propГіsito de las columnas efГӯmeras es utilizarlas para construir expresiones de valor predeterminado para otras columnas.

Una inserciГіn sin columnas especificadas explГӯcitamente omitirГЎ las columnas de este tipo. Esto preserva la invariante de que el resultado de un `SELECT *` siempre puede volver a insertarse en la tabla mediante `INSERT`.

Ejemplo:

```sql

(
    id UInt64,
    unhexed String EPHEMERAL,
    hexed FixedString(4) DEFAULT unhex(unhexed)
)
ENGINE = MergeTree
ORDER BY id;

INSERT INTO test (id, unhexed) VALUES (1, '5a90b714');

SELECT
    id,
    hexed,
    hex(hexed)
FROM test
FORMAT Vertical;

Row 1:
в”Җв”Җв”Җв”Җв”Җв”Җ

```

<div id="alias">
  ### ALIAS
</div>

`ALIAS expr`

Columnas calculadas (sinГіnimo). Las columnas de este tipo no se almacenan en la tabla y no es posible insertar valores en ellas con `INSERT`.

Cuando las consultas `SELECT` hacen referencia explГӯcita a columnas de este tipo, el valor se calcula en el momento de la consulta a partir de `expr`. De forma predeterminada, `SELECT *` excluye las columnas ALIAS. Este comportamiento se puede desactivar con el ajuste `asterisk_include_alias_columns`.

Al usar la consulta `ALTER` para aГұadir columnas nuevas, no se escriben los datos antiguos de esas columnas. En su lugar, al leer datos antiguos que no tienen valores para las columnas nuevas, las expresiones se calculan sobre la marcha de forma predeterminada. Sin embargo, si para ejecutar las expresiones se necesitan otras columnas que no se indican en la consulta, esas columnas tambiГ©n se leerГЎn, pero solo para los bloques de datos que lo requieran.

Si aГұade una nueva columna a una tabla pero mГЎs tarde cambia su expresiГіn por defecto, los valores usados para los datos antiguos cambiarГЎn (para los datos cuyos valores no se almacenaron en disco). Tenga en cuenta que, al ejecutar fusiones en segundo plano, los datos de las columnas que faltan en una de las partes que se estГЎn fusionando se escriben en la parte fusionada.

No es posible establecer valores predeterminados para elementos de estructuras de datos anidadas.

<div id="primary-key">
  ## Clave primaria
</div>

Puede definir una [clave primaria](/es/reference/engines/table-engines/mergetree-family/mergetree#primary-keys-and-indexes-in-queries) al crear una tabla. La clave primaria se puede especificar de dos maneras:

* En la lista de columnas

```sql
CREATE TABLE [db.]table_name
(
    name1 type1, name2 type2, ...,
    PRIMARY KEY(expr1[, expr2,...])
)
ENGINE = engine;
```

* Fuera de la lista de columnas

```sql
CREATE TABLE [db.]table_name
(
    name1 type1, name2 type2, ...
)
ENGINE = engine
PRIMARY KEY(expr1[, expr2,...]);
```

<Tip>
  No se pueden combinar ambas formas en una sola consulta.
</Tip>

<div id="constraints">
  ## Restricciones
</div>

AdemГЎs de las descripciones de las columnas, se pueden definir restricciones:

<div id="constraint">
  ### CONSTRAINT
</div>

```sql
CREATE TABLE [IF NOT EXISTS] [db.]table_name [ON CLUSTER cluster]
(
    name1 [type1] [DEFAULT|MATERIALIZED|ALIAS expr1] [compression_codec] [TTL expr1],
    ...
    CONSTRAINT constraint_name_1 CHECK boolean_expr_1,
    ...
) ENGINE = engine
```

`boolean_expr_1` puede ser cualquier expresiГіn booleana. Si se definen restricciones para la tabla, cada una de ellas se comprobarГЎ en cada fila de la consulta `INSERT`. Si alguna restricciГіn no se cumple, el servidor lanzarГЎ una excepciГіn con el nombre de la restricciГіn y la expresiГіn de comprobaciГіn.

AГұadir una gran cantidad de restricciones puede afectar negativamente al rendimiento de consultas `INSERT` grandes.

Las restricciones existentes en todas las tablas pueden inspeccionarse mediante la tabla [`system.constraints`](/es/reference/system-tables/constraints).

<div id="assume">
  ### ASSUME
</div>

La clГЎusula `ASSUME` se utiliza para definir una `CONSTRAINT` sobre una tabla que se asume verdadera. Esta restricciГіn puede ser utilizada posteriormente por el optimizador para mejorar el rendimiento de las consultas SQL.

Veamos este ejemplo en el que `ASSUME CONSTRAINT` se usa al crear la tabla `users_a`:

```sql
CREATE TABLE users_a (
    uid Int16, 
    name String, 
    age Int16, 
    name_len UInt8 MATERIALIZED length(name), 
    CONSTRAINT c1 ASSUME length(name) = name_len
) 
ENGINE=MergeTree 
ORDER BY (name_len, name);
```

AquГӯ, `ASSUME CONSTRAINT` se usa para indicar que la funciГіn `length(name)` siempre es igual al valor de la columna `name_len`. Esto significa que, cada vez que se llama a `length(name)` en una consulta, ClickHouse puede sustituirla por `name_len`, lo que deberГӯa ser mГЎs rГЎpido porque evita llamar a la funciГіn `length()`.

Luego, al ejecutar la consulta `SELECT name FROM users_a WHERE length(name) < 5;`, ClickHouse puede optimizarla como `SELECT name FROM users_a WHERE name_len < 5`; gracias a `ASSUME CONSTRAINT`. Esto puede hacer que la consulta se ejecute mГЎs rГЎpido porque evita calcular la longitud de `name` para cada fila.

`ASSUME CONSTRAINT` **no impone la restricciГіn**, simplemente informa al optimizador de que la restricciГіn se cumple. Si la restricciГіn en realidad no es cierta, los resultados de las consultas pueden ser incorrectos. Por lo tanto, solo debes usar `ASSUME CONSTRAINT` si estГЎs seguro de que la restricciГіn es cierta.

<div id="ttl-expression">
  ## ExpresiГіn TTL
</div>

Define el tiempo de almacenamiento de los valores. Solo se puede especificar para tablas de la familia MergeTree. Para obtener una descripciГіn detallada, consulte [TTL para columnas y tablas](/es/reference/engines/table-engines/mergetree-family/mergetree#table_engine-mergetree-ttl).

<div id="column_compression_codec">
  ## CГіdecs de compresiГіn de columnas
</div>

De forma predeterminada, ClickHouse aplica la compresiГіn `lz4` en la versiГіn autogestionada y `zstd` en ClickHouse Cloud.

En la familia de motores `MergeTree`, puede cambiar el mГ©todo de compresiГіn predeterminado en la secciГіn [compression](/es/reference/settings/server-settings/settings#compression) de la configuraciГіn del servidor.

TambiГ©n puede definir el mГ©todo de compresiГіn para cada columna en la consulta `CREATE TABLE`.

```sql
CREATE TABLE codec_example
(
    dt Date CODEC(ZSTD),
    ts DateTime CODEC(LZ4HC),
    float_value Float32 CODEC(NONE),
    double_value Float64 CODEC(LZ4HC(9)),
    value Float32 CODEC(Delta, ZSTD)
)
ENGINE = <Engine>
...
```

El cГіdec `Default` puede especificarse para hacer referencia a la compresiГіn predeterminada, que en tiempo de ejecuciГіn puede depender de distintos ajustes (y de las propiedades de los datos).
Ejemplo: `value UInt64 CODEC(Default)` вҖ” equivale a no especificar ningГәn cГіdec.

TambiГ©n puede quitar el CODEC actual de la columna y usar la compresiГіn predeterminada de config.xml:

```sql
ALTER TABLE codec_example MODIFY COLUMN float_value CODEC(Default);
```

Los cГіdecs pueden combinarse en una pipeline; por ejemplo, `CODEC(Delta, Default)`.

<Tip>
  No es posible descomprimir archivos de bases de datos de ClickHouse con utilidades externas como `lz4`. En su lugar, usa la utilidad especial [clickhouse-compressor](https://github.com/ClickHouse/ClickHouse/tree/master/programs/compressor).
</Tip>

Se admite compresiГіn en los siguientes motores de tabla:

* Familia [MergeTree](/es/reference/engines/table-engines/mergetree-family/mergetree). Admite cГіdecs de compresiГіn por columna y la selecciГіn del mГ©todo de compresiГіn predeterminado mediante la configuraciГіn de [compression](/es/reference/settings/server-settings/settings#compression).
* Familia [Log](/es/reference/engines/table-engines/log-family/index). Usa el mГ©todo de compresiГіn `lz4` de forma predeterminada y admite cГіdecs de compresiГіn por columna.
* [Set](/es/reference/engines/table-engines/special/set). Solo admite la compresiГіn predeterminada.
* [Join](/es/reference/engines/table-engines/special/join). Solo admite la compresiГіn predeterminada.

ClickHouse admite cГіdecs de uso general y cГіdecs especializados.

<div id="general-purpose-codecs">
  ### CГіdecs de uso general
</div>

<div id="none">
  #### NONE
</div>

`NONE` вҖ” Sin compresiГіn.

<div id="lz4">
  #### LZ4
</div>

`LZ4` вҖ” Algoritmo de [compresiГіn de datos](https://github.com/lz4/lz4) sin pГ©rdida usado de forma predeterminada. Utiliza la compresiГіn rГЎpida LZ4.

<div id="lz4hc">
  #### LZ4HC
</div>

`LZ4HC[(level)]` вҖ” algoritmo LZ4 HC (alta compresiГіn) con un nivel configurable. Nivel predeterminado: 9. La opciГіn `level <= 0` aplica el nivel predeterminado. Niveles posibles: [1, 12]. Rango de niveles recomendado: [4, 9].

<div id="zstd">
  #### ZSTD
</div>

`ZSTD[(level)]` вҖ” [algoritmo de compresiГіn ZSTD](https://en.wikipedia.org/wiki/Zstandard) con `level` configurable. Niveles posibles: [1, 22]. Nivel predeterminado: 1.

Los niveles de compresiГіn altos son Гәtiles en escenarios asimГ©tricos, como comprimir una vez y descomprimir repetidamente. Los niveles mГЎs altos implican una mejor compresiГіn y un mayor uso de CPU.

<div id="zxc">
  #### ZXC
</div>

`ZXC[(level)]` вҖ” algoritmo de compresiГіn asimГ©trico [`algoritmo de compresiГіn zxc`](https://github.com/hellobertrand/zxc) con `level` configurable. Niveles posibles: [1, 7]. Nivel predeterminado: 3.

`ZXC` sacrifica velocidad de compresiГіn a cambio de una descompresiГіn muy rГЎpida, con una relaciГіn de compresiГіn entre `LZ4` y `ZSTD`. Es una buena opciГіn para el patrГіn de comprimir una vez y descomprimir muchas veces, y ofrece la descompresiГіn mГЎs rГЎpida en nГәcleos ARM modernos. Los niveles mГЎs altos implican mejor compresiГіn y una compresiГіn mГЎs lenta, mientras que la descompresiГіn sigue siendo rГЎpida.

<div id="specialized-codecs">
  ### CГіdecs especializados
</div>

Estos cГіdecs estГЎn diseГұados para hacer que la compresiГіn sea mГЎs eficaz aprovechando caracterГӯsticas especГӯficas de los datos. Algunos de estos cГіdecs no comprimen los datos por sГӯ mismos, sino que los preprocesan para que una segunda etapa de compresiГіn con un cГіdec de propГіsito general pueda alcanzar una mayor tasa de compresiГіn.

<div id="delta">
  #### Delta
</div>

`Delta(delta_bytes)` вҖ” MГ©todo de compresiГіn en el que los valores originales se sustituyen por la diferencia entre dos valores adyacentes, excepto el primero, que permanece sin cambios. `delta_bytes` es el tamaГұo mГЎximo de los valores originales; el valor predeterminado es `sizeof(type)`. Especificar `delta_bytes` como argumento estГЎ obsoleto y la compatibilidad se eliminarГЎ en una versiГіn futura. Delta es un cГіdec de preparaciГіn de datos, es decir, no puede usarse por sГӯ solo.

<div id="doubledelta">
  #### DoubleDelta
</div>

`DoubleDelta(bytes_size)` вҖ” Calcula el delta de los deltas y lo escribe en formato binario compacto. `bytes_size` tiene un significado similar al de `delta_bytes` en el cГіdec [Delta](#delta). Especificar `bytes_size` como argumento estГЎ obsoleto y esta compatibilidad se eliminarГЎ en una versiГіn futura. Se obtienen tasas de compresiГіn Гіptimas para secuencias monГіtonas con un paso constante, como los datos de series temporales. Puede usarse con cualquier tipo numГ©rico. Implementa el algoritmo utilizado en Gorilla TSDB y lo amplГӯa para admitir tipos de 64 bits. Usa 1 bit adicional para deltas de 32 bits: prefijos de 5 bits en lugar de prefijos de 4 bits. Para obtener mГЎs informaciГіn, consulte Compressing Time Stamps en [Gorilla: A Fast, Scalable, In-Memory Time Series Database](http://www.vldb.org/pvldb/vol8/p1816-teller.pdf). DoubleDelta es un cГіdec de preparaciГіn de datos; es decir, no puede usarse por sГӯ solo.

<div id="gcd">
  #### GCD
</div>

`GCD()` - - Calcula el mГЎximo comГәn divisor (GCD) de los valores de la columna y, a continuaciГіn, divide cada valor por el GCD. Puede usarse con columnas de enteros, decimales y fecha/hora. El cГіdec es especialmente adecuado para columnas con valores que cambian (aumentan o disminuyen) en mГәltiplos del GCD, p. ej., 24, 28, 16, 24, 8, 24 (GCD = 4). GCD es un cГіdec de preparaciГіn de datos, es decir, no puede usarse por sГӯ solo.

<div id="gorilla">
  #### Gorilla
</div>

`Gorilla(bytes_size)` вҖ” Calcula el XOR entre el valor actual y el valor anterior de coma flotante, y lo escribe en formato binario compacto. Cuanto menor sea la diferencia entre valores consecutivos, es decir, cuanto mГЎs lentamente cambien los valores de la serie temporal, mejor serГЎ la tasa de compresiГіn. Implementa el algoritmo utilizado en Gorilla TSDB y lo amplГӯa para admitir tipos de 64 bits. Los valores posibles de `bytes_size` son: 1, 2, 4, 8; el valor predeterminado es `sizeof(type)` si es igual a 1, 2, 4 u 8. En todos los demГЎs casos, es 1. Para obtener mГЎs informaciГіn, consulte la secciГіn 4.1 de [Gorilla: A Fast, Scalable, In-Memory Time Series Database](https://doi.org/10.14778/2824032.2824078).

<div id="alp">
  #### ALP
</div>

<ExperimentalBadge />

`ALP(variant)` вҖ” CompresiГіn adaptativa sin pГ©rdida para datos de coma flotante. Compatible con `Float32` y `Float64`. Para mГЎs detalles, consulta [ALP: Adaptive lossless floating-point compression](https://ir.cwi.nl/pub/33334).

El cГіdec acepta un argumento opcional de variante:

* `ALP()` o `ALP(AUTO)` (predeterminado) вҖ” Usa STD y recurre a RD en funciГіn del tamaГұo comprimido estimado.
* `ALP(STD)` вҖ” Variante estГЎndar de ALP. Representa cada valor como un entero escalado exacto usando potencias decimales y, despuГ©s, comprime los enteros resultantes con Frame-of-Reference y empaquetado de bits. Los valores que no pueden representarse se almacenan como excepciones sin codificar. Funciona mejor con nГәmeros procedentes de valores decimales (p. ej., mediciones, precios).
* `ALP(RD)` вҖ” Variante Real Doubles. Reinterpreta el patrГіn de bits de cada valor y lo divide en una parte alta (signo + exponente + bits superiores de la mantisa) y una parte baja. Las partes altas se codifican mediante diccionario (hasta 8 entradas) y las partes bajas se empaquetan en bits. Funciona mejor cuando muchos valores comparten los mismos bits altos.

<Note>
  Este cГіdec es experimental y requiere `SET allow_experimental_codecs = 1` para poder usarlo.
</Note>

<div id="fpc">
  #### FPC
</div>

`FPC(level, float_size)` - Predice repetidamente el siguiente valor de coma flotante de la secuencia usando el mejor de dos predictores; luego aplica XOR entre el valor real y el valor predicho, y comprime el resultado suprimiendo los ceros iniciales. Al igual que Gorilla, es eficiente para almacenar una serie de valores de coma flotante que cambian lentamente. Para valores de 64 bits (double), FPC es mГЎs rГЎpido que Gorilla; para valores de 32 bits, el rendimiento puede variar. Posibles valores de `level`: 1-28; el valor predeterminado es 12. Posibles valores de `float_size`: 4, 8; el valor predeterminado es `sizeof(type)` si el tipo es Float. En todos los demГЎs casos, es 4. Para obtener una descripciГіn detallada del algoritmo, consulte [High Throughput Compression of Double-Precision Floating-Point Data](https://userweb.cs.txstate.edu/~burtscher/papers/dcc07a.pdf).

<div id="sz3">
  #### SZ3
</div>

<ExperimentalBadge />

`SZ3` or `SZ3(algorithm, error_bound_mode, error_bound)` - Un cГіdec con pГ©rdida, pero con error acotado ([SZ3 Lossy Compressor](https://szcompressor.org/)), para columnas de tipo Float32, Float64, Array(Float32) o Array(Float64). En el caso de las columnas de array, la compresiГіn es mГЎs eficaz cuando todos los arrays tienen la misma longitud (en ese caso, se comprimen como vectores de ancho fijo); los arrays de distinta longitud tambiГ©n son compatibles y se comprimen como una secuencia plana de valores. El cГіdec no se puede aplicar a columnas Map, porque sus claves se corromperГӯan con la compresiГіn con pГ©rdida. Los valores admitidos para &#39;algorithm&#39; son `ALGO_LORENZO_REG`, `ALGO_INTERP_LORENZO` y `ALGO_INTERP`. Los valores admitidos para &#39;error&#95;bound&#95;mode&#39; son `ABS`, `REL`, `PSNR` y `ABS_AND_REL`. El argumento &#39;error&#95;bound&#39; es el error mГЎximo y es de tipo Float64.

<Note>
  Este cГіdec es experimental y requiere `SET allow_experimental_codecs = 1` para poder usarlo.
</Note>

<div id="t64">
  #### T64
</div>

`T64` вҖ” Enfoque de compresiГіn que recorta los bits altos no utilizados de los valores en tipos de datos enteros (incluidos `Enum`, `Date` y `DateTime`). En cada paso de su algoritmo, el cГіdec toma un bloque de 64 valores, los coloca en una matriz de 64x64 bits, la transpone, recorta los bits no utilizados de los valores y devuelve el resto como una secuencia. Los bits no utilizados son los bits que no difieren entre los valores mГЎximo y mГӯnimo en toda la parte de datos para la que se usa la compresiГіn.

Los cГіdecs `DoubleDelta` y `Gorilla` se usan en Gorilla TSDB como componentes de su algoritmo de compresiГіn. El enfoque de Gorilla es eficaz en escenarios en los que hay una secuencia de valores que cambian lentamente junto con sus marcas de tiempo. Las marcas de tiempo se comprimen eficazmente con el cГіdec `DoubleDelta`, y los valores se comprimen eficazmente con el cГіdec `Gorilla`. Por ejemplo, para obtener una tabla almacenada de forma eficiente, puede crearla con la siguiente configuraciГіn:

```sql
CREATE TABLE codec_example
(
    timestamp DateTime CODEC(DoubleDelta),
    slow_values Float32 CODEC(Gorilla)
)
ENGINE = MergeTree()
```

<div id="encryption-codecs">
  ### CГіdecs de cifrado
</div>

En realidad, estos cГіdecs no comprimen los datos, sino que los cifran en disco. Solo estГЎn disponibles cuando se especifica una clave de cifrado mediante la configuraciГіn de [encryption](/es/reference/settings/server-settings/settings#encryption). Tenga en cuenta que el cifrado solo tiene sentido al final de las canalizaciones de cГіdecs, porque, por lo general, los datos cifrados no pueden comprimirse de forma significativa.

CГіdecs de cifrado:

<div id="aes_128_gcm_siv">
  #### AES_128_GCM_SIV
</div>

`CODEC('AES-128-GCM-SIV')` вҖ” Cifra los datos con AES-128 en modo GCM-SIV segГәn [RFC 8452](https://tools.ietf.org/html/rfc8452).

<div id="aes-256-gcm-siv">
  #### AES-256-GCM-SIV
</div>

`CODEC('AES-256-GCM-SIV')` вҖ” Cifra los datos con AES-256 en modo GCM-SIV.

Estos cГіdecs usan un nonce fijo y, por lo tanto, el cifrado es determinista. Esto los hace compatibles con motores con deduplicaciГіn, como [ReplicatedMergeTree](/es/reference/engines/table-engines/mergetree-family/replication), pero tiene una debilidad: cuando el mismo bloque de datos se cifra dos veces, el texto cifrado resultante serГЎ exactamente el mismo, por lo que un adversario que pueda leer el disco podrГЎ detectar esta equivalencia (aunque solo la equivalencia, sin obtener su contenido).

<Note>
  La mayorГӯa de los motores, incluida la familia &quot;*MergeTree&quot;, crean archivos de Гӯndice en disco sin aplicar cГіdecs. Esto significa que el texto en claro aparecerГЎ en disco si se indexa una columna cifrada.
</Note>

<Note>
  Si realiza una consulta SELECT que menciona un valor especГӯfico en una columna cifrada (como en su clГЎusula WHERE), el valor puede aparecer en [system.query&#95;log](/es/reference/system-tables/query_log). Puede que le convenga deshabilitar el registro.
</Note>

**Ejemplo**

```sql
CREATE TABLE mytable
(
    x String CODEC(AES_128_GCM_SIV)
)
ENGINE = MergeTree ORDER BY x;
```

<Note>
  Si es necesario aplicar compresiГіn, debe especificarse explГӯcitamente. De lo contrario, solo se aplicarГЎ cifrado a los datos.
</Note>

**Ejemplo**

```sql
CREATE TABLE mytable
(
    x String CODEC(Delta, LZ4, AES_128_GCM_SIV)
)
ENGINE = MergeTree ORDER BY x;
```

<div id="temporary-tables">
  ## Tablas temporales
</div>

<Note>
  Tenga en cuenta que las tablas temporales no estГЎn replicadas. Como resultado, no hay garantГӯa de que los datos insertados en una tabla temporal estГ©n disponibles en otras rГ©plicas. El caso de uso principal en el que las tablas temporales pueden resultar Гәtiles es para consultar o combinar pequeГұos conjuntos de datos externos durante una sola sesiГіn.
</Note>

ClickHouse admite tablas temporales con las siguientes caracterГӯsticas:

* Las tablas temporales desaparecen al finalizar la sesiГіn, incluso si se pierde la conexiГіn.
* Una tabla temporal usa el motor de tabla Memory cuando no se especifica ningГәn motor, y puede usar cualquier motor de tabla excepto los motores Replicated y `KeeperMap`.
* No se puede especificar la DB para una tabla temporal. Se crea fuera de las bases de datos.
* No es posible crear una tabla temporal con una consulta DDL distribuida en todos los servidores del clГәster (usando `ON CLUSTER`): esta tabla solo existe en la sesiГіn actual.
* Si una tabla temporal tiene el mismo nombre que otra y una consulta especifica el nombre de la tabla sin especificar la DB, se usarГЎ la tabla temporal.
* Para el procesamiento distribuido de consultas, las tablas temporales con motor Memory usadas en una consulta se pasan a los servidores remotos.

Para crear una tabla temporal, use la siguiente sintaxis:

```sql
CREATE [OR REPLACE] TEMPORARY TABLE [IF NOT EXISTS] table_name
(
    name1 [type1] [DEFAULT|MATERIALIZED|ALIAS expr1],
    name2 [type2] [DEFAULT|MATERIALIZED|ALIAS expr2],
    ...
) [ENGINE = engine]
```

En la mayorГӯa de los casos, las tablas temporales no se crean manualmente, sino que se crean al usar datos externos para una consulta o para `(GLOBAL) IN` distribuido. Para obtener mГЎs informaciГіn, consulte las secciones correspondientes

Es posible usar tablas con [ENGINE = Memory](/es/reference/engines/table-engines/special/memory) en lugar de tablas temporales.

<div id="replace-table">
  ## REPLACE TABLE
</div>

La sentencia `REPLACE` permite actualizar una tabla [de forma atГіmica](/es/concepts/core-concepts/glossary#atomicity).

<Note>
  Esta sentencia es compatible con los motores de base de datos [`Atomic`](/es/reference/engines/database-engines/atomic) y [`Replicated`](/es/reference/engines/database-engines/replicated),
  que son los motores de base de datos predeterminados de ClickHouse y ClickHouse Cloud, respectivamente.
</Note>

Normalmente, si necesita eliminar algunos datos de una tabla,
puede crear una tabla nueva y llenarla con una sentencia `SELECT` que no recupere los datos no deseados;
despuГ©s, eliminar la tabla antigua y cambiar el nombre de la nueva.
Este enfoque se muestra en el ejemplo siguiente:

```sql
CREATE TABLE myNewTable AS myOldTable;

INSERT INTO myNewTable
SELECT * FROM myOldTable 
WHERE CounterID <12345;

DROP TABLE myOldTable;

RENAME TABLE myNewTable TO myOldTable;
```

En lugar del enfoque anterior, tambiГ©n es posible usar `REPLACE` (si se usan los motores de base de datos predeterminados) para obtener el mismo resultado:

```sql
REPLACE TABLE myOldTable
ENGINE = MergeTree()
ORDER BY CounterID 
AS
SELECT * FROM myOldTable
WHERE CounterID <12345;
```

<div id="syntax">
  ### Sintaxis
</div>

```sql
{CREATE [OR REPLACE] | REPLACE} TABLE [db.]table_name
```

<Note>
  Todas las formas sintГЎcticas de la sentencia `CREATE` tambiГ©n sirven para esta sentencia. Invocar `REPLACE` sobre una tabla inexistente provocarГЎ un error.
</Note>

<div id="comment-clause">
  ## ClГЎusula COMMENT
</div>

Puede aГұadir un comentario a la tabla cuando la cree.

**Sintaxis**

```sql

(
    name1 type1, name2 type2, ...
)
```

<Note>
  La clГЎusula `COMMENT` debe especificarse **despuГ©s** de cualquier clГЎusula especГӯfica de almacenamiento, como `PARTITION BY`, `ORDER BY` y `SETTINGS` especГӯficos del almacenamiento.

  * `ENGINE`
  * clГЎusulas de almacenamiento
  * `COMMENT`
  * ajustes de la consulta (si los hay)
</Note>

**Ejemplo**

```sql title="Query"

SELECT name, comment FROM system.tables WHERE name = 't1';
```
