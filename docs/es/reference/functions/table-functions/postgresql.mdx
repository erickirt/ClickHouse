Permite ejecutar consultas `SELECT` e `INSERT` sobre datos almacenados en un servidor PostgreSQL remoto.

<div id="syntax">
  ## Sintaxis
</div>

```sql
postgresql({host:port, database, table, user, password[, schema, [, on_conflict]] | named_collection[, option=value [,..]]} [, SETTINGS name=value, ...])
```

<div id="arguments">
  ## Argumentos
</div>

| Argumento     | DescripciГіn                                                                                                                                             |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `host:port`   | DirecciГіn del servidor de PostgreSQL.                                                                                                                   |
| `database`    | Nombre de la base de datos remota.                                                                                                                      |
| `table`       | Nombre de la tabla remota o una consulta pasada a PostgreSQL tal cual (consulte [Pasar una consulta en lugar de un nombre de tabla](#passing-a-query)). |
| `user`        | Usuario de PostgreSQL.                                                                                                                                  |
| `password`    | ContraseГұa del usuario.                                                                                                                                 |
| `schema`      | Esquema de tabla distinto del predeterminado. Opcional.                                                                                                 |
| `on_conflict` | Estrategia de resoluciГіn de conflictos. Ejemplo: `ON CONFLICT DO NOTHING`. Opcional.                                                                    |

Los argumentos tambiГ©n pueden pasarse mediante [colecciones con nombre](/es/concepts/features/configuration/server-config/named-collections). En este caso, `host` y `port` deben especificarse por separado. Este enfoque se recomienda para entornos de producciГіn.

<div id="returned_value">
  ## Valor devuelto
</div>

Un objeto de tipo tabla con las mismas columnas que la tabla original de PostgreSQL.

<Note>
  En la consulta `INSERT`, para distinguir la funciГіn de tabla `postgresql(...)` del nombre de la tabla con una lista de nombres de columnas, debe usar las palabras clave `FUNCTION` o `TABLE FUNCTION`. Vea los ejemplos a continuaciГіn.
</Note>

<div id="settings">
  ## ConfiguraciГіn
</div>

El pool de conexiones que utiliza la funciГіn de tabla `postgresql` (y el motor de tabla [`PostgreSQL`](/es/reference/engines/table-engines/integrations/postgresql)) puede configurarse con una clГЎusula `SETTINGS` al final. Si no se especifica una configuraciГіn, se usa de forma predeterminada el valor de la configuraciГіn `postgresql_*` correspondiente a nivel de consulta. Consulte la secciГіn [ConfiguraciГіn](/es/reference/engines/table-engines/integrations/postgresql#settings) del motor de tabla para ver la lista completa de las configuraciones `postgresql_connection_pool_*` y `postgresql_connection_attempt_timeout`, asГӯ como sus valores predeterminados.

Ejemplo:

```sql
SELECT * FROM postgresql('localhost:5432', 'test', 'test', 'postgresql_user', 'password', SETTINGS postgresql_connection_pool_size = 32);
```

<div id="implementation-details">
  ## Detalles de implementaciГіn
</div>

Las consultas `SELECT` del lado de PostgreSQL se ejecutan como `COPY (SELECT ...) TO STDOUT` dentro de una transacciГіn de PostgreSQL de solo lectura, con commit despuГ©s de cada consulta `SELECT`.

Las clГЎusulas `WHERE` simples, como `=`, `!=`, `>`, `>=`, `<`, `<=` e `IN`, se ejecutan en el servidor PostgreSQL.

Todos los joins, las agregaciones, la ordenaciГіn, las condiciones `IN [ array ]` y la restricciГіn de muestreo `LIMIT` se ejecutan en ClickHouse solo despuГ©s de que finaliza la consulta a PostgreSQL.

<div id="passing-a-query">
  ## Pasar una consulta en lugar del nombre de una tabla
</div>

En lugar del nombre de una tabla, el tercer argumento puede ser una consulta `SELECT` que se pasa a PostgreSQL tal cual. La estructura de la tabla resultante se infiere a partir del resultado de la consulta. La consulta puede escribirse como una subconsulta o envolverse en la funciГіn `query`:

```sql
SELECT * FROM postgresql('localhost:5432', 'test', (SELECT a, b FROM t1 JOIN t2 USING (id) WHERE a > 0), 'user', 'password');
SELECT * FROM postgresql('localhost:5432', 'test', query('SELECT a, b FROM t1 JOIN t2 USING (id) WHERE a > 0'), 'user', 'password');
```

Esto es Гәtil para hacer pushdown de joins, agregaciones o cualquier otro procesamiento a PostgreSQL. Dicha tabla es de solo lectura: no se permite hacer `INSERT` en ella. La misma sintaxis es compatible con el motor de tabla [`PostgreSQL`](/es/reference/engines/table-engines/integrations/postgresql).

<Note>
  La forma de subconsulta `(SELECT ...)` es analizada por ClickHouse y se vuelve a serializar en el dialecto de PostgreSQL (entrecomillado de identificadores de PostgreSQL y escaping de literales de cadena) antes de enviarse al servidor. Por lo tanto, debe ser una consulta ClickHouse SQL vГЎlida. Para pasar sintaxis especГӯfica de PostgreSQL que ClickHouse no analiza, usa la forma `query('...')`, cuyo texto se envГӯa a PostgreSQL literalmente.

  Cualquier `WHERE`, `LIMIT`, agregaciГіn, etc., externos de la consulta de ClickHouse circundante **no** se hacen pushdown en la consulta pasada; se aplican en ClickHouse despuГ©s de recuperar el resultado completo de la consulta. Para restringir los datos leГӯdos desde PostgreSQL, coloca el filtro dentro de la consulta pasada. Con [`external_table_strict_query = 1`](/es/reference/settings/session-settings#external_table_strict_query), un filtro externo que no puede hacerse pushdown se rechaza con una excepciГіn en lugar de aplicarse localmente.
</Note>

Las consultas `INSERT` del lado de PostgreSQL se ejecutan como `COPY "table_name" (field1, field2, ... fieldN) FROM STDIN` dentro de una transacciГіn de PostgreSQL, con autocommit despuГ©s de cada sentencia `INSERT`.

Los tipos Array de PostgreSQL se convierten en arrays de ClickHouse.

<Note>
  Ten cuidado: en PostgreSQL, una columna de tipo array como Integer[] puede contener arrays de distintas dimensiones en diferentes filas, pero en ClickHouse solo se permite tener arrays multidimensionales de la misma dimensiГіn en todas las filas.
</Note>

Admite varias rГ©plicas, que deben listarse con `|`. Por ejemplo:

```sql
SELECT name FROM postgresql(`postgres{1|2|3}:5432`, 'postgres_database', 'postgres_table', 'user', 'password');
```

o

```sql
SELECT name FROM postgresql(`postgres1:5431|postgres2:5432`, 'postgres_database', 'postgres_table', 'user', 'password');
```

Admite prioridades de rГ©plicas para la fuente de diccionario de PostgreSQL. Cuanto mayor sea el nГәmero en el mapa, menor serГЎ la prioridad. La prioridad mГЎs alta es `0`.

<div id="examples">
  ## Ejemplos
</div>

Tabla en PostgreSQL:

```text

"int_id" SERIAL,
"int_nullable" INT NULL DEFAULT NULL,
"float" FLOAT NOT NULL,
"str" VARCHAR(100) NOT NULL DEFAULT '',

CREATE TABLE

postgres=# INSERT INTO test (int_id, str, "float") VALUES (1,'test',2);
INSERT 0 1

postgresql> SELECT * FROM test;
  int_id | int_nullable | float | str  | float_nullable

       1 |              |     2 | test |
```

Seleccionar datos de ClickHouse con argumentos simples:

```sql
SELECT * FROM postgresql('localhost:5432', 'test', 'test', 'postgresql_user', 'password') WHERE str IN ('test');
```

O bien usando [colecciones con nombre](/es/concepts/features/configuration/server-config/named-collections):

```sql
CREATE NAMED COLLECTION mypg AS
        host = 'localhost',
        port = 5432,
        database = 'test',
        user = 'postgresql_user',
        password = 'password';
SELECT * FROM postgresql(mypg, table='test') WHERE str IN ('test');
```
