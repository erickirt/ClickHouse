<div id="description">
  ## Descripción
</div>

Contiene una lista de todas las operaciones `BACKUP` o `RESTORE`, con sus estados actuales y otras propiedades. Tenga en cuenta que esta tabla no es persistente y solo muestra las operaciones ejecutadas después del último reinicio del servidor.

<div id="columns">
  ## Columnas
</div>

{/*AUTOGENERATED_START*/}

* `id` ([String](/es/reference/data-types/index)) — ID de la operación; se puede pasar mediante SETTINGS id=... o ser un UUID generado aleatoriamente.
* `name` ([String](/es/reference/data-types/index)) — Nombre de la operación, una cadena como `Disk('backups', 'my_backup')`
* `base_backup_name` ([String](/es/reference/data-types/index)) — Nombre de la operación de copia de seguridad base, una cadena como `Disk('backups', 'my_base_backup')`
* `query_id` ([String](/es/reference/data-types/index)) — Query ID de la consulta que inició la copia de seguridad.
* `status` ([Enum8(&#39;CREATING&#95;BACKUP&#39; = 0, &#39;BACKUP&#95;CREATED&#39; = 1, &#39;BACKUP&#95;FAILED&#39; = 2, &#39;RESTORING&#39; = 3, &#39;RESTORED&#39; = 4, &#39;RESTORE&#95;FAILED&#39; = 5, &#39;BACKUP&#95;CANCELLED&#39; = 6, &#39;RESTORE&#95;CANCELLED&#39; = 7)](/es/reference/data-types/index)) — Estado de la operación de copia de seguridad o restauración.
* `error` ([String](/es/reference/data-types/index)) — El mensaje de error, si lo hay.
* `start_time` ([DateTime64(6)](/es/reference/data-types/index)) — Momento en que se inició la operación.
* `end_time` ([DateTime64(6)](/es/reference/data-types/index)) — Momento en que finalizó la operación.
* `num_files` ([UInt64](/es/reference/data-types/index)) — Número de archivos almacenados en la copia de seguridad.
* `total_size` ([UInt64](/es/reference/data-types/index)) — Tamaño total de los archivos almacenados en la copia de seguridad.
* `num_entries` ([UInt64](/es/reference/data-types/index)) — Número de entradas de la copia de seguridad; es decir, el número de archivos dentro de la carpeta si la copia de seguridad se almacena como una carpeta.
* `uncompressed_size` ([UInt64](/es/reference/data-types/index)) — Tamaño sin comprimir de la copia de seguridad.
* `compressed_size` ([UInt64](/es/reference/data-types/index)) — Tamaño comprimido de la copia de seguridad.
* `files_read` ([UInt64](/es/reference/data-types/index)) — Devuelve el número de archivos leídos durante RESTORE desde esta copia de seguridad.
* `bytes_read` ([UInt64](/es/reference/data-types/index)) — Devuelve el tamaño total de los archivos leídos durante RESTORE desde esta copia de seguridad.
* `ProfileEvents` ([Map(LowCardinality(String), UInt64)](/es/reference/data-types/index)) — Todos los profile events capturados durante esta operación.
* `settings` ([Map(LowCardinality(String), String)](/es/reference/data-types/index)) — Configuración específica de la copia de seguridad/restauración usada efectivamente para esta operación (de la cláusula `SETTINGS`, incluidos los valores predeterminados). La configuración sensible no se expone.
* `engine_settings` ([Map(LowCardinality(String), String)](/es/reference/data-types/index)) — Configuración usada efectivamente por el lector/escritor del engine de copia de seguridad (p. ej., S3 `allow_native_copy`). Vacío cuando la operación involucra más de un engine que un mapa plano no puede representar: copias de seguridad incrementales y restauraciones, restauraciones ligeras de snapshots y operaciones `ON CLUSTER` no internas.

<div id="restore-atomicity">
  ## Atomicidad de la restauración
</div>

`RESTORE` no es transaccional y no revierte los cambios en caso de error. En cada tabla, todas las partes seleccionadas se copian antes de adjuntar cualquiera de ellas, pero la fase de adjuntar en sí no es transaccional: las partes pasan a ser visibles una a una. Las tablas se procesan de forma independiente.

**Las tablas son independientes.** Una tabla cuya restauración se completa permanece en su lugar aunque más adelante falle otra tabla del mismo comando:

```sql
RESTORE TABLE db.t0, TABLE db.t1
FROM S3('<endpoint>', '<access_key>', '<secret_key>')
SETTINGS
    allow_non_empty_tables = true;
```

Si este comando falla después de que `db.t0` se haya restaurado por completo, pero `db.t1` aún no haya terminado, `db.t0` sigue restaurada.

**La cláusula `PARTITIONS` no marca un límite de commit.** Solo selecciona qué partes de una tabla se restauran:

```sql
RESTORE TABLE db.t0 PARTITIONS '2026-06-01', '2026-06-02', '2026-06-03'
FROM S3('<endpoint>', '<access_key>', '<secret_key>')
SETTINGS
    allow_non_empty_tables = true;
```

Primero se copian todas las partes seleccionadas de la tabla y solo se adjuntan cuando todas están listas. Por lo tanto, si este comando falla durante la fase de copia —por ejemplo, después de que la partición `2026-06-01` se haya copiado por completo pero `2026-06-02` y `2026-06-03` no hayan terminado— entonces **no** se hace commit de `2026-06-01` y la tabla queda sin datos restaurados por este comando. Una vez que finaliza la fase de copia y comienza el paso de adjuntar, se hace commit de las partes de una en una, por lo que un fallo durante el adjuntar puede dejar la tabla parcialmente restaurada, sin rollback.

Para hacer commit de las particiones de forma independiente (de modo que una partición completada sobreviva a un fallo posterior y pueda reintentarse de forma aislada), ejecute un `RESTORE` independiente por partición, usando `SETTINGS allow_non_empty_tables = true` después de la primera.
