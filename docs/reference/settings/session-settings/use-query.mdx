## use_query_cache {#use_query_cache}

<SettingsInfoBlock type="Bool" default_value="0" />

If turned on, `SELECT` queries may utilize the [query cache](/concepts/features/performance/caches/query-cache). Parameters [enable_reads_from_query_cache](/reference/settings/session-settings/enable#enable_reads_from_query_cache)
and [enable_writes_to_query_cache](/reference/settings/session-settings/enable#enable_writes_to_query_cache) control in more detail how the cache is used.

Possible values:

- 0 - Disabled
- 1 - Enabled

## use_query_condition_cache {#use_query_condition_cache}

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "25.4"},{"label": "1"},{"label": "A new optimization"}]}, {"id": "row-2","items": [{"label": "25.3"},{"label": "0"},{"label": "New setting."}]}]}/>

Enable the [query condition cache](/concepts/features/performance/caches/query-condition-cache). The cache stores ranges of granules in data parts which do not satisfy the condition in the `WHERE` clause,
and reuse this information as an ephemeral index for subsequent queries.

Possible values:

- 0 - Disabled
- 1 - Enabled
