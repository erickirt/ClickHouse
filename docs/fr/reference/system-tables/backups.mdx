<div id="description">
  ## Description
</div>

Contient la liste de toutes les opérations `BACKUP` ou `RESTORE`, avec leur état actuel et d&#39;autres propriétés. Notez que cette table n&#39;est pas persistante et qu&#39;elle n&#39;affiche que les opérations exécutées depuis le dernier redémarrage du serveur.

<div id="columns">
  ## Colonnes
</div>

{/*AUTOGENERATED_START*/}

* `name` ([String](/fr/reference/data-types/index)) — Nom de l’opération, une chaîne comme `Disk('backups', 'my_backup')`
* `base_backup_name` ([String](/fr/reference/data-types/index)) — Nom de l’opération de sauvegarde de base, une chaîne comme `Disk('backups', 'my_base_backup')`

<div id="restore-atomicity">
  ## Atomicité de la restauration
</div>

`RESTORE` n&#39;est pas transactionnel et n&#39;effectue pas d&#39;annulation en cas d&#39;échec. Pour chaque table, toutes les parts sélectionnées sont copiées avant que la moindre ne soit attachée, mais la phase d&#39;attachement elle-même n&#39;est pas transactionnelle — les parts deviennent visibles une par une. Les tables sont traitées indépendamment.

**Les tables sont indépendantes.** Une table dont la restauration aboutit reste en place même si une autre table de la même commande échoue ensuite :

```sql
RESTORE TABLE db.t0, TABLE db.t1
FROM S3('<endpoint>', '<access_key>', '<secret_key>')
SETTINGS
    allow_non_empty_tables = true;
```

Si cette commande échoue après la restauration complète de `db.t0`, mais avant la fin de celle de `db.t1`, `db.t0` reste restaurée.

**La clause `PARTITIONS` n&#39;est pas une frontière de commit.** Elle sert uniquement à sélectionner les parts d&#39;une table à restaurer :

```sql
RESTORE TABLE db.t0 PARTITIONS '2026-06-01', '2026-06-02', '2026-06-03'
FROM S3('<endpoint>', '<access_key>', '<secret_key>')
SETTINGS
    allow_non_empty_tables = true;
```

Toutes les parts sélectionnées de la table sont d’abord copiées, puis attachées uniquement une fois qu’elles sont toutes prêtes. Ainsi, si cette commande échoue pendant la phase de copie — par exemple après que la partition `2026-06-01` a été entièrement copiée, mais avant que `2026-06-02` et `2026-06-03` ne soient terminées — alors `2026-06-01` n’est **pas** validée et la table ne contient aucune donnée restaurée par cette commande. Une fois la phase de copie terminée et l’étape d’attachement commencée, les parts sont validées une par une ; un échec pendant l’attachement peut donc laisser la table partiellement restaurée, sans possibilité d’annulation.

Pour valider les partitions indépendamment (afin qu’une partition terminée survive à un échec ultérieur et puisse être retentée isolément), exécutez un `RESTORE` distinct pour chaque partition, en utilisant `SETTINGS allow_non_empty_tables = true` après la première.
