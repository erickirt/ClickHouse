Ces paramètres configurent le serveur ClickHouse et sont générés automatiquement à partir du code source de ClickHouse.

<div id="max_server_memory_usage">
  ## max_server_memory_usage
</div>

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="Yes" />

Quantité maximale de mémoire que le serveur est autorisé à utiliser, exprimée en octets.

<Note>
  La consommation maximale de mémoire du serveur est également limitée par le paramètre `max_server_memory_usage_to_ram_ratio`.
</Note>

Cas particulier : une valeur de `0` (par défaut) signifie que le serveur peut consommer toute la mémoire disponible (hors restrictions supplémentaires imposées par `max_server_memory_usage_to_ram_ratio`).

<div id="max_server_memory_usage_to_ram_ratio">
  ## max_server_memory_usage_to_ram_ratio
</div>

<SettingsInfoBlock type="Double" default_value="0.9" changeable_without_restart="No" />

Quantité maximale de mémoire que le serveur est autorisé à utiliser, exprimée sous forme de ratio par rapport à toute la mémoire disponible.

Par exemple, une valeur de `0.9` (par défaut) signifie que le serveur peut consommer 90 % de la mémoire disponible.

Permet de réduire l’utilisation de la mémoire sur les systèmes disposant de peu de mémoire.
Sur les hôtes avec peu de RAM et de swap, vous devrez parfois définir [`max_server_memory_usage_to_ram_ratio`](/fr/reference/settings/server-settings/settings/max-server-memory-usage#max_server_memory_usage_to_ram_ratio) sur une valeur supérieure à 1.

Ce ratio s’applique de deux façons :

* Au démarrage (et lors du rechargement de la configuration), il plafonne la hard memory limit du serveur à cette fraction de la RAM physique totale, comme décrit ci-dessus.
* À l’exécution, le background memory worker recalcule périodiquement la hard memory limit selon la formule `(resident memory + system available memory) * max_server_memory_usage_to_ram_ratio`, afin que le serveur conserve une marge pour les autres processus exécutés sur le même hôte. Lorsqu’il s’exécute dans un cgroup avec une memory limit finie, c’est la mémoire disponible du cgroup qui est utilisée à la place de la valeur à l’échelle de l’hôte. À mesure que d’autres processus grossissent et que la mémoire disponible diminue, la hard memory limit du serveur diminue elle aussi, ce qui plafonne la croissance avant que l’hôte (ou le cgroup) ne soit à court de mémoire.

Définir ce ratio sur `0` désactive à la fois le plafonnement au démarrage et l’ajustement à l’exécution. L’ajustement à l’exécution est également sans effet sur les systèmes non-Linux et dans `clickhouse-keeper`, qui n’expose pas ce paramètre. Pour conserver le plafonnement statique au démarrage/rechargement tout en désactivant l’ajustement à l’exécution (comportement des versions précédentes), définissez `memory_worker_dynamic_hard_limit` sur `0`.

<Note>
  La consommation maximale de mémoire du serveur est également limitée par le paramètre `max_server_memory_usage`.
</Note>
