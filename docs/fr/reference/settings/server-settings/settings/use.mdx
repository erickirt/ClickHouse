Ces paramètres configurent le serveur ClickHouse et sont générés automatiquement à partir du code source de ClickHouse.

<div id="use_minimalistic_part_header_in_zookeeper">
  ## use_minimalistic_part_header_in_zookeeper
</div>

Méthode de stockage des en-têtes des parties de données dans ZooKeeper. Ce paramètre s&#39;applique uniquement à la famille [`MergeTree`](/fr/reference/engines/table-engines/mergetree-family/index). Il peut être défini :

**Globalement dans la section [merge&#95;tree](/fr/reference/settings/server-settings/settings/merge#merge_tree) du fichier `config.xml`**

ClickHouse utilise ce paramètre pour toutes les tables du serveur. Vous pouvez le modifier à tout moment. Les tables existantes changent de comportement lorsque ce paramètre est modifié.

**Pour chaque table**

Lors de la création d&#39;une table, spécifiez le [paramètre de moteur](/fr/reference/engines/table-engines/mergetree-family/mergetree#table_engine-mergetree-creating-a-table) correspondant. Le comportement d&#39;une table existante dotée de ce paramètre ne change pas, même si le paramètre global est modifié.

**Valeurs possibles**

* `0` — La fonctionnalité est désactivée.
* `1` — La fonctionnalité est activée.

Si [`use_minimalistic_part_header_in_zookeeper = 1`](/fr/reference/settings/server-settings/settings/use#use_minimalistic_part_header_in_zookeeper), les tables [répliquées](/fr/reference/engines/table-engines/mergetree-family/replication) stockent les en-têtes des parties de données de manière compacte à l&#39;aide d&#39;un seul `znode`. Si la table contient de nombreuses colonnes, cette méthode de stockage réduit considérablement le volume de données stockées dans ZooKeeper.

<Note>
  Après avoir appliqué `use_minimalistic_part_header_in_zookeeper = 1`, vous ne pouvez pas revenir à une version antérieure du serveur ClickHouse qui ne prend pas en charge ce paramètre. Soyez prudent lors de la mise à niveau de ClickHouse sur les serveurs d&#39;un cluster. Ne mettez pas tous les serveurs à niveau en même temps. Il est plus sûr de tester les nouvelles versions de ClickHouse dans un environnement de test, ou sur seulement quelques serveurs du cluster.

  Les en-têtes des parties de données déjà stockés avec ce paramètre ne peuvent pas être rétablis dans leur représentation précédente (non compacte).
</Note>

<div id="use_separate_cache_arena">
  ## use_separate_cache_arena
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

Active une arène jemalloc dédiée pour les allocations de cache (mark cache, uncompressed cache, page cache).
Isole les données de cache des allocations liées au traitement des requêtes, ce qui réduit la fragmentation mémoire.

<div id="use_shared_merge_tree_log_pipeline">
  ## use_shared_merge_tree_log_pipeline
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

Disponible uniquement sur ClickHouse Cloud. Lorsqu’elle est activée, les tables `system.*_log` sont prises en charge par `SharedMergeTree` via un pipeline basé sur S3.
Pour chaque log `<log>`, les objets suivants sont créés automatiquement au démarrage :

* `system.<log>_s3` — une table adossée à `S3` qui sert de cible directe au flush ; chaque
  appel à `SYSTEM FLUSH LOGS` y écrit un nouveau fichier partitionné.
* `system.<log>_s3queue` — une table `S3Queue` (mode ordonné) qui récupère les fichiers depuis
  `<log>_s3` et achemine les lignes dans le pipeline. Chaque nœud ne traite que ses propres fichiers via
  un partitionnement basé sur `partition_regex`.
* `system.<log>_mv` — une `MATERIALIZED VIEW` qui achemine les lignes de `<log>_s3queue` vers
  la table `SharedMergeTree` finale.
* `system.<log>` — la table `SharedMergeTree` finale où les lignes s’accumulent et sont interrogées.

En cas de modification du schéma ou des paramètres, la table concernée est renommée en `<log>_0`, `<log>_1`, etc.
puis recréée, conformément au comportement existant de rotation de `SystemLog`.

Nécessite que `<shared_log_pipeline><endpoint>` soit défini dans la configuration du serveur.
Voir aussi : `shared_log_pipeline.enable_sync_flush`, `shared_log_pipeline.flush_timeout_seconds`.
