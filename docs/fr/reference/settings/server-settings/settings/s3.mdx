Ces paramètres configurent le serveur ClickHouse et sont générés automatiquement à partir du code source de ClickHouse.

<div id="s3_allow_server_credentials_for_system_table_disks">
  ## s3_allow_server_credentials_for_system_table_disks
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

Permet à un `disk(type = s3, ...)` dynamique d&#39;une table de la base de données `system` d&#39;utiliser les propres identifiants S3 du serveur (ambient),
en l&#39;exemptant de la restriction `s3_allow_server_credentials_in_user_queries`. Ce paramètre est destiné à l&#39;infrastructure interne du serveur qui envoie des system tables vers S3 avec l&#39;identité du serveur (attachée par le
cloud operator dans la base de données `system`). Contrairement au paramètre de session, il s&#39;agit d&#39;un server-level setting, donc
l&#39;exemption s&#39;applique également lorsque la table est rechargée à partir des métadonnées au redémarrage. Sa portée est limitée à la
base de données `system`, dans laquelle les utilisateurs ordinaires ne peuvent pas créer de tables ; elle n&#39;assouplit donc pas la restriction pour les requêtes utilisateur.

<div id="s3_load_table_anonymously_if_credentials_restricted">
  ## s3_load_table_anonymously_if_credentials_restricted
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

Contrôle ce qui se passe lorsqu’une table persistante `S3` ou `S3Queue`, un `disk(type = s3, ...)` dynamique, ou une base de données `DataLakeCatalog` (Glue, BigLake) est chargée à partir de métadonnées existantes (démarrage du serveur ou `RESTORE`) et que sa définition résoudrait les identifiants S3/Cloud propres au serveur (ambient), bloqués pour les requêtes utilisateur par `s3_allow_server_credentials_in_user_queries`.

Lorsqu’elle est activée (par défaut), l’objet est chargé sans ces identifiants au lieu d’interrompre le démarrage : une table `S3`/`S3Queue` ou un disque S3 dynamique est construit avec un client S3 anonyme, et une base de données `DataLakeCatalog` conserve un catalogue indisponible. Le serveur démarre, mais l’objet est inaccessible (ses requêtes vers un bucket privé sont refusées, ou le catalogue signale clairement une erreur) jusqu’à ce que ses identifiants soient de nouveau résolus à partir d’une source autorisée. Il ne récupère jamais silencieusement l’identité du serveur. Cela évite qu’un seul objet de ce type — par exemple créé dans une ancienne version avant l’existence de cette restriction, ou dont la named collection a ensuite été réassociée à `use_environment_credentials = 1` — n’empêche le démarrage du serveur.

Lorsqu’elle est désactivée, le chargement d’un tel objet échoue à la place, ce qui peut empêcher le serveur de démarrer. Utilisez ce réglage uniquement si vous préférez un échec franc à une table, un disque ou une base de données de catalogue rendus silencieusement inaccessibles.
