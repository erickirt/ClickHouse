Ces paramètres configurent le serveur ClickHouse et sont générés automatiquement à partir du code source de ClickHouse.

<div id="interserver_listen_host">
  ## interserver_listen_host
</div>

Restriction des hôtes autorisés à échanger des données entre les serveurs ClickHouse.
Si Keeper est utilisé, la même restriction s&#39;applique à la communication entre différentes instances de Keeper.

<Note>
  Par défaut, la valeur est celle du paramètre [`listen_host`](/fr/reference/settings/server-settings/settings/listen#listen_host).
</Note>

**Exemple**

```xml
<interserver_listen_host>::ffff:a00:1</interserver_listen_host>
<interserver_listen_host>10.0.0.1</interserver_listen_host>
```

Type :

Valeur par défaut :

<div id="interserver_tables_status_require_auth">
  ## interserver_tables_status_require_auth
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

Exige que le `TablesStatusRequest` interserveur soit authentifié avec le
`<secret>` du cluster. Les clients suffisamment récents pour envoyer un hash du secret (révision du protocole
`DBMS_MIN_REVISION_WITH_INTERSERVER_SECRET_TABLES_STATUS`) sont toujours validés ; ce
paramètre rejette en outre les anciens clients qui n’envoient aucun hash, ce qui
empêche par défaut la divulgation non authentifiée de l’état des tables.

La valeur par défaut est `true` (sécurisé par défaut). Lors d’une mise à niveau progressive, un
nœud qui n’a pas encore été mis à niveau utilise l’ancien protocole et n’envoie aucun hash ; ainsi, une requête `Distributed` lancée depuis
un tel nœud vers un nœud déjà mis à niveau verrait son `TablesStatusRequest`
rejeté. Si vous devez exécuter un cluster avec des versions mixtes, définissez ce paramètre sur `false` sur les
nœuds mis à niveau jusqu’à ce que tous les nœuds le soient, puis supprimez la surcharge.
