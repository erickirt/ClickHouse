Ces paramètres configurent le serveur ClickHouse et sont générés automatiquement à partir du code source de ClickHouse.

<div id="async_load_databases">
  ## async_load_databases
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

Chargement asynchrone des bases de données et des tables.

* Si `true`, toutes les bases de données non système avec le moteur `Ordinary`, `Atomic` ou `Replicated` seront chargées de manière asynchrone après le démarrage du serveur ClickHouse. Voir la table `system.asynchronous_loader`, ainsi que les paramètres du serveur `tables_loader_background_pool_size` et `tables_loader_foreground_pool_size`. Toute requête qui tente d’accéder à une table qui n’est pas encore chargée attendra précisément l’initialisation de cette table. Si la tâche de chargement échoue, la requête relancera une erreur (au lieu d’arrêter l’ensemble du serveur dans le cas de `async_load_databases = false`). Une table attendue par au moins une requête sera chargée avec une priorité plus élevée. Les requêtes DDL sur une base de données attendront précisément l’initialisation de cette base de données. Pensez également à définir une limite `max_waiting_queries` pour le nombre total de requêtes en attente.
* Si `false`, toutes les bases de données sont chargées au démarrage du serveur.

**Exemple**

```xml
<async_load_databases>true</async_load_databases>
```

<div id="async_load_system_database">
  ## async_load_system_database
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

Chargement asynchrone des tables système. Utile lorsqu&#39;il y a un grand nombre de tables de logs et de parts dans la base de données `system`. Indépendant du paramètre `async_load_databases`.

* Si défini sur `true`, toutes les bases de données système avec les moteurs `Ordinary`, `Atomic` et `Replicated` seront chargées de manière asynchrone après le démarrage du serveur ClickHouse. Voir la table `system.asynchronous_loader`, ainsi que les paramètres du serveur `tables_loader_background_pool_size` et `tables_loader_foreground_pool_size`. Toute requête tentant d&#39;accéder à une table système qui n&#39;est pas encore chargée attendra que cette table soit précisément initialisée. La table attendue par au moins une requête sera chargée avec une priorité plus élevée. Pensez également à définir le paramètre `max_waiting_queries` pour limiter le nombre total de requêtes en attente.
* Si défini sur `false`, la base de données système est chargée avant le démarrage du serveur.

**Exemple**

```xml
<async_load_system_database>true</async_load_system_database>
```
