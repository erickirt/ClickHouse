Ces paramètres configurent le serveur ClickHouse et sont générés automatiquement à partir du code source de ClickHouse.

<div id="cpu_slot_preemption">
  ## cpu_slot_preemption
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Oui" />

Définit comment s’effectue l’ordonnancement des workloads pour les ressources CPU (MASTER THREAD et WORKER THREAD).

* Si `true` (par défaut), la comptabilisation est basée sur le temps CPU réellement consommé. Une part équitable de temps CPU est allouée aux workloads en concurrence. Les slots sont alloués pour une durée limitée, puis redemandés à l’expiration. La demande d’un slot peut bloquer l’exécution d’un thread en cas de surcharge des ressources CPU, c’est-à-dire qu’une préemption peut se produire. Cela garantit une répartition équitable du temps CPU.
* Si `false`, la comptabilisation est basée sur le nombre de slots CPU alloués. Un nombre équitable de slots CPU est alloué aux workloads en concurrence. Un slot est alloué lorsqu’un thread démarre, conservé pendant toute son exécution, puis libéré lorsque le thread se termine. Le nombre de threads alloués à l’exécution d’une requête peut uniquement augmenter de 1 jusqu’à `max_threads` et ne jamais diminuer. Cela favorise davantage les requêtes de longue durée et peut entraîner une starvation CPU pour les requêtes courtes.

**Exemple**

```xml
<cpu_slot_preemption>true</cpu_slot_preemption>
```

<div id="cpu_slot_preemption_timeout_ms">
  ## cpu_slot_preemption_timeout_ms
</div>

<SettingsInfoBlock type="UInt64" default_value="1000" changeable_without_restart="Yes" />

Définit pendant combien de millisecondes un worker thread peut attendre lors de la préemption, c’est-à-dire en attendant qu’un autre slot CPU lui soit attribué. Passé ce délai, si le thread n’a pas pu obtenir un nouveau slot CPU, il s’arrête et la query est dynamiquement ramenée à un plus faible nombre de threads s’exécutant en parallèle. Notez que le master thread n’est jamais réduit, mais peut être préempté indéfiniment. Cela n’a de sens que lorsque `cpu_slot_preemption` est activé et qu’une ressource CPU est définie pour WORKER THREAD.

**Exemple**

```xml
<cpu_slot_preemption_timeout_ms>1000</cpu_slot_preemption_timeout_ms>
```

<div id="cpu_slot_quantum_ns">
  ## cpu_slot_quantum_ns
</div>

<SettingsInfoBlock type="UInt64" default_value="10000000" changeable_without_restart="Yes" />

Définit le nombre de nanosecondes de CPU qu&#39;un thread est autorisé à consommer après avoir acquis un slot CPU et avant de devoir en demander un autre. N&#39;a de sens que si `cpu_slot_preemption` est activé et que la ressource CPU est définie pour MASTER THREAD ou WORKER THREAD.

**Exemple**

```xml
<cpu_slot_quantum_ns>10000000</cpu_slot_quantum_ns>
```
