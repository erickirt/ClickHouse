Ces paramètres configurent le serveur ClickHouse et sont générés automatiquement à partir du code source de ClickHouse.

<div id="concurrent_threads_lazy_allocation">
  ## concurrent_threads_lazy_allocation
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Oui" />

Contrôle la façon dont les slot CPU sont alloués aux requêtes.

Lorsque `true` (par défaut), une requête démarre avec un slot CPU et ne demande des slot CPU supplémentaires à l’ordonnanceur que lorsque son pipeline génère réellement davantage de travail parallélisable. Cela évite qu’une requête réserve jusqu’à `max_threads` slot CPU sans n’en utiliser qu’une fraction, au détriment d’autres requêtes concurrentes. S’applique à la fois au contrôle de la concurrence et à l’ordonnanceur CPU préemptif pour les charges de travail.

Lorsque `false`, la requête demande d’emblée tous les `max_threads` slot CPU au démarrage.

<div id="concurrent_threads_scheduler">
  ## concurrent_threads_scheduler
</div>

<SettingsInfoBlock type="String" default_value="max_min_fair" changeable_without_restart="Yes" />

Politique définissant la manière d’ordonnancer les slots CPU spécifiés par `concurrent_threads_soft_limit_num` et `concurrent_threads_soft_limit_ratio_to_cores`. Algorithme utilisé pour déterminer comment un nombre limité de slots CPU est réparti entre des requêtes concurrentes. L’ordonnanceur peut être modifié à l’exécution sans redémarrer le serveur.

Valeurs possibles :

* `round_robin` — Chaque requête avec le setting `use_concurrency_control` = 1 alloue jusqu’à `max_threads` slots CPU. Un slot par thread. En cas de contention, les slots CPU sont accordés aux requêtes selon une méthode round-robin. Notez que le premier slot est accordé inconditionnellement, ce qui peut entraîner un manque d’équité et une latence accrue pour les requêtes ayant un `max_threads` élevé en présence d’un grand nombre de requêtes avec `max_threads` = 1.
* `fair_round_robin` — Chaque requête avec le setting `use_concurrency_control` = 1 alloue jusqu’à `max_threads - 1` slots CPU. Variante de `round_robin` qui ne nécessite pas de slot CPU pour le premier thread de chaque requête. Ainsi, les requêtes ayant `max_threads` = 1 ne nécessitent aucun slot et ne peuvent pas accaparer injustement tous les slots. Aucun slot n’est accordé inconditionnellement.
* `max_min_fair` — Chaque requête avec le setting `use_concurrency_control` = 1 alloue jusqu’à `max_threads - 1` slots CPU. Similaire à `fair_round_robin`, mais les slots libérés sont toujours accordés à la requête ayant actuellement le plus petit nombre de slots alloués. Cela offre une meilleure équité en cas de forte surallocation, lorsque de nombreuses requêtes se disputent un nombre limité de slots CPU. Les requêtes de courte durée ne sont pas pénalisées par les requêtes de longue durée qui ont accumulé davantage de slots au fil du temps.

<div id="concurrent_threads_soft_limit_num">
  ## concurrent_threads_soft_limit_num
</div>

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="Oui" />

Nombre maximal de threads de traitement des requêtes, hors threads utilisés pour récupérer des données depuis des serveurs distants, pouvant être utilisés pour exécuter l’ensemble des requêtes. Il ne s’agit pas d’une limite stricte. Si cette limite est atteinte, la requête disposera tout de même d’au moins un thread pour s’exécuter. La requête peut passer au nombre de threads souhaité pendant l’exécution si davantage de threads deviennent disponibles.

<Note>
  Une valeur de `0` (par défaut) signifie qu’il n’y a pas de limite.
</Note>

<div id="concurrent_threads_soft_limit_ratio_to_cores">
  ## concurrent_threads_soft_limit_ratio_to_cores
</div>

<SettingsInfoBlock type="UInt64" default_value="2" changeable_without_restart="Yes" />Identique à [`concurrent_threads_soft_limit_num`](/fr/reference/settings/server-settings/settings/concurrent-threads#concurrent_threads_soft_limit_num), mais exprimé sous forme de ratio par rapport au nombre de cœurs.
