<div id="max_streams_for_union_step">
  ## max_streams_for_union_step
</div>

<SettingsInfoBlock type="UInt64" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.5"},{"label": "0"},{"label": "Nouveau paramètre permettant de limiter le nombre de flux de données actifs simultanément dans une étape `UNION` afin de réduire le pic d'utilisation mémoire."}]}]} />

Limite le nombre de flux de données actifs simultanément dans une étape `UNION` (s&#39;applique à la fois à `UNION ALL` et à `UNION DISTINCT`, car `UNION DISTINCT` est implémenté par une étape `UNION ALL` suivie d&#39;une étape `DISTINCT`). Lorsqu&#39;une requête `UNION` comporte de nombreuses sous-requêtes, elles ouvrent toutes leurs tampons de lecture en même temps, ce qui entraîne une utilisation mémoire proportionnelle au nombre de sous-requêtes. Ce paramètre insère des processeurs `Concat` pour resserrer le pipeline afin qu&#39;il n&#39;y ait pas plus de ce nombre de flux actifs à la fois, ce qui réduit considérablement le pic de mémoire. La limite réelle correspond au minimum entre cette valeur et `max_threads * max_streams_for_union_step_to_max_threads_ratio` (si l&#39;une ou l&#39;autre vaut 0, elle est ignorée). Lorsque les deux valent 0, aucun resserrement n&#39;est appliqué. La limite n&#39;est pas non plus appliquée lorsque le plan de requête exige que chaque flux de sortie du `UNION` reste trié individuellement (par exemple lorsque l&#39;optimisation de lecture dans l&#39;ordre est appliquée sur le `UNION`) ; dans ce cas, le respect de l&#39;ordre est prioritaire et le resserrement est ignoré.

<div id="max_streams_for_union_step_to_max_threads_ratio">
  ## max_streams_for_union_step_to_max_threads_ratio
</div>

<SettingsInfoBlock type="Float" default_value="8" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.5"},{"label": "8"},{"label": "Nouveau paramètre : la limite du nombre de flux simultanément actifs dans une étape `UNION` est calculée comme min(max_streams_for_union_step, max_threads * max_streams_for_union_step_to_max_threads_ratio) ; si l’un des deux vaut 0, il n’est pas pris en compte."}]}]} />

Ce ratio, multiplié par `max_threads`, détermine la limite du nombre de flux simultanément actifs dans une étape `UNION` (s’applique à la fois à `UNION ALL` et à `UNION DISTINCT`). La limite réelle est la plus petite des deux valeurs suivantes : cette valeur calculée et `max_streams_for_union_step` (si l’une ou l’autre vaut 0, elle est ignorée). Par exemple, avec `max_threads = 8` et ce ratio défini sur 1, il y aura au plus 8 flux actifs. Définissez cette valeur sur 0 pour désactiver cette limite basée sur le ratio. Comme pour `max_streams_for_union_step`, la limite n’est pas appliquée lorsque le plan de requête exige que chaque flux de sortie du `UNION` reste trié individuellement.

<div id="max_streams_multiplier_for_merge_tables">
  ## max_streams_multiplier_for_merge_tables
</div>

<SettingsInfoBlock type="Float" default_value="5" />

Demande davantage de flux lors de la lecture à partir d&#39;une table Merge. Les flux seront répartis entre les tables utilisées par la table Merge. Cela permet de mieux répartir la charge entre les threads et s&#39;avère particulièrement utile lorsque les tables fusionnées n&#39;ont pas la même taille.

<div id="max_streams_to_max_threads_ratio">
  ## max_streams_to_max_threads_ratio
</div>

<SettingsInfoBlock type="Float" default_value="1" />

Permet d’utiliser plus de sources que de threads, afin de répartir le travail plus uniformément entre eux. Il s’agit vraisemblablement d’une solution temporaire, car il sera possible à l’avenir d’avoir autant de sources que de threads, chaque source pouvant alors sélectionner dynamiquement le travail disponible pour elle-même.
