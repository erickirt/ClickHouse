<div id="enable_add_distinct_to_in_subqueries">
  ## enable_add_distinct_to_in_subqueries

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "25.8"},{"label": "0"},{"label": "Nouveau paramètre permettant de réduire la taille des tables temporaires transférées pour les sous-requêtes `IN` distribuées."}]}]} />

Active `DISTINCT` dans les sous-requêtes `IN`. Ce paramètre implique un compromis : son activation peut réduire considérablement la taille des tables temporaires transférées pour les sous-requêtes `IN` distribuées et accélérer nettement le transfert de données entre les shards, en garantissant que seules des valeurs uniques sont envoyées.

<div id="enable_order_by_all">

</div>

<SettingsInfoBlock type="Bool" default_value="1" />

**Exemple**

Requête :


```text

│ 20 │ 20 │  10 │
│ 30 │ 10 │  20 │
│ 10 │ 20 │  30 │
└────┴────┴─────┘
```

<div id="enable_shared_storage_snapshot_in_query">
  ## enable_shared_storage_snapshot_in_query
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "25.6"},{"label": "0"},{"label": "Nouveau paramètre permettant de partager l’instantané de stockage dans la requête"}]}, {"id": "row-2","items": [{"label": "25.12"},{"label": "1"},{"label": "Active par défaut le partage de l’instantané de stockage dans la requête"}]}]} />

Si cette option est activée, toutes les sous-requêtes d’une même requête partageront le même StorageSnapshot pour chaque table.
Cela garantit une vue cohérente des données dans l’ensemble de la requête, même si la même table est accédée plusieurs fois.

Ce paramètre est nécessaire pour les requêtes où la cohérence interne des data parts est importante. Exemple :

```sql
SELECT
    count()
FROM events
WHERE (_part, _part_offset) IN (
    SELECT _part, _part_offset
    FROM events
    WHERE user_id = 42
)
```

Sans ce paramètre, la requête externe et la requête interne peuvent s’exécuter sur des instantanés de données différents, ce qui peut entraîner des résultats incorrects.

<Note>
  L’activation de ce paramètre désactive l’optimisation qui supprime les parties de données inutiles des instantanés une fois l’étape de planification terminée.
  Par conséquent, les requêtes de longue durée peuvent conserver des parties obsolètes pendant toute leur exécution, ce qui retarde leur nettoyage et accroît la pression sur le stockage.
