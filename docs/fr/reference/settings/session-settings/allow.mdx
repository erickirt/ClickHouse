<div id="allow_key_condition_coalesce_rewrite">
  ## allow_key_condition_coalesce_rewrite
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.5"},{"label": "1"},{"label": "Nouveau paramètre permettant de réécrire, avant l’analyse des index, les prédicats de la forme `coalesce(a_1, ..., a_N) <op> const` (ou, de manière équivalente, `ifNull`, ou avec la constante à gauche) en une disjonction, afin de pouvoir utiliser la clé primaire par colonne et les skip indexes sur chaque `a_i`. Les formes partiellement constantes telles que `coalesce(a, 42, b)` et `coalesce(a, b, 42)` sont également prises en charge."}]}]} />

Permet à la clé primaire MergeTree et aux skip indexes d’exclure des granules pour les prédicats `WHERE`/`PREWHERE` qui utilisent `coalesce` ou `ifNull`. Sans ce paramètre, ces prédicats sont opaques pour l’analyse des index et ne permettent pas d’exclure de granules, de sorte que des granules qui ne peuvent pas correspondre sont quand même lus. Cela n’affecte que les granules lus ; le résultat de la requête reste inchangé, car les lignes sont toujours filtrées par le prédicat d’origine.

Deux formes de prédicats sont réécrites avant l’analyse des index :

* Une comparaison avec `coalesce`/`ifNull`, telle que `coalesce(a, b) = 5`, devient une disjonction afin qu’un index sur chaque argument puisse exclure des granules : `a = 5 OR (a IS NULL AND b = 5)`, avec extension à davantage d’arguments.
* Un `coalesce`/`ifNull` avec une constante par défaut falsy (zéro) utilisée directement comme condition, telle que `ifNull(a = 5, 0)` ou `coalesce(a = 5, 0)`, est ramené à son prédicat interne `a = 5`. De tels wrappers ramènent le résultat ternaire du prédicat interne à un booléen déterminé (en faisant correspondre `NULL` à `false`).

<div id="allow_rank_dense_rank_arguments">
  ## allow_rank_dense_rank_arguments
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.5"},{"label": "0"},{"label": "Nouveau paramètre. Autorise le passage d&#39;arguments aux fonctions de fenêtre `RANK` et `DENSE_RANK` pour assurer la rétrocompatibilité.

Selon la norme SQL, `RANK` et `DENSE_RANK` ne prennent aucun argument — elles classent les lignes uniquement en fonction de la fenêtre
`OVER (ORDER BY ...)`. Dans les versions de ClickHouse antérieures à la 26.5, des requêtes telles que
`RANK(x) OVER (...)` acceptaient et ignoraient silencieusement l&#39;argument, ce qui prêtait à confusion
(l&#39;argument visible laissait penser qu&#39;il influençait le classement, mais ce n&#39;était pas le cas).

Lorsque ce paramètre est `false` (valeur par défaut), `RANK` et `DENSE_RANK` rejettent tout argument et
renvoient `NUMBER_OF_ARGUMENTS_DOESNT_MATCH`. Lorsqu&#39;il est défini sur `true`, le comportement permissif legacy est
rétabli — les arguments sont silencieusement ignorés, comme avant la version 26.5.

</div>

<SettingsInfoBlock type="Bool" default_value="0" />

Autorise `ALTER TABLE ... REPLACE PARTITION ... FROM ...` à supprimer silencieusement la partition de destination lorsque la source n’a aucune part dans cette partition.

Par défaut, ce n’est pas autorisé : un `REPLACE PARTITION` à partir d’une source qui ne contient aucune donnée dans la partition demandée lève une exception, car dans ce cas l’opération revient en pratique à un `DROP PARTITION` silencieux sur la destination (les données de destination sont supprimées et rien ne les remplace), ce qui est une cause fréquente de perte accidentelle de données (voir [#23727](https://github.com/ClickHouse/ClickHouse/issues/23727)).

Activez ce paramètre pour rétablir le comportement précédent, par exemple si vous utilisez volontairement une partition source vide pour effacer les données de destination. Pour une suppression inconditionnelle, préférez `ALTER TABLE ... DROP PARTITION ...`.

* 0 — Interdire.
* 1 — Autoriser.
