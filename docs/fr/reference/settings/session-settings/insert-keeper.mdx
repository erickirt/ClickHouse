<div id="insert_keeper_max_retries">
  ## insert_keeper_max_retries
</div>

<SettingsInfoBlock type="UInt64" default_value="20" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "23.2"},{"label": "20"},{"label": "Active les reconnexions à Keeper lors d'INSERT et améliore la fiabilité"}]}]} />

Ce paramètre définit le nombre maximal de nouvelles tentatives pour les requêtes ClickHouse Keeper (ou ZooKeeper) lors d&#39;un insert dans un MergeTree répliqué. Seules les requêtes Keeper ayant échoué en raison d&#39;une erreur réseau, d&#39;un dépassement du délai de session Keeper ou d&#39;un dépassement du délai de requête font l&#39;objet de nouvelles tentatives.

Valeurs possibles :

* Entier positif.
* 0 — Les nouvelles tentatives sont désactivées

Les nouvelles tentatives pour les requêtes Keeper sont effectuées après un certain délai. Ce délai est contrôlé par les paramètres suivants : `insert_keeper_retry_initial_backoff_ms`, `insert_keeper_retry_max_backoff_ms`.
La première nouvelle tentative est effectuée après le délai `insert_keeper_retry_initial_backoff_ms`. Les délais suivants sont calculés comme suit :

```
timeout = min(insert_keeper_retry_max_backoff_ms, latest_timeout * 2)
```

Par exemple, si `insert_keeper_retry_initial_backoff_ms=100`, `insert_keeper_retry_max_backoff_ms=10000` et `insert_keeper_max_retries=8`, alors les délais d&#39;attente seront de `100, 200, 400, 800, 1600, 3200, 6400, 10000`.

Au-delà de la tolérance aux pannes, les nouvelles tentatives visent à offrir une meilleure expérience utilisateur : elles permettent d&#39;éviter de renvoyer une erreur pendant l&#39;exécution de INSERT si Keeper est redémarré, par exemple à l&#39;occasion d&#39;une mise à niveau.
