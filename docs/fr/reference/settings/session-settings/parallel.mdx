<div id="parallel_distributed_insert_select">
  ## parallel_distributed_insert_select
</div>

<SettingsInfoBlock type="UInt64" default_value="2" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "25.7"},{"label": "2"},{"label": "Active parallel distributed insert select par défaut"}]}]} />

Active l&#39;exécution parallèle des requêtes distribuées `INSERT ... SELECT`.

Si nous exécutons des requêtes `INSERT INTO distributed_table_a SELECT ... FROM distributed_table_b`, que les deux tables utilisent le même cluster et qu&#39;elles sont toutes deux soit [répliquées](/fr/reference/engines/table-engines/mergetree-family/replication), soit non répliquées, alors la requête est traitée localement sur chaque shard.

Valeurs possibles :

* `0` — Désactivé.
* `1` — `SELECT` sera exécuté sur chaque shard à partir de la table sous-jacente du moteur Distributed.
* `2` — `SELECT` et `INSERT` seront exécutés sur chaque shard, respectivement depuis/vers la table sous-jacente du moteur Distributed.

Depuis la v25.4, `INSERT ... SELECT` à partir d&#39;une source `ReplicatedMergeTree` ou `SharedMergeTree` peut également être parallélisé entre les répliques. Pour l&#39;activer :

* `parallel_distributed_insert_select = 2`
* `enable_parallel_replicas = 1`

<div id="parallel_hash_join_threshold">
  ## parallel_hash_join_threshold
</div>

<SettingsInfoBlock type="UInt64" default_value="100000" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "25.5"},{"label": "100000"},{"label": "Nouveau paramètre"}]}, {"id": "row-2","items": [{"label": "25.4"},{"label": "0"},{"label": "Nouveau paramètre"}]}, {"id": "row-3","items": [{"label": "25.3"},{"label": "0"},{"label": "Nouveau paramètre"}]}]} />

Lorsque l&#39;algorithme de jointure basé sur le hachage est utilisé, ce seuil permet de choisir entre `hash` et `parallel_hash` (uniquement si une estimation de la taille de la table de droite est disponible).
Le premier est utilisé lorsque l&#39;on sait que la taille de la table de droite est inférieure au seuil.

<div id="parallel_non_joined_rows_processing">
  ## parallel_non_joined_rows_processing
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "26.2"},{"label": "1"},{"label": "Nouveau paramètre permettant d'activer le traitement en parallèle des lignes sans correspondance dans les JOIN RIGHT/FULL `parallel_hash`."}]}]} />

Permet à plusieurs threads de traiter en parallèle les lignes sans correspondance de la table de droite lors des JOIN RIGHT et FULL.
Cela peut accélérer cette phase lors de l&#39;utilisation de l&#39;algorithme de jointure `parallel_hash` avec de grandes tables.
Lorsqu&#39;il est désactivé, les lignes sans correspondance sont traitées par un seul thread.

<div id="parallel_replica_offset">
  ## parallel_replica_offset
</div>

<BetaBadge />

<SettingsInfoBlock type="UInt64" default_value="0" />

Il s&#39;agit d&#39;un paramètre interne qui ne doit pas être utilisé directement et qui correspond à un détail d&#39;implémentation du mode « répliques parallèles ». Ce paramètre sera automatiquement défini par le serveur initiateur, pour les requêtes distribuées, sur l&#39;index de la réplique participant au traitement de la requête parmi les répliques parallèles.
