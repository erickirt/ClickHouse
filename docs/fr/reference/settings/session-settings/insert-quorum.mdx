<div id="insert_quorum">
  ## insert_quorum
</div>

<SettingsInfoBlock type="UInt64Auto" default_value="0" />

<Note>
  Ce paramètre ne s&#39;applique pas à SharedMergeTree. Voir [la cohérence de SharedMergeTree](/fr/products/cloud/features/infrastructure/shared-merge-tree#consistency) pour plus d&#39;informations.
</Note>

Active les écritures en quorum.

* Si `insert_quorum < 2`, les écritures en quorum sont désactivées.
* Si `insert_quorum >= 2`, les écritures en quorum sont activées.
* Si `insert_quorum = 'auto'`, utilise le nombre majoritaire (`number_of_replicas / 2 + 1`) comme quorum.

Écritures en quorum

`INSERT` réussit uniquement lorsque ClickHouse parvient à écrire correctement les données sur `insert_quorum` répliques pendant `insert_quorum_timeout`. Si, pour une raison quelconque, le nombre de répliques sur lesquelles l&#39;écriture a réussi n&#39;atteint pas `insert_quorum`, l&#39;écriture est considérée comme ayant échoué et ClickHouse supprimera le bloc inséré de toutes les répliques où les données ont déjà été écrites.

Lorsque `insert_quorum_parallel` est désactivé, toutes les répliques du quorum sont cohérentes, c&#39;est-à-dire qu&#39;elles contiennent les données de toutes les requêtes `INSERT` précédentes (la séquence des `INSERT` est linéarisée). Lors de la lecture de données écrites à l&#39;aide de `insert_quorum`, si `insert_quorum_parallel` est désactivé, vous pouvez activer la cohérence séquentielle pour les requêtes `SELECT` à l&#39;aide de [select&#95;sequential&#95;consistency](/fr/reference/settings/session-settings/other#select_sequential_consistency).

ClickHouse génère une exception :

* Si le nombre de répliques disponibles au moment de la requête est inférieur à `insert_quorum`.
* Lorsque `insert_quorum_parallel` est désactivé et qu&#39;une tentative d&#39;écriture de données est effectuée alors que le bloc précédent n&#39;a pas encore été inséré sur `insert_quorum` répliques. Cette situation peut se produire si l&#39;utilisateur essaie d&#39;exécuter une autre requête `INSERT` sur la même table avant que la précédente avec `insert_quorum` ne soit terminée.

  ## insert_quorum_parallel
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "21.1"},{"label": "1"},{"label": "Utiliser par défaut les insertions avec quorum parallèles. Elles sont nettement plus pratiques que les insertions avec quorum séquentielles"}]}]} />

<Note>
  Ce paramètre ne s&#39;applique pas à SharedMergeTree ; voir [cohérence de SharedMergeTree](/fr/products/cloud/features/infrastructure/shared-merge-tree#consistency) pour plus d&#39;informations.
</Note>

Active ou désactive le parallélisme pour les requêtes `INSERT` avec quorum. S&#39;il est activé, des requêtes `INSERT` supplémentaires peuvent être envoyées avant la fin des requêtes précédentes. S&#39;il est désactivé, les écritures supplémentaires dans la même table seront rejetées.

Valeurs possibles :

* 0 — Désactivé.
* 1 — Activé.

<div id="insert_quorum_timeout">
  ## insert_quorum_timeout
</div>

<SettingsInfoBlock type="Millisecondes" default_value="600000" />

Délai d’expiration d’une écriture vers un quorum, en millisecondes. Si ce délai est dépassé et qu’aucune écriture n’a encore eu lieu, ClickHouse renverra une exception et le client devra relancer la requête pour écrire le même bloc sur la même réplique ou sur une autre réplique.
