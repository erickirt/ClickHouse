<div id="query_plan_optimize_join_order_algorithm">
  ## query_plan_optimize_join_order_algorithm
</div>

<ExperimentalBadge />

<SettingsInfoBlock type="JoinOrderAlgorithm" default_value="greedy" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "25.12"},{"label": "greedy"},{"label": "新的实验性设置。"}]}]} />

指定在优化查询计划时要尝试哪些 JOIN 顺序算法。可用算法如下：

* &#39;greedy&#39; - 基础贪心算法 - 速度快，但不一定能生成最佳 JOIN 顺序
* &#39;dpsize&#39; - 实现 DPsize 算法，目前仅支持 INNER JOIN - 会考虑所有可能的 JOIN 顺序并找出最优方案，但对于包含大量表和 JOIN 谓词的查询可能较慢。
* &#39;dpsub&#39; - 实现 DPsub 算法，同时支持 INNER JOIN 和非 INNER JOIN - 会考虑所有可能的 JOIN 顺序并找出最优方案，但对于包含大量表和 JOIN 谓词的查询可能较慢。
* &#39;dphyp&#39; - 实现 DPhyp (通过超图分区进行动态规划) 算法，目前仅支持 INNER JOIN - 探索与 `dpsize` 相同的搜索空间，但只枚举连通子图对，因此在稀疏 JOIN 图上生成的中间 JOIN 更少，代价是不考虑笛卡尔积
  可以将多个算法指定为逗号分隔的列表，例如 `dphyp,greedy`。系统会按顺序尝试这些算法；如果某个算法无法处理该查询 (例如由于存在 OUTER JOIN 或不连通分量) ，则会回退到下一个算法。
