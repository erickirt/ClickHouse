<div id="apply_deleted_mask">
  ## apply_deleted_mask
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

启用后会过滤掉通过轻量级 DELETE 删除的行。若禁用，查询仍可读取这些行。这对调试和“取消删除”场景很有用

<div id="apply_mutations_on_fly">
  ## apply_mutations_on_fly
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

如果为 true，则尚未在数据分区片段中 materialized 的变更 (UPDATE 和 DELETE) 会在 SELECT 查询时应用。

<div id="apply_prewhere_after_final">
  ## apply_prewhere_after_final
</div>

<SettingsInfoBlock type="Bool" default_value="0" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "25.12"},{"label": "0"},{"label": "新设置。启用后，PREWHERE 条件将在 FINAL 处理后应用。"}]}]} />

启用后，对于 ReplacingMergeTree 及类似引擎，PREWHERE 条件将在 FINAL 处理后应用。
当 PREWHERE 引用了在重复行之间可能取值不同的列时，这会很有用，
因为你可能希望先由 FINAL 选出最终保留的行，再进行过滤。禁用时，PREWHERE 会在读取期间应用。
注意：如果启用了 apply&#95;row&#95;level&#95;security&#95;after&#95;final，且行策略使用了非排序键列，PREWHERE 也会
被延后，以保持正确的执行顺序 (必须先应用行策略，再应用 PREWHERE) 。

<div id="apply_row_policy_after_final">
  ## apply_row_policy_after_final
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "25.10"},{"label": "1"},{"label": "该设置是为修复 post-#87303 回归问题而新增的；在该回归中，行策略和 PREWHERE 会在 FINAL 之前应用。PR #91065 引入了此设置，PR #97279 则将确保正确性的 true 设为默认值。记录为 {true, true}，因此 compatibility 永远不会回退到修复前的 false 行为。"}]}]} />

启用后，对于 *MergeTree 表，行策略和 PREWHERE 会在 FINAL 处理之后应用。 (尤其是 ReplacingMergeTree)
禁用后，行策略会在 FINAL 之前应用；当策略
过滤掉本应在 ReplacingMergeTree 或类似引擎中用于去重的行时，可能会导致结果不同。

如果行策略表达式仅依赖 ORDER BY 中的列，出于优化考虑，它仍会在 FINAL 之前应用，
因为这种过滤不会影响去重结果。

可能的值：

* 0 — 行策略和 PREWHERE 在 FINAL 之前应用 (默认) 。
* 1 — 行策略和 PREWHERE 在 FINAL 之后应用。

<div id="apply_settings_from_server">
  ## apply_settings_from_server
</div>

<SettingsInfoBlock type="Bool" default_value="1" />

<VersionHistory rows={[{"id": "row-1","items": [{"label": "25.2"},{"label": "1"},{"label": "客户端代码（例如 INSERT 输入解析和查询输出格式化）将使用与服务器相同的设置，包括服务器配置中的设置。"}]}]} />

客户端是否接受来自服务器的设置。

这只会影响在客户端执行的操作，尤其是 INSERT 输入数据的解析和查询结果的格式化。大多数查询执行都发生在服务器上，不受此设置影响。

通常，这个设置应在用户 profile 中设置 (通过 users.xml 或 `ALTER USER` 之类的查询) ，而不是通过客户端来设置 (客户端命令行参数、`SET` 查询，或 `SELECT` 查询中的 `SETTINGS` 部分) 。通过客户端可以将其改为 false，但不能改为 true (因为如果用户 profile 中设置了 `apply_settings_from_server = false`，服务器就不会发送这些设置) 。

请注意，最初 (24.12) 使用的是一个服务器设置 (`send_settings_to_client`) ，但后来为提升易用性，它被这个客户端设置取代了。
