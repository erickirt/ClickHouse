<div id="max_avg_part_size_for_too_many_parts">
  ## max_avg_part_size_for_too_many_parts
</div>

<SettingsInfoBlock type="UInt64" default_value="1073741824" />

根据 &#39;parts&#95;to&#95;delay&#95;insert&#39; 和
&#39;parts&#95;to&#95;throw&#95;insert&#39; 的 &#39;parts 过多&#39; 检查，
只有当相关分区中的平均 part 大小不超过指定阈值时，该检查才会生效。如果平均 part 大小
大于指定阈值，则 INSERT 操作既不会被延迟，也不会被
拒绝。这样一来，如果 parts 能成功 merge 为更大的 parts，就可以在单个
server 上的单个表中存储数百 TB 的数据。这
不会影响非活跃 parts 或总 parts 数的阈值。

</div>

<SettingsInfoBlock type="UInt64" default_value="0" />

<div id="max_concurrent_queries">
  ## max_concurrent_queries
</div>

<SettingsInfoBlock type="UInt64" default_value="0" />

与 MergeTree 表相关的可并发执行的查询最大数量。
查询仍会受到其他 `max_concurrent_queries` 设置的限制。

可能的值：

* 正整数。
* `0` — 不限制。

默认值：`0` (不限制) 。

**示例**

```xml
<max_concurrent_queries>50</max_concurrent_queries>
```

<div id="max_file_name_length">
  ## max_file_name_length
</div>

<SettingsInfoBlock type="UInt64" default_value="127" />

在不对文件名进行哈希处理的情况下，保留原文件名时允许的最大长度。
仅在启用设置 `replace_long_file_name_to_hash` 时生效。
此设置的值不包括文件扩展名的长度。因此，
建议将其设置得低于文件名最大长度 (通常为 255
字节) ，并预留一定余量，以避免文件系统错误。
