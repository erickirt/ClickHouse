这些设置用于配置 ClickHouse server，由 ClickHouse 源代码自动生成。

<div id="interserver_listen_host">
  ## interserver_listen_host
</div>

限制可在 ClickHouse 服务器之间交换数据的主机。
如果使用 Keeper，相同的限制也会应用于不同 Keeper 实例之间的通信。

<Note>
  默认情况下，该值等于 [`listen_host`](/zh/reference/settings/server-settings/settings/listen#listen_host) 设置。
</Note>

**示例**

```xml
<interserver_listen_host>::ffff:a00:1</interserver_listen_host>
<interserver_listen_host>10.0.0.1</interserver_listen_host>
```

类型：

默认值：

<div id="interserver_tables_status_require_auth">
  ## interserver_tables_status_require_auth
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

要求对 interserver `TablesStatusRequest` 使用集群
`<secret>` 进行身份验证。足够新的客户端会发送 secret 哈希 (协议修订版本
`DBMS_MIN_REVISION_WITH_INTERSERVER_SECRET_TABLES_STATUS`) ，因此始终会通过验证；此
设置还会额外拒绝不发送哈希的旧版客户端，从而默认关闭
未经身份验证的表状态泄露问题。

默认为 `true` (默认安全) 。在滚动升级期间，尚未升级的
节点会使用旧协议，且不会发送哈希，因此，如果在这样的节点上发起针对
已升级节点的 `Distributed` 查询，其 `TablesStatusRequest`
将被拒绝。如果你必须运行混合版本集群，请在已升级的
节点上将此项设为 `false`，直到所有节点都完成升级，然后移除此覆盖设置。
