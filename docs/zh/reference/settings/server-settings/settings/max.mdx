<div id="max_authentication_methods_per_user">
  ## max_authentication_methods_per_user
</div>

<SettingsInfoBlock type="UInt64" default_value="100" changeable_without_restart="No" />

创建用户时可配置，或修改用户时可更改为的最大身份验证方法数量。
更改此设置不会影响现有用户。如果与身份验证相关的创建/修改查询超出此设置指定的限制，则会失败。
与身份验证无关的创建/修改查询将成功。

<Note>
  值为 `0` 表示无限制。
</Note>

<div id="max_os_cpu_wait_time_ratio_to_drop_connection">
  ## max_os_cpu_wait_time_ratio_to_drop_connection
</div>

<SettingsInfoBlock type="Float" default_value="0" changeable_without_restart="Yes" />

用于判断是否应断开连接时，OS CPU 等待时间 (`OSCPUWaitMicroseconds` 指标) 与忙碌时间 (`OSCPUVirtualTimeMicroseconds` 指标) 之间的最大比率。系统会使用最小比率与最大比率之间的线性插值来计算概率；在该比率下，概率为 1。
更多详情，请参见[控制服务器 CPU 过载时的行为](/zh/concepts/features/configuration/settings/server-overload)。

<div id="max_partition_size_to_drop">
  ## max_partition_size_to_drop
</div>

<SettingsInfoBlock type="UInt64" default_value="50000000000" changeable_without_restart="Yes" />

对删除分区操作的限制。

如果 [MergeTree](/zh/reference/engines/table-engines/mergetree-family/mergetree) 表的大小超过 [`max_partition_size_to_drop`](/zh/reference/settings/server-settings/settings/max#max_partition_size_to_drop) (以字节为单位) ，则不能使用 [DROP PARTITION](/zh/reference/statements/alter/partition#drop-partitionpart) 查询删除分区。
此设置生效无需重启 ClickHouse 服务器。另一种禁用此限制的方法是创建 `<clickhouse-path>/flags/force_drop_table` 文件。

<Note>
  值 `0` 表示你可以不受任何限制地删除分区。

  此限制不适用于 drop table 和 truncate table，参见 [max&#95;table&#95;size&#95;to&#95;drop](/zh/reference/settings/session-settings#max_table_size_to_drop)
</Note>

**示例**

```xml
<max_partition_size_to_drop>0</max_partition_size_to_drop>
```

<div id="max_per_cpu_untracked_memory">
  ## max_per_cpu_untracked_memory
</div>

<SettingsInfoBlock type="UInt64" default_value="8388608" changeable_without_restart="Yes" />

在刷新到 memory tracker 之前，单个 CPU 上运行的所有线程可同时持有的未跟踪内存总量上限 (以字节为单位) 。`max_untracked_memory` 限制的是单个线程，而此设置限制的是每个 CPU 的总量，因此多个线程无法将各自的线程配额叠加成整个服务器范围内的大量 overcommit。因此，未跟踪内存总量大致受限于 `number_of_cpus * max_per_cpu_untracked_memory`。值为 `0` 时将禁用每 CPU 上限 (仅应用每线程的 `max_untracked_memory`) 。仅适用于 Linux。

<div id="max_waiting_queries">
  ## max_waiting_queries
</div>

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="Yes" />

对并发等待状态的查询总数进行限制。
当所需表正在异步加载时，等待中的查询会被阻塞，无法执行 (参见 [`async_load_databases`](/zh/reference/settings/server-settings/settings/async-load#async_load_databases)。

<Note>
  在检查以下设置控制的限制时，处于等待状态的查询不会计入其中：

  * [`max_concurrent_queries`](/zh/reference/settings/server-settings/settings/max-concurrent#max_concurrent_queries)
  * [`max_concurrent_insert_queries`](/zh/reference/settings/server-settings/settings/max-concurrent#max_concurrent_insert_queries)
  * [`max_concurrent_select_queries`](/zh/reference/settings/server-settings/settings/max-concurrent#max_concurrent_select_queries)
  * [`max_concurrent_queries_for_user`](/zh/reference/settings/session-settings#max_concurrent_queries_for_user)
  * [`max_concurrent_queries_for_all_users`](/zh/reference/settings/session-settings#max_concurrent_queries_for_all_users)

  这样调整是为了避免在 server 刚启动后立即触及这些限制。
</Note>

<Note>
  `0` (默认值) 表示不受限制。

  此设置可在运行时修改，并会立即生效。已在运行中的查询不会受到影响。
</Note>
